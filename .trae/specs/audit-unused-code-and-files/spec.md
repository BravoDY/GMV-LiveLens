# 无用代码与无用文件深度审计 Spec

## Why
当前项目经历了多轮功能迭代、兼容分支和调试脚本沉淀，仓库中可能存在历史遗留模块、未接入入口的脚本、被替代但未清理的实现，以及页面不再引用的前端资源。若不先做系统性审计就贸然删除，容易误删隐式入口、手工运维脚本或运行时动态加载文件。

## What Changes
- 对项目进行只读深度审计，识别疑似无用代码、无用文件、历史残留脚本和未接入前后端资源。
- 建立“无引用”判定口径：静态引用、动态入口、配置引用、运行时扫描、脚本入口、文档约定都要交叉检查。
- 对每个候选项给出证据链：文件位置、导入/调用搜索结果、入口链路、保留风险、删除风险。
- 将候选项分级为“高置信可清理”“需人工确认”“不能判定为无用”三类，避免误杀。
- 输出审计结论时明确哪些文件/代码已经确认未被任何引用，哪些因为动态加载、手工脚本或外部流程而不能下结论。
- **不在本次任务中直接删除文件或修改业务逻辑**，仅完成深度分析与证据归档。

## Impact
- Affected specs: 仓库清理策略、入口链路审计、脚本资产治理、前后端引用关系梳理
- Affected code:
  - `backend/`
  - `frontend/`
  - `scripts/`
  - `tools/`
  - `tests/`
  - `data/` 下受版本控制或作为模板存在的静态资源

## ADDED Requirements
### Requirement: 无用代码与文件审计口径
系统 SHALL 采用多维度交叉验证来判断某段代码或某个文件是否“未被任何引用”，不能仅凭单次全文搜索就下结论。

#### Scenario: 判定候选 Python 文件
- **WHEN** 审计某个后端 Python 文件或模块
- **THEN** 需要同时检查 `import`/调用链、FastAPI 路由入口、脚本入口、调度入口、配置入口和运行时动态引用可能性
- **AND** 只有在这些路径都未发现引用证据时，才可进入“高置信可清理”候选

#### Scenario: 判定候选前端文件
- **WHEN** 审计某个前端 JS/CSS/HTML 资源
- **THEN** 需要同时检查 HTML script/style 引用、JS import 关系、模板拼接、运行时 fetch/load、手工打开入口和页面切换逻辑
- **AND** 若存在动态拼接或页面直开可能性，则不能直接判为无用

### Requirement: 审计输出分级
系统 SHALL 将候选项分为“高置信可清理”“需人工确认”“不能判定为无用”三档，并给出理由。

#### Scenario: 高置信可清理
- **WHEN** 某文件不存在静态引用、动态入口、配置引用、脚本调用、文档约定用途
- **THEN** 标记为“高置信可清理”
- **AND** 输出删除风险较低的理由与证据

#### Scenario: 需人工确认
- **WHEN** 某文件未发现直接代码引用，但疑似由手工运维、浏览器直开、外部工具或未来预留流程使用
- **THEN** 标记为“需人工确认”
- **AND** 明确无法自动证伪的原因

### Requirement: 审计覆盖范围
系统 SHALL 至少覆盖后端模块、前端页面与脚本、项目工具脚本、测试脚本、说明文档中提到的关键入口文件。

#### Scenario: 文档声明的入口文件
- **WHEN** README 或完整说明书中声明某文件是启动、维护、排障或数据处理入口
- **THEN** 该文件不能仅凭搜索无引用就判定为无用
- **AND** 需要结合文档用途一起判断

## MODIFIED Requirements
### Requirement: 文件清理前验证方式
在形成清理建议前，验证方式从“文本搜索是否出现文件名/函数名”提升为“代码搜索 + 配置搜索 + 文档搜索 + 入口链路 + 运行时行为”联合判断。

## REMOVED Requirements
### Requirement: 单凭一次 Grep 结果即可认定无用
**Reason**: 项目存在脚本入口、动态加载、手工运维和多页面切换场景，单次文本搜索容易误判。  
**Migration**: 改为输出带证据链的分级审计结论，再由用户决定是否进入清理实施。
