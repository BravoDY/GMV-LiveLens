# Tasks

- [ ] Task 1: 复核并统一前端“当前任务绑定上下文”来源
  - [ ] 盘点 `frontend/config.js`、`frontend/edge.js`、`frontend/core.js` 中所有读取 `task.page_id/page_url/page_title/edge_session_id` 的入口。
  - [ ] 明确哪些入口应改为读取“最新绑定上下文”，避免继续直接依赖旧任务快照。
  - [ ] 明确最小改动面，避免重写整个 `setupFlowState()` 或工作台渲染流程。

- [ ] Task 2: 复核并统一“切换店铺”入口的数据来源
  - [ ] 盘点 `renderSetupShopJumper()`、`configuredSetupTasks()`、`buildSetupSummary()`、`state.shopConfigs` 与 `state.snapshot.tasks` 的口径差异。
  - [ ] 明确缺项成因与修复边界，避免“统计口径是 4 家、切换下拉只有 3 家”继续出现。
  - [ ] 确认缺失任务时是否走受控补齐，且不引入无条件全量初始化。

- [ ] Task 3: 实现候选页预加载后的任务快照同步
  - [ ] 在 `frontend/config.js` 中处理 `/api/tasks/{task_id}/page-candidates` 返回值，若后端已自动恢复或确认当前绑定，更新前端内存中的当前任务字段。
  - [ ] 保证 `currentSetupTask()`、步骤文案、已选中页签卡片、恢复提示使用同一份最新绑定数据。
  - [ ] 如有必要，抽一个最小公共同步函数，避免手动绑定、自动恢复、局部刷新三套分支重复写状态。

- [ ] Task 4: 修正“生成预览”使用的 page_id 来源
  - [ ] 在 `frontend/edge.js` 中让 `previewRemotePage()` 优先读取最新绑定上下文。
  - [ ] 若存在手动重新绑定、自动恢复、预览后继续 OCR 等分支，保证后续动作继续使用同一份最新绑定信息。
  - [ ] 保留现有失败兜底分支，只降低其成为主流程的概率。

- [ ] Task 5: 修复“切换店铺”下拉栏缺项问题
  - [ ] 让 `切换店铺` 下拉栏以最新店铺配置为主渲染，而不是仅依赖现有任务快照。
  - [ ] 当某个店铺配置缺少任务记录时，提供受控补齐或修复路径，确保该店铺仍可被切换进入。
  - [ ] 保证下拉栏顺序、当前焦点、下一家队列、统计口径一致。
  - [ ] 确保这次修复不会影响任务管理列表、已完成店铺显示和“保存并进入下一家”逻辑。

- [ ] Task 6: 做一致性兜底与文案修正
  - [ ] 当候选页状态与任务快照不一致时，增加保护逻辑，避免界面显示“已选中”却请求旧页签。
  - [ ] 仅在最新绑定也失效时，才显示“当前店铺绑定页签已失效”的恢复卡片。
  - [ ] 当店铺配置与任务快照不一致时，提示应说明是“配置已存在但任务状态未对齐”，而不是静默少一项。
  - [ ] 保留当前用户已熟悉的主要按钮与路径，不改动核心交互顺序。

- [ ] Task 7: 回归验证关键链路
  - [ ] 场景 1：旧 `page_id` 失效但 `page-candidates` 自动恢复成功，进入工作台后可直接生成预览。
  - [ ] 场景 2：当前页签真实有效，点击“生成预览”后不会立刻跳回“重新选择页面”。
  - [ ] 场景 3：最新店铺配置里有 4 家店时，`切换店铺` 下拉栏完整展示 4 家。
  - [ ] 场景 4：某个店铺缺少任务记录时，系统能受控补齐或恢复，且该店铺可正常切换进入。
  - [ ] 场景 5：当前最新绑定真实失效，工作台直接进入“重新选择页面”，提示与按钮行为正确。
  - [ ] 场景 6：手动重新绑定后，预览、OCR 测试、保存进入下一家链路保持正常。
  - [ ] 场景 7：执行诊断检查与必要的前端冒烟，确认没有新增明显错误。
