# 店铺 Profile 逐店迁移执行方案 Spec

## Why
当前实库审计已确认没有目录共用，但存在多家店铺沿用历史 Profile 目录。需要把审计结果落成可执行的逐店迁移方案，明确哪些店保留旧目录、哪些店必须人工确认、哪些店在确认后才迁移，避免误伤真实登录态。

## What Changes
- 增加一份基于当前实库审计结果的逐店迁移执行方案
- 明确每家店铺的处理分组：保留旧目录、人工确认后再决定、确认后优先迁移
- 为每个迁移对象定义迁移前检查项、备份要求、执行步骤和验收标准
- 不直接修改数据库、不直接迁移目录、不自动切换会话绑定

## Impact
- Affected specs: Edge 会话隔离实库审计、Profile 目录迁移决策、真实登录态保护
- Affected code: `data/gmv_livelens.sqlite3`、`data/shops_default.json`、`backend/services/store.py`、`backend/services/shop_config.py`

## ADDED Requirements
### Requirement: 逐店迁移分组方案
系统 SHALL 基于当前实库审计结果，按店铺输出明确的迁移分组与处理建议。

#### Scenario: 输出店铺迁移分组
- **WHEN** 生成逐店迁移执行方案
- **THEN** 每家店都被归类为“保留旧目录”“人工确认后再决定”或“确认后优先迁移”
- **AND** 每条结论都附带当前目录、期望目录和风险说明

### Requirement: 迁移前人工确认清单
系统 SHALL 为需要人工确认的店铺提供统一的确认步骤，避免仅凭目录名误判真实登录态。

#### Scenario: 确认目标店铺身份
- **WHEN** 某家店被归类为“人工确认后再决定”或“确认后优先迁移”
- **THEN** 方案中给出必须核对的商家后台店铺名、主体身份和业务看板归属
- **AND** 说明哪些结果代表“继续保留旧目录”，哪些结果代表“进入迁移”

### Requirement: 迁移执行与验收口径
系统 SHALL 为需要迁移的店铺给出最小风险执行步骤与验收标准。

#### Scenario: 对单店执行目录迁移
- **WHEN** 用户确认某家店需要迁移到新目录
- **THEN** 方案中给出迁移前备份、停用条件、执行顺序和回滚关注点
- **AND** 方案中给出迁移后的登录、页面绑定、预览和采集验收口径

## MODIFIED Requirements
### Requirement: Edge 目录差异处理
现有“发现目录差异仅提示、不自动迁移”的策略保持不变，但补充一份人工执行所需的逐店迁移方案，作为后续操作依据。

## REMOVED Requirements
### Requirement: 自动批量迁移所有目录漂移店铺
**Reason**: 当前目录漂移店铺涉及真实登录态与平台风控，自动批量迁移风险过高。  
**Migration**: 先按逐店方案人工确认，再针对少量确认对象单独执行迁移。
