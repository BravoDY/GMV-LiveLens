# 跨电脑运行与可移植部署 Spec

## Why
当前项目虽然已经把大部分运行数据放在项目相对目录下，但仍存在当前电脑专属的绝对路径、`.venv` 固定启动方式、Edge 路径探测分散和安装步骤依赖人工记忆等问题，导致项目拷到另一台电脑后无法稳定复现。

用户真正需要的不是“理论上可迁移”，而是“换一台 Windows 电脑后，按固定步骤即可跑起来，并明确知道哪些步骤无法完全自动化”。

## What Changes
- 先在 `D:\` 下创建当前项目的完整备份副本，所有改造仅在备份副本内进行，避免污染现有工作目录
- 清理默认配置、脚本和文档中的当前机器绝对路径与固定目录假设
- 统一运行时路径解析与 Edge 可执行文件发现逻辑，避免多处重复硬编码
- 提供面向新电脑的首次安装与启动入口，覆盖虚拟环境、依赖安装、Playwright 浏览器组件安装和基础目录检查
- 提供启动前环境自检，明确提示缺失的依赖、浏览器、权限或不支持的平台
- 补充跨电脑迁移说明，区分“可自动化步骤”和“必须人工完成的步骤”
- 明确“开箱即用”的边界：目标是其它 Windows + Microsoft Edge 环境的一键初始化与启动，不承诺 macOS/Linux 兼容，也不承诺自动完成电商平台登录

## Impact
- Affected specs: 项目备份、部署启动、运行环境检测、默认配置、迁移文档、Edge 会话初始化
- Affected code: 备份副本根目录、`data/shops_default.json`、`backend/collectors/remote_edge.py`、`backend/tools/start_shop_edges.py`、`backend/services/shop_config.py`、`README.md`、`第1步_启动GMV服务.bat`

## ADDED Requirements
### Requirement: 改造必须基于独立备份副本
系统 SHALL 先在 `D:\` 创建当前项目的完整备份副本，并且后续开箱即用改造只允许发生在该备份副本内。

#### Scenario: 用户要求保留当前工作项目不被污染
- **WHEN** 用户要求先备份整个项目再改造
- **THEN** 系统会先创建可独立运行的项目副本
- **AND** 后续代码、脚本、文档和验证都基于该副本完成
- **AND** 原项目目录保持不变

### Requirement: 运行路径必须与机器目录解耦
系统 SHALL 使用项目根目录相对路径或运行时推导路径生成默认数据目录与示例配置，不得在默认配置中保留当前开发电脑的绝对路径。

#### Scenario: 在新电脑的任意目录解压项目
- **WHEN** 用户将项目放到任意磁盘和任意文件夹
- **THEN** 默认数据目录、店铺 Profile 目录和数据库目录仍能正确定位
- **AND** 系统不会因为旧电脑用户名、盘符或工作目录不同而启动失败

### Requirement: 启动入口必须支持首次安装
系统 SHALL 提供一个面向新电脑的统一启动入口，在检测到虚拟环境或依赖缺失时给出明确处理，尽量将首次安装步骤自动化。

#### Scenario: 新电脑首次运行项目
- **WHEN** 用户在未创建 `.venv` 的 Windows 电脑上执行推荐启动入口
- **THEN** 系统会提示或自动完成虚拟环境创建、依赖安装和 Playwright 组件安装
- **AND** 若某一步失败，输出可执行的下一步指引，而不是直接报错退出

### Requirement: 环境自检必须先于正式启动
系统 SHALL 在正式启动前检查操作系统、Python、Edge、Playwright 组件、端口占用和关键目录权限，并返回结构化结果。

#### Scenario: 新电脑缺少关键依赖
- **WHEN** 用户执行启动入口或手动执行环境自检
- **THEN** 系统能明确指出缺失项，例如未安装 Edge、未安装 Playwright 浏览器组件或端口被占用
- **AND** 每个失败项都附带可操作的修复建议

### Requirement: Edge 路径发现必须统一且可覆盖
系统 SHALL 将 Edge 可执行文件发现逻辑统一收敛到单一入口，并支持通过配置或环境变量覆盖默认探测结果。

#### Scenario: Edge 安装在非标准目录
- **WHEN** 用户电脑上的 Edge 不在默认安装路径，或企业环境使用自定义安装目录
- **THEN** 系统仍可通过配置值或环境变量找到可执行文件
- **AND** 不需要分别修改多个脚本或源码文件

### Requirement: 默认配置必须可迁移
系统 SHALL 保证示例店铺配置和初始化数据不包含开发机私有目录、用户名或不可复用的本地状态。

#### Scenario: 使用默认配置初始化
- **WHEN** 用户在新电脑上用默认配置初始化店铺和会话
- **THEN** 系统生成的 `user_data_dir`、数据目录和会话信息都基于当前机器运行环境
- **AND** 不会把旧机器的绝对路径写入数据库或配置

### Requirement: 部署文档必须明确开箱即用边界
系统 SHALL 在文档中清楚区分“可直接自动完成”的步骤与“必须人工完成”的步骤，并给出其它电脑上的标准执行流程。

#### Scenario: 用户评估是否能真正开箱即用
- **WHEN** 用户阅读部署文档
- **THEN** 能清楚知道项目当前能否做到开箱即用
- **AND** 能清楚知道仍需人工完成的平台登录、账号准备和 Windows + Edge 依赖前提

## MODIFIED Requirements
### Requirement: 快速启动说明
快速启动说明必须从“基于当前开发机路径的命令示例”改为“基于项目相对目录与统一启动入口的跨电脑步骤说明”。

## REMOVED Requirements
### Requirement: 默认配置依赖开发机绝对路径
**Reason**: 该约定会导致换电脑、换用户名、换盘符后直接失效，不符合可移植部署目标。
**Migration**: 将默认配置中的绝对路径改为运行时推导；已有数据库中的历史路径通过迁移脚本或启动时修复逻辑处理。
