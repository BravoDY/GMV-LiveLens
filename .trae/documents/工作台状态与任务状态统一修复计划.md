# 工作台状态与任务状态统一修复计划

## Summary
- 目标：修复“任务管理卡片显示 `已恢复到登录页`，但采集配置工作台仍显示 `待标定/已完成`”的状态不一致问题。
- 结论：当前采集配置工作台的阶段判断没有使用任务真实运行状态，只用“是否存在 `page_id` + 是否仍是默认标定框”来决定 `待绑定 / 待标定 / 已完成`，这会把“已绑定到登录页”“已绑定到非目标页”错误地算成可继续标定甚至已完成。
- 预期修复后：
  - 登录页绑定应明确显示为“待登录/待打开业务页”，不能再落到“待标定”或“已完成”。
  - 非目标页绑定应明确显示为“待切换业务页”。
  - 只有绑定到目标业务页后，才允许进入“待标定/已完成”的后续阶段。

## Current State Analysis

### 1. 当前现场事实
- 通过 `GET /api/tasks` 实际读取到：
  - `天猫 / 官方旗舰店`：`status = edge_login_page_bound`
  - `天猫 / 儿童旗舰店`：`status = edge_login_page_bound`
- 两家店当前 `page_url` 都是：
  - `https://sycm.taobao.com/custom/login.htm?...`
- 差异不在“是否恢复到登录页”，而在框选参数：
  - `官方旗舰店` 已有非默认标定框，因此工作台按旧逻辑判成“已完成”
  - `儿童旗舰店` 仍是默认框选，因此工作台按旧逻辑判成“待标定”

### 2. 代码根因
- 采集配置工作台阶段判断在 `frontend/core.js` 的 `setupStageMeta()`：
  - 无 `page_id` -> `待绑定`
  - 有 `page_id` 且仍是默认框选 -> `待标定`
  - 否则 -> `已完成`
- 这段逻辑没有读取任务真实状态：
  - `edge_login_page_bound`
  - `edge_page_bound`
  - `edge_target_page_ready`
- 因此“登录页/非目标页/目标业务页”被压扁成同一条链路，导致工作台与任务管理卡片状态语义冲突。

### 3. 影响范围
- `frontend/core.js`
  - `setupStageMeta()`
  - `buildSetupSummary()`
- `frontend/config.js`
  - 工作台当前步骤、下一步提示、摘要统计、按钮文案
- 必要时补充：
  - `frontend/app.js` 中涉及“进入下一家”“保存并进入下一家”的提示串
- 后端大概率不需要新增接口；当前 `task.status / last_reason / page_url` 已足够支持修复。

## Proposed Changes

### 1. 重构工作台阶段模型
- 文件：`frontend/core.js`
- 调整内容：
  - 扩展 `setupStageMeta(task)`，优先读取真实任务状态，而不是只看 `page_id` 和默认框选。
  - 建议阶段顺序：
    1. `pending_bind`：没有 `page_id`
    2. `pending_login`：`status === edge_login_page_bound`
    3. `pending_target_page`：`status === edge_page_bound`
    4. `pending_calibrate`：`status === edge_target_page_ready` 且仍是默认框选
    5. `completed`：`status === edge_target_page_ready` 且已完成非默认框选
  - 兼容处理：
    - `remote_page_not_found / edge_debug_* / edge_session_not_ready` 仍归入待处理队列，但不要误算成“待标定”
    - 老数据若没有新状态，则回退到现有逻辑，避免一次性破坏全部历史任务
- 目的：
  - 让工作台和任务卡片共享同一套业务语义
  - “已绑定登录页”不再等价于“可开始标定”

### 2. 修正摘要统计与待处理队列
- 文件：`frontend/core.js`
- 调整内容：
  - `buildSetupSummary()` 统计口径改为基于新阶段模型：
    - `待绑定`
    - `待登录/待业务页`
    - `待标定`
    - `已完成`
  - 如不想改 UI 指标字段数量，可采取兼容映射：
    - `pendingBind`：包含 `pending_bind + pending_login + pending_target_page`
    - `pendingCalibrate`：仅统计真正已到目标业务页但尚未完成框选的任务
  - `setupQueue` 排序建议：
    - 先处理 `pending_bind`
    - 再处理 `pending_login`
    - 再处理 `pending_target_page`
    - 最后才是 `pending_calibrate`
- 目的：
  - “进入下一家”优先把真正阻塞流程的问题往前排
  - 避免把登录页店铺误导成“只差框选”

### 3. 让工作台文案说人话
- 文件：`frontend/config.js`
- 调整内容：
  - `renderSetupWorkbench()`、`setupFlowState()`、步骤提示改为按新阶段输出：
    - `pending_login`：当前会话已恢复，但当前绑定页是登录页。请先在 Edge 中完成登录并打开目标业务页。
    - `pending_target_page`：当前已绑定其它页，不是目标业务页。请切换到目标业务页后重新绑定或再次显示 Edge。
    - `pending_calibrate`：当前已到目标业务页，请开始框选 GMV 区域。
    - `completed`：当前店铺已完成业务页绑定与框选，可直接测试识别或进入下一家。
  - “步骤 2 / 步骤 3”标题要随阶段变化，避免登录页场景还写“生成预览”。
  - `setupSummaryLine` 文案同步更新，至少保证：
    - 不再把登录页店铺归类成“待标定”
    - 若保留原统计字段名，文案里明确“待绑定含登录页/待切业务页”
- 目的：
  - 用户一眼能分清是“还没登录”、还是“没打开业务页”、还是“只差框选”

### 4. 校正切店下拉与当前店状态标签
- 文件：`frontend/config.js`
- 调整内容：
  - 店铺切换下拉中的状态标签不要只显示 `[待标定] / [已完成]`
  - 改为复用 `setupStageMeta()` 的 label，例如：
    - `[待登录]`
    - `[待切业务页]`
    - `[待标定]`
    - `[已完成]`
- 目的：
  - 避免当前卡片和下拉列表继续出现相互矛盾的状态

### 5. 自检历史兼容与最小侵入
- 文件：`frontend/core.js`, `frontend/config.js`
- 调整内容：
  - 对缺失 `task.status` 的历史任务保留旧兜底逻辑
  - 不修改后端数据结构，不新增数据库字段
  - 仅把前端工作台的阶段判断与后端已有状态语义对齐
- 目的：
  - 用最小代价完成统一，降低回归风险

## Assumptions & Decisions
- 决策 1：以任务真实状态 `task.status` 作为工作台主判据，默认框选只用于区分“待标定”与“已完成”。
- 决策 2：登录页绑定不是“已完成”，也不是“待标定”，而是新的待处理阶段。
- 决策 3：其它非目标页绑定同样不允许进入“待标定/已完成”。
- 决策 4：本次优先修工作台状态一致性，不在此计划内追加“自动打开目标业务页”。
- 假设：后端持续会返回已实现的状态值：
  - `edge_login_page_bound`
  - `edge_page_bound`
  - `edge_target_page_ready`

## Verification Steps
- 代码级验证：
  - 检查 `setupStageMeta()` 是否优先消费 `task.status`
  - 检查 `buildSetupSummary()` 是否不再把登录页任务算入“待标定/已完成”
- 场景验证：
  1. `官方旗舰店`
     - 当前为 `edge_login_page_bound`
     - 工作台应显示 `待登录` 或等价直白文案，而不是 `已完成`
  2. `儿童旗舰店`
     - 当前为 `edge_login_page_bound`
     - 工作台应显示 `待登录`，而不是 `待标定`
  3. 绑定到目标业务页但仍是默认框选
     - 应显示 `待标定`
  4. 绑定到目标业务页且已完成非默认框选
     - 应显示 `已完成`
  5. 没有 `page_id`
     - 仍显示 `待绑定`
- UI 验证：
  - 工作台顶部当前状态、下一步提示、摘要统计、切店下拉标签全部一致
  - 不再出现“任务卡片显示登录页，工作台却显示待标定/已完成”的冲突
