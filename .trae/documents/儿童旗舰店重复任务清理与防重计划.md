# 儿童旗舰店重复任务清理与防重计划

## Summary
- 目标：解决“天猫 / 儿童旗舰店”在任务管理中出现两个任务的问题，并确保后续再从 `shops.csv` 同步时不再残留/重新生成重复任务。
- 用户偏好：对重复任务采用“直接删除重复”方案，不保留多余重复任务记录。
- 成功标准：
  - 任务管理中同一 `platform + shop_name` 只保留一个“儿童旗舰店”任务。
  - `shops.csv` / `shops_default.json` 当前只有一条配置时，数据库内不再残留同店铺重复任务。
  - `/api/shops/init` 或启动同步后，不会再次让重复任务回到界面。
  - 删除后，当前店铺的会话、页面绑定、预览与 OCR 主链路仍正常。

## Current State Analysis
- `shop_config.load_shop_configs()` 在读取 `shops.csv` / `shops_default.json` 时会对 `(platform, shop_name)` 做去重校验；若配置文件里重复，会直接抛出 `duplicate shop config`，见 `backend/services/shop_config.py`。
- 当前任务同步逻辑 `sync_tasks_with_shop_configs()` 只会：
  - 优先按 `edge_session_id` 找现有任务；
  - 其次按 `(platform, shop_name)` 找现有任务；
  - 找到一个后更新，找不到才创建；
  - 但**不会扫描并清理数据库中已经存在的重复任务**，见 `backend/services/store.py`。
- `capture_tasks` 表当前没有对 `(platform, shop_name)` 或 `edge_session_id` 建唯一约束，见 `backend/services/store.py` 的 `CREATE TABLE capture_tasks`。
- 任务列表 `list_tasks()` 是按 `sort_order, id` 全量读取，因此数据库里若已有两个同店铺任务，前端一定会如实显示两个，见 `backend/services/store.py`。
- 应用启动时会自动执行 `store.sync_tasks_with_shop_configs()`，并且 `/api/shops/init` 也会再次执行同一同步逻辑，所以如果不同步层不具备“防重清理”能力，历史重复任务会长期留存，见 `backend/main.py`。
- 结合以上事实，可以判断：
  - 你“最新填报只有一个儿童旗舰店”是成立的；
  - 现在的两个任务不是来自当前配置文件，而是数据库历史遗留重复任务；
  - 仅靠现有同步逻辑不会自动修复，必须补一层“识别主任务 + 删除重复任务 + 后续防重”的能力。

## Proposed Changes
### 1. 后端：补充重复任务识别与清理入口
- 文件：`backend/services/store.py`
- 变更内容：
  - 新增“按 `(platform, shop_name)` 分组识别重复任务”的辅助逻辑。
  - 为每组重复任务定义“主任务”判定规则，优先保留当前最可信的那一条，删除其余重复项。
  - 主任务判定建议按以下优先级排序：
    - 已绑定页面且 `page_id` 非空；
    - 状态/运行时信息更完整（如有 `last_success_at`、`last_sample_at`）；
    - `sort_order` 更贴近当前配置；
    - `id` 更大或更小作为最终稳定兜底（二选一，执行时固定一种）。
  - 删除重复任务时，同时清理其关联 `gmv_samples`，统一走现有 `delete_task(task_id)`，避免孤儿样本。
- 设计原因：
  - 需要一个明确、可复用的“数据库去重”能力，而不是只在前端隐藏。
  - 复用现有 `delete_task()` 可避免遗漏样本表清理。

### 2. 后端：把“防重清理”接入店铺同步主链路
- 文件：`backend/services/store.py`
- 变更内容：
  - 在 `sync_tasks_with_shop_configs()` 中加入重复任务处理阶段。
  - 推荐顺序：
    - 先读取当前配置；
    - 先做常规 create/update；
    - 再按 `(platform, shop_name)` 对数据库任务做一次定向去重；
    - 仅对“当前配置仍存在的店铺”强制去重，避免误删完全脱离当前配置、但用户可能仍想保留的历史任务。
  - 返回值新增 `deleted_duplicates` 或同类统计字段，便于前端/API 提示同步结果。
- 设计原因：
  - 启动同步和 `/api/shops/init` 都走这里，防重逻辑接在这里才能真正闭环。
  - 只对“当前配置覆盖到的店铺”去重，可降低误删历史任务的风险。

### 3. 后端：视需要补充单次清理/诊断 API
- 文件：`backend/main.py`
- 变更内容：
  - 评估是否复用 `/api/shops/init` 即可完成清理。
  - 若现有接口返回信息不足，则扩展 `/api/shops/init` 返回：
    - `created`
    - `updated`
    - `unchanged`
    - `deleted_duplicates`
    - `deduped_shops`
  - 不新增多余接口，优先维持最小改动。
- 设计原因：
  - 用户已经有现成的“从店铺配置同步”入口，优先复用，避免再添一条新维护入口。

### 4. 前端：让同步结果对用户可见
- 文件：`frontend` 中当前触发 `/api/shops/init` 的相关模块（实施时按实际调用点定位）
- 变更内容：
  - 在用户触发“按店铺配置初始化/同步任务”后，消息提示中补充“已删除 N 个重复任务”。
  - 若本次清理命中了“儿童旗舰店”，在提示中明确展示店铺名，方便用户确认问题已修复。
- 设计原因：
  - 这次是用户明确感知到的问题，清理结果需要可见，避免“修了但用户不知道是否真的生效”。

### 5. 可选：数据库层增加软保护，避免未来再写入同店铺重复任务
- 文件：`backend/services/store.py`
- 变更内容：
  - 先不急于直接上 SQLite 唯一索引迁移，因为当前库里已有历史脏数据，直接加唯一约束风险较高。
  - 本次先通过代码层防重：
    - `sync_tasks_with_shop_configs()` 内部先匹配/去重；
    - 若后续还存在其它写入口创建重复任务，再单独补一层写入前校验。
- 设计原因：
  - 你要求“不能影响这个已跑通的项目”，所以这次先采用低风险、可回滚的代码层去重，不先做硬约束迁移。

## Assumptions & Decisions
- 决策：重复任务处理采用“直接删除重复”，不做归档。
- 决策：本次只针对“当前配置中仍存在的店铺”执行自动去重，避免误删彻底脱离当前配置的历史任务。
- 决策：去重以数据库中的 `(platform, shop_name)` 为主键语义，而不是只靠 `edge_session_id`，因为当前用户关注的是“同一个店铺出现两个任务”。
- 决策：主任务保留策略必须稳定且可解释，优先保留已绑定页面、最近仍有使用痕迹的任务。
- 假设：当前“儿童旗舰店”两个任务中至少有一条是历史遗留重复项；这与已有代码结构和现象一致。
- 约束：不能通过简单前端隐藏来“假装修好”，必须从数据层真正去重。

## Verification Steps
### A. 代码级验证
- 检查 `shops.csv` / `shop_config.load_shop_configs()` 在只有一条“儿童旗舰店”配置时，仍只生成一条配置对象。
- 检查 `sync_tasks_with_shop_configs()` 在存在重复数据库任务时，返回值中能体现重复清理结果。
- 检查删除逻辑统一走 `delete_task()`，确保 `gmv_samples` 一并清理。

### B. 数据与接口验证
- 造或复用一个“同 platform + shop_name 存在两个任务”的场景。
- 执行同步入口（优先 `/api/shops/init` 或启动同步）。
- 确认同步后：
  - `capture_tasks` 中“天猫 / 儿童旗舰店”只剩一条；
  - 接口返回包含删除重复数量；
  - 任务快照里不再出现重复任务。

### C. 前端冒烟测试
- 打开“任务管理”，确认“儿童旗舰店”只剩一张卡片。
- 打开“采集配置”，确认切换店铺下拉框里“儿童旗舰店”只剩一个任务项。
- 对保留下来的“儿童旗舰店”任务执行主链路冒烟：
  - 选择当前店铺 Edge 会话；
  - 扫描页签；
  - 生成预览；
  - 测试 OCR 或保存任务；
  - 确认链路未因去重导致绑定丢失或报错。

### D. 回归边界
- 验证其他店铺不被误删。
- 验证启动同步后不会再次生成新的重复“儿童旗舰店”任务。
- 验证现有任务管理、实时看板、调度器读取任务列表不因去重统计字段变更而出错。
