# 更改 GMV 呈现与确认逻辑计划

## 1. 目标与需求分析
用户希望修改目前的 GMV 呈现判断逻辑（即 `scheduler.py` 中的 `_judge` 方法）：
1. **取消数值下降的连续确认机制**：不再尝试通过多次确认来接受下降的值。
2. **同一天内只增不减**：如果 `后采集 < 前采集`（退款或截断误差），只要还在同一天，直接忽略并丢弃该值，保持原有的最高值。
3. **跨天复位**：如果发生跨天（新一天的采集），即使数值下降也立即接受作为新一天的起始值。
4. **保留大额跳变防抖**：对于数值变大，如果是正常增长则走默认确认逻辑；如果是异常大额跳变（如突然大5倍），依然保留防误识别的连续确认机制。

## 2. 涉及文件
- `backend/services/scheduler.py`

## 3. 具体修改内容
修改 `CaptureScheduler._judge` 方法：
1. 引入 `datetime` 模块用于判断是否跨天。
2. 移除原有的 `is_abnormal` 包含下降和上升两种情况的逻辑。
3. 增加 `is_cross_day` 的判断：比较 `time.time()` 与 `task.last_success_at` 的日期。
4. 针对 `selected < last` 的情况：
   - 如果 `is_cross_day` 为真：直接返回 `ok`，重置并接受最新的跨天 GMV。
   - 如果 `is_cross_day` 为假：直接返回 `ignored`，并附带原因“同一天数值下降...已忽略”，传入的 `trusted` 和 `pending_value` 均为 `None`。
5. 将原来的跳变判定更名为 `is_abnormal_jump`，仅在 `selected > last * 5` 时触发，并走原有的连续确认防抖逻辑。

## 4. 验证步骤
1. 查看并更新 `backend/services/scheduler.py` 的代码。
2. 执行测试或重启检查，验证逻辑正常加载无语法错误。
3. 确保跨天逻辑能够正确重置数值，同一天的退款数值不再更新到看板。
