# 重构洞察配置分析与建议

## 一、是否应该开启？

**建议：开启。**

理由如下：

1. **零成本的第二双眼睛**：重构洞察不会自动修改代码，只是提供建议。你作为最终决策者，每一条建议都可以选择采纳或忽略，没有强制性风险。

2. **对单人项目尤其有价值**：GMV-LiveLens 目前是单人维护的项目，缺少 Code Review 环节。重构洞察可以在一定程度上充当"自动化 Code Review"，帮你发现自己可能忽略的代码坏味道。

3. **渐进式学习价值**：即使你不采纳某些建议，看到 AI 为什么会提出这些建议，本身也是一种对代码质量的"思维校验"。

4. **前提条件已满足**：你的项目已有完整的 CI 检查（ruff、smoke_api、smoke_edge），即使采纳重构建议后出现问题，也能快速通过 CI 发现。

---

## 二、重构风格选哪个？

**建议：保守（Conservative）。**

逐选项分析：

| 风格 | 适合场景 | 不适合 GMV-LiveLens 的原因 |
|------|----------|---------------------------|
| **保守** | 生产环境、频繁迭代的项目 | ✅ 最适配 |
| **中性** | 相对稳定、有团队协作的中型项目 | 改动范围可能超出预期 |
| **激进** | 早期原型、重构窗口期、有充分测试覆盖 | ❌ 高风险，可能建议大规模架构变更 |

**核心判断依据**：

1. **GMV-LiveLens 是生产工具**，不是实验性项目。采集链路（Edge/CDP → OCR → SQLite → WebSocket）涉及真实店铺登录态和数据采集，任何"深度优化"带来的回归风险都是不可接受的。

2. **你的项目已有明确的架构约定**（如 `remote_edge` 已拆分为 `edge/` 子包、路由已拆到 `routers/`），激进模式可能会建议进一步拆分或引入新抽象，但当前架构已经足够清晰，不需要大动。

3. **保守模式的渐进式优化**与你现有的"最小改动提升代码质量"原则一致。它更可能指出：重复代码抽取、异常处理完善、命名优化、死代码清理等低风险改进。

## 三、自定义偏好怎么写？

**建议内容**（通用版，约 200 字，在 500 字限制内）：

```
优先级排序：
1. 优先关注边界情况与异常处理：网络超时、外部服务调用失败、文件读写异常、进程/资源泄漏、并发竞争等场景的错误处理是否完善。
2. 关注数据安全与副作用：涉及配置文件、持久化数据、认证凭据的读写操作是否有足够保护，是否存在意外的数据覆盖或丢失风险。
3. 关注重复代码与抽取机会：跨模块、跨平台、跨业务线之间是否有可复用的公共逻辑，是否存在复制粘贴导致的代码发散。
4. 关注命名清晰度与业务语义：变量、函数、类、模块名是否准确表达其业务含义，能否让后续维护者一眼看懂。
5. 暂不关注：引入新框架/新依赖、大规模架构重组、过度设计的设计模式套用。
```

**为什么这样写**：

- **第1条（边界与异常）**：大多数生产 Bug 都出现在边界条件而非主流程，提前发现异常处理缺口是性价比最高的质量投资。
- **第2条（数据安全）**：配置文件、持久化数据和凭据一旦被误操作，影响面往往不可逆，应优先保护。
- **第3条（重复代码）**：复制粘贴是技术债务的主要来源，及时抽取公共逻辑能避免后续多处分叉维护。
- **第4条（命名质量）**：好的命名是最低成本的文档，让代码自解释，降低沟通和维护成本。
- **第5条（明确排除项）**：告诉 AI 不要在这些方向上浪费时间，避免收到大量不相关的"激进"建议。

---

## 四、总结

| 决策项 | 推荐值 | 核心理由 |
|--------|--------|----------|
| 是否开启 | **开启** | 零成本自动化 Code Review，单人项目的第二双眼睛 |
| 重构风格 | **保守** | 生产工具不能承受回归风险，渐进优化最稳妥 |
| 自定义偏好 | **如上文** | 聚焦边界/异常/数据安全/重复代码，排除架构大改 |

---

## 五、使用建议

1. **初期观察**：开启后先观察 1-2 周，看看建议的质量和噪音比。如果大部分建议都很合理且有用，就继续开着；如果大量噪音，可以关闭或调整偏好。
2. **不盲从**：每条建议都要结合你对项目业务的理解来判断，AI 不知道"这个 OCR 误识别特例为什么要这样处理"。
3. **结合 CI**：采纳建议后跑一次 `scripts/ci_check.py`，确保没有引入回归。
