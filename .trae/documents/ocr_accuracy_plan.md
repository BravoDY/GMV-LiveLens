# GMV-LiveLens OCR 精度提升计划

## 1. 摘要
当前项目在采集 GMV 时，OCR 引擎有时会将数字 `0` 错误识别为英文字母 `o` 或 `O`，导致数字被异常截断或无法识别，从而触发 `suspect` 或 `parse_failed`。通过扫描数据库中已有的历史样本 (`gmv_samples` 表)，我们发现除了 `o`/`O` 被误认为 `0` 外，还存在 `l`/`I`/`i` 被误认为 `1`，以及 `D` 被误认为 `0` 的情况。此外，货币符号 `￥` 极易被误识别为汉字 `半`、`旧`、`举`、`忙`、`夫`。本计划旨在通过增加文本纠错规则和扩展正则匹配，大幅提升 OCR 候选金额提取的准确率和评分。

## 2. 现状分析
1. **数字内部断裂**：目前 `AMOUNT_PATTERN` 的正则表达式 `([0-9][0-9,\uff0c.\s]{2,})` 严格要求内容为数字、逗号、点或空格。如果 OCR 输出了 `￥3o,000`，正则会在 `3` 处因为遇到 `o` 而中断，导致整个金额完全无法被提取。如果输出 `半1051,l11`，则只能提取出 `1051`，导致金额剧烈缩水。
2. **货币符号漏判**：目前 `_candidate_reason` 仅对 `\u00a5|\uffe5|RMB|CNY|\u7f8a|Y` 赋予 `+35` 的强特征加分。如果 `￥` 被识别成了 `半`（在样本中出现超过 500 次），该候选值将丢失这关键的 35 分，可能导致正确数值落选。

## 3. 提议的变更
**修改文件**：`backend/collectors/ocr_reader.py`

### 3.1 增加 OCR 结果预纠错规则
在 `extract_candidates` 函数的开头，对 `ocr_text` 以及 `details` 中的 `text` 字段进行安全的全局字符替换，修正常见的形近字错误：
- 将小写字母 `o`、大写字母 `O`、大写字母 `D` 替换为数字 `0`
- 将小写字母 `l`、大写字母 `I`、小写字母 `i` 替换为数字 `1`
*(注：电商 GMV 截图中的文字极少包含这些孤立的英文字母，且中文提示词"成交金额"、单位"万/亿"也不受影响，全局替换非常安全。)*

### 3.2 扩展 `AMOUNT_PATTERN` 正则表达式
在文件顶部的 `AMOUNT_PATTERN` 中，将常见的误识别货币符号加入到非捕获组中：
```python
# 原版:
# r"(?:(?:RMB|CNY|\u00a5|\uffe5|\u7f8a|Y)\s*)?([0-9][0-9,\uff0c.\s]{2,})(?:\s*([\u4e07\u4ebf]))?"
# 新版:
r"(?:(?:RMB|CNY|\u00a5|\uffe5|\u7f8a|Y|半|旧|举|忙|夫)\s*)?([0-9][0-9,\uff0c.\s]{2,})(?:\s*([\u4e07\u4ebf]))?"
```

### 3.3 扩展 `_candidate_reason` 的货币符号匹配
在 `_candidate_reason` 函数内，同步更新判断货币符号加分的正则：
```python
# 原版:
# if re.search(r"\u00a5|\uffe5|RMB|CNY", raw, re.IGNORECASE):
# 新版:
if re.search(r"\u00a5|\uffe5|RMB|CNY|\u7f8a|Y|半|旧|举|忙|夫", raw, re.IGNORECASE):
```

## 4. 假设与决策
- **假设 1**：由于处理的文本范围限定于已经过裁剪的纯数字/短文字区域，对 `o, O, l, I, i, D` 进行全局替换是安全的，不会破坏其他有价值的文字结构。
- **决策 1**：使用简单高效的 `.replace()` 方法进行预处理，而不在正则中引入复杂的条件匹配，从而保证性能和代码的可读性。

## 5. 验证步骤
1. 代码修改完成后，使用 `RunCommand` 在命令行执行针对问题文本（如 `￥3o,000` 和 `半1051,l11`）的 `extract_candidates` 单元测试。
2. 确认 `￥3o,000` 能被正确提取为 `30000` 候选值。
3. 确认 `半1051,l11` 能被正确提取为 `1051111`，并且获得 `currency` 的评分（即能看到加分且 reason 中包含 `currency`）。
4. 通过 `POST /api/test-ocr` 在前端也可进行类似截图的验证。
