## Summary

- 目标：将 `采集配置工作台` 从“多区块、多按钮、多分叉路径”收敛为一条面向 `真实 Edge` 的极简主流程，让用户始终清楚“当前在哪一步、下一步点哪里”。
- 用户偏好已锁定：
  - 只走 `真实 Edge` 路线，不再考虑 `window_capture` 兼容模式作为工作台主路径
  - `Edge` 的启动/显示/隐藏/关闭只保留在 `任务管理`，不在 `采集配置工作台` 中出现
  - 采用“固定双栏常驻”布局：左侧操作区、右侧截图标定区，移除宽度拖拽条，尽量避免上下拖拉页面
- 成功标准：
  - 工作台只保留三步主流程，用户无需判断多个平行按钮该先点哪一个
  - 工作台不再出现“启动 Edge”这类与当前标定无关的控制
  - 截图标定区在配置过程中持续可见，不需要上下滚动页面去找预览
  - 右侧截图标定是主视图，左侧仅保留必要操作与当前任务上下文

## Current State Analysis

- 当前 `采集配置` 区域由左侧配置面板 + 中间宽度拖拽条 + 右侧截图预览组成：
  - `frontend/index.html`
  - 结构为 `.config-panel`、`#configResizeHandle`、`.preview-panel`
- 当前左侧工作台内容过多，已形成多个平行决策点：
  - `frontend/index.html`
  - 同时包含：
    - 配置摘要
    - 当前店铺卡片
    - 快速初始化
    - 绑定当前店铺页面
    - 标定并保存
    - 真实 Edge 面板（由脚本动态插入）
    - 兼容窗口模式表单
  - 用户确实很难判断先操作哪个按钮
- 当前工作台仍保留与“只走真实 Edge”相冲突的通道：
  - `frontend/index.html`
  - `采集模式` 下拉依然同时提供：
    - `remote_edge`
    - `window_capture`
  - `windowCompatBox` 仍在页面中保留
- 当前脚本中仍初始化并绑定了兼容窗口模式的相关行为：
  - `frontend/app.js`
  - `setupEvents()` 中仍绑定：
    - `refreshWindows`
    - `previewWindow`
    - `captureMode` 切换显示逻辑
  - 文件底部仍执行：
    - `refreshWindows()`
    - `captureMode.dispatchEvent(...)`
- 当前真实 Edge 路线同时存在两套操作入口：
  - `frontend/app.js`
  - `任务管理` 中每个任务和每个平台卡片都有 `启动 / 显示 / 隐藏 / 关闭 Edge`
  - `采集配置工作台` 中还会插入真实 Edge 面板，形成重复控制入口
- 当前布局确实会迫使用户处理拖拽和滚动：
  - `frontend/styles.css`
  - `.config-layout` 使用 `grid-template-columns: var(--config-panel-width, 600px) 10px minmax(0, 1fr)`
  - 中间存在 `.config-resize-handle`
  - `.preview-wrap` 使用 `overflow: auto` 和 `max-height: calc(100vh - 210px)`
  - 这意味着截图标定虽然有独立滚动容器，但配置区内容变长时，用户仍会在“看表单”和“看截图”之间来回移动视线
- 当前右侧截图区并没有被设计成绝对优先视图：
  - `frontend/index.html`
  - 左侧配置面板内容远多于右侧控制，且标定按钮仍位于左侧底部
  - 用户在框选时，需要回到左侧较下方去完成测试与保存
- 当前后端支持真实 Edge 的完整链路，足够支撑极简前端：
  - `backend/main.py`
  - 已有接口包括：
    - `GET /api/edge-sessions`
    - `GET /api/edge-sessions/{session_id}/pages`
    - `POST /api/edge-sessions/{session_id}/pages/{page_id}/preview`
    - `POST /api/tasks`
    - `POST /api/tasks/{task_id}/capture-once`
    - 平台/任务级别的 Edge 启停接口
  - 因此本次简化重点应放在前端入口裁剪，而不是后端重做

## Proposed Changes

### 1. 将工作台收敛为“三步主流程”，隐藏次要区块与分叉路径

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
- 改动内容：
  - 将当前左侧工作台重构为三个明确步骤：
    - `步骤 1：选择当前任务页面`
    - `步骤 2：截图标定`
    - `步骤 3：保存并处理下一家`
  - 删除或隐藏以下工作台元素：
    - `快速初始化` 独立说明块中的冗余说明
    - 当前“绑定当前店铺页面”中的次要说明文本
    - “下一家待配置 / 重新扫描 / 清空标定”三个并列快捷入口的并列暴露方式
  - 将“当前店铺”信息与“当前步骤提示”合并成一个顶部状态条，只展示：
    - 当前店铺
    - 当前步骤
    - 当前页面是否已绑定
    - 下一步主动作
- 为什么这样做：
  - 现在的复杂感主要来自同屏出现多个同等级按钮
  - 三步主流程可以把“当前该做什么”直接显式化
- 实现方式：
  - 通过 `app.js` 渲染唯一主流程状态
  - 页面只在当前步骤显示当前需要的按钮，其他按钮收起或次级化

### 2. 工作台只保留真实 Edge 主路径，移除兼容模式入口

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
- 改动内容：
  - 在工作台中移除 `captureMode` 下拉的用户选择权，固定为 `remote_edge`
  - 移除 `windowCompatBox` 在工作台中的显示和交互
  - 删除与 `window_capture` 工作台流程相关的初始化与事件绑定：
    - `refreshWindows`
    - `previewWindow`
    - `captureMode` 切换显示逻辑
  - 保留后端和任务管理对 `window_capture` 的兼容能力，但不让其继续干扰工作台主界面
- 为什么这样做：
  - 用户明确表示这次只走 `真实 Edge`
  - 继续显示双模式只会增加理解成本
- 实现方式：
  - 工作台层面固定 `capture_mode = remote_edge`
  - 若任务管理中仍存在旧的 `window_capture` 任务，只在任务管理编辑时做兼容处理，不在主工作台展示

### 3. 将真实 Edge 的启停控制彻底移出工作台，只留“当前任务页面选择”

- 文件：
  - `frontend/app.js`
  - `frontend/index.html`
- 改动内容：
  - 删除或隐藏 `setupRemoteEdgeUi()` 在 `采集配置工作台` 中插入的完整真实 Edge 操作面板
  - 工作台中只保留与当前任务直接相关的两类信息：
    - 当前任务绑定的 `Edge` 会话
    - 当前任务可选页面列表 / 扫描结果
  - 不在工作台中展示：
    - 启动 Edge
    - 显示 Edge
    - 隐藏 Edge
    - 关闭 Edge
  - 这些能力继续保留在 `任务管理` 中
- 为什么这样做：
  - 用户已经明确 Edge 控制只想留在 `任务管理`
  - 工作台只应承担“选页、预览、框选、保存”四个动作
- 实现方式：
  - 将 `scanBind()` 保留为“刷新当前任务候选页面”
  - 将页面绑定结果直接映射到当前任务，不再暴露额外的会话运维控件

### 4. 将多个快捷按钮收敛为“每步一个主按钮”

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
- 改动内容：
  - 重新设计按钮层级：
    - 步骤 1 主按钮：`扫描当前店铺页面`
    - 步骤 1 结果确认按钮：`使用此页面`
    - 步骤 2 主按钮：`生成预览`
    - 步骤 3 主按钮：`保存并进入下一家`
  - 将以下按钮降级为文本链接、次级按钮或仅在特定状态出现：
    - `下一家待配置`
    - `重新扫描本店铺页面`
    - `清空当前标定并重来`
    - `测试识别`
  - `测试识别` 改为仅在已框选后显示，作为保存前校验项，而不是长期主按钮
- 为什么这样做：
  - 当前最大问题是“永远有很多按钮在等你点”
  - 每一步只暴露一个主按钮，用户决策会明显简单
- 实现方式：
  - 由 `setupStageMeta()` 和新的工作流渲染函数决定当前主按钮
  - 非当前步骤的按钮统一折叠

### 5. 将布局改为固定双栏常驻，移除宽度拖拽条和工作台内部纵向竞争

- 文件：
  - `frontend/index.html`
  - `frontend/styles.css`
  - `frontend/app.js`
- 改动内容：
  - 删除 `#configResizeHandle` 和 `setupConfigResize()` 初始化
  - 将 `.config-layout` 改为固定双栏：
    - 左侧为窄操作栏
    - 右侧为更宽的截图标定区
  - 左侧操作区高度控制在首屏内，避免需要上下长滚动
  - 右侧截图标定区改为更强的常驻视图，优先保证截图与框选区域可见
  - 优化 `preview-wrap` 的高度策略，让标定区尽量占满可视空间
- 为什么这样做：
  - 用户明确不希望“对页面进行上下拖拉”
  - 当前宽度拖拽条和长表单会破坏“持续盯住截图标定”的体验
- 实现方式：
  - 采用固定列宽比例，而不是可拖拽宽度
  - 对左侧信息密度做压缩，优先减少高度而不是继续堆区块

### 6. 将“当前步骤提示”做成强引导文案，而不是状态统计板

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
  - `frontend/styles.css`
- 改动内容：
  - 弱化当前四个统计卡片：
    - `总店铺`
    - `待绑定`
    - `待标定`
    - `已完成`
  - 将顶部区域改为一个“当前步骤卡”，直接告诉用户：
    - 你现在正在配置哪家店铺
    - 当前处于哪一步
    - 你下一步只需要点击哪个按钮
  - 统计信息若保留，则改成单行简要文本，而不是四张并列卡片
- 为什么这样做：
  - 统计卡片提供的是“全局概况”，不是“当前操作引导”
  - 用户的问题核心是找不到下一步，而不是看不到总数
- 实现方式：
  - 用一步一提示替换四宫格摘要
  - 保留待处理数量作为辅助信息即可

### 7. 将“测试识别”和“保存”靠近截图标定区，减少视线来回切换

- 文件：
  - `frontend/index.html`
  - `frontend/styles.css`
- 改动内容：
  - 将当前底部的 `测试识别 / 保存并启用` 操作重新布局：
    - 贴近截图标定区顶部或底部操作条
    - 不再埋在左侧长表单末尾
  - OCR 结果反馈区域与截图标定区放在同一视觉上下文中
- 为什么这样做：
  - 用户在框选后最关注的是“这个框识别是否正确”
  - 相关按钮和反馈应当离截图更近
- 实现方式：
  - 调整 DOM 位置或改为右侧标定区工具条
  - 保持按钮数量最少，但与标定动作强关联

### 8. 保留任务管理作为 Edge 运维入口，工作台只做配置闭环

- 文件：
  - `frontend/app.js`
  - `frontend/index.html`
- 改动内容：
  - 保持 `任务管理` 中的平台级和任务级 `Edge` 控制按钮不动
  - 从工作台发起配置时，默认假设会话已由任务管理准备好
  - 如果当前任务没有可用的 `edge_session_id` 或页面列表为空：
    - 工作台只提示用户回到 `任务管理` 处理会话
    - 不在工作台内再提供运维按钮
- 为什么这样做：
  - 这与用户“Edge 控制只留任务管理”的选择完全一致
  - 能把“会话运维”和“截图标定”两类心智彻底分离

## Assumptions & Decisions

- 本次方案只优化 `真实 Edge` 配置体验，不扩展新的采集模式。
- `window_capture` 的后端兼容能力不删除，但其工作台入口会被隐藏或退出主路径。
- `任务管理` 继续承担：
  - 启动 Edge
  - 显示 Edge
  - 隐藏 Edge
  - 关闭 Edge
- `采集配置工作台` 只承担：
  - 当前店铺上下文展示
  - 候选页面选择
  - 预览截图
  - 框选标定
  - 保存并进入下一家
- 当前统计卡片不是本次保留重点；若与空间冲突，优先删减统计展示而不是压缩截图区。
- 当前宽度拖拽条属于明确反目标，本次必须移除。
- 当前页面应尽量做到“首屏内可完成主要动作”；若完全无滚动不可实现，也优先保证：
  - 截图区常驻
  - 左侧不出现长链路分散控件

## Verification Steps

- 路径验证：
  - 打开 `采集配置工作台`
  - 确认页面只呈现三步主流程，而不是多个平行操作区
  - 确认当前步骤提示清楚写明“下一步做什么”
- 模式验证：
  - 确认工作台中不再暴露 `window_capture` 入口
  - 确认兼容窗口相关控件不再出现在工作台首屏
- Edge 入口验证：
  - 确认工作台不再显示 `启动 / 显示 / 隐藏 / 关闭 Edge`
  - 确认这些按钮仍可在 `任务管理` 中使用
- 布局验证：
  - 确认中间宽度拖拽条已移除
  - 确认右侧截图标定区始终可见，用户不需要上下拖拉页面寻找截图
  - 确认左侧操作区高度显著缩短
- 标定验证：
  - 选择页面后生成预览
  - 框选区域后能直接看到 OCR 校验入口和保存入口
  - 保存后自动进入下一家待处理店铺
- 边界验证：
  - 当前任务没有可用页面时，工作台明确提示回 `任务管理` 处理，而不是展示无关按钮
  - 自动推进失败时，仍能明确停留在当前步骤，不让用户迷失

## Implementation Order

1. 重构 `frontend/index.html`，把工作台改成三步主流程，并移除宽度拖拽条。
2. 调整 `frontend/styles.css`，实现固定双栏、缩短左侧操作区、增强右侧截图区的常驻显示。
3. 精简 `frontend/app.js` 中的工作台状态渲染，改为“当前步骤 + 单一主按钮”驱动。
4. 从工作台逻辑中移除 `window_capture` 主路径与真实 Edge 运维控件，仅保留当前任务页面选择与预览链路。
5. 调整 `测试识别 / 保存` 的位置和显示时机，使其紧贴截图标定过程。
6. 做前端诊断与交互回归，重点验证：三步是否清晰、截图区是否常驻、任务管理 Edge 控制是否未受影响。
