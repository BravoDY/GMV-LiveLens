# 全平台正式采集 1s 轮询方案

## Summary
- 目标：仅针对“正式采集”链路统一轮询频率，不改前端 WebSocket 断线兜底轮询，也不改前端单页调试态的大屏只读测试轮询。
- 结论：`1s` 具备实施可行性，但必须按“稳定优先”方式收口；`0.5s` 风险明显更高，不建议直接全平台启用。
- 推荐决策：先把全平台正式采集统一到 `1s`，并保留平台级例外能力；不建议当前阶段把全平台直接降到 `0.5s`。

## Current State Analysis
- 当前运行态观察结果（基于本地正在运行的 `http://127.0.0.1:8100`）：
  - `uvicorn backend.main:app --host 127.0.0.1 --port 8100` 正在运行。
  - `/api/settings` 返回 `interval_seconds = 1.0`，说明 OCR/非固定只读任务的全局正式采样间隔实际上已经是 `1s`。
  - `/api/tasks` 显示当前共有 `9` 个启用正式任务，全部为 `remote_edge + screen_readonly`，覆盖天猫、京东、唯品、抖音、得物。
  - 这 `9` 个任务当前全部 `status = ok`，`last_reason = 大屏只读读取成功`，最近 `last_sample_at` 基本集中在同一秒内，未看到运行态阻塞证据。
  - 任务行中的 `interval_seconds` 字段仍普遍显示为 `2.0`，但从当前运行结果看，它并不是这批 `screen_readonly` 正式任务的实际调度值；真实调度仍由 `scheduler.py` 的固定只读间隔/全局间隔决定。
- 正式采集调度主循环位于 `backend/services/scheduler.py`。
  - OCR 任务使用全局配置 `store.get_setting("interval_seconds", "0.5")` 作为调度间隔。
  - `screen_readonly` 任务优先读取 `shop_config.screen_readonly_fixed_interval_seconds(task.platform)`。
  - 若平台没有固定间隔，则回退到只读结果里的 `screen.interval_seconds` 或全局间隔。
- 当前已知只读固定间隔配置位于 `backend/services/shop_config.py`。
  - 已固定 `1s`：京东、抖音、得物。
  - 未固定：天猫、唯品会，仍可能回退到页面返回值或全局配置。
- 调度循环每轮会：
  - `list_tasks(include_disabled=False)` 读任务列表；
  - `store.get_setting("interval_seconds", "0.5")` 读全局采样间隔；
  - 把到期任务通过 `asyncio.to_thread(self.capture_once, task_id)` 并发执行。
- SQLite 位于 `backend/services/store.py`。
  - 使用 `sqlite3.connect(DB_PATH, timeout=30, check_same_thread=False)`。
  - 当前未看到 WAL、busy_timeout、写入批处理等显式优化。
  - `capture_once()` 会在一次采样中触发多次数据库操作：至少更新任务运行态、追加样本记录，部分失败路径也会写运行态。
- 正式采集的重负载部分位于 `backend/services/scheduler.py` + `capture_once()`。
  - OCR 任务会涉及截图、裁剪、OCR、候选筛选、截图落盘、数据库写入。
  - 大屏只读任务会涉及真实 Edge 控制、页签恢复、页面读取、数据库写入。
- 前端 `frontend/app.js` 中 `setInterval(poll, 2000)` 是看板断线后的 HTTP 兜底轮询，不属于本次“正式采集”范围。

## Risk Assessment
### 1s 的可行性
- 可行。
- 原因：
  - 你的当前运行态已经有 `9` 个正式直连任务在稳定运行，且最近采样时间表现接近 `1s` 级别，没有出现大面积失败或阻塞。
  - 当前系统已经允许 OCR 全局最小值为 `0.5`，数据模型和前端表单并不排斥 `1s`。
  - 京东、抖音、得物的大屏只读已经在正式链路上固定 `1s`，说明框架层已具备该频率能力。
  - 调度循环 tick 频率为 `0.2s`，足以支撑 `1s` 到期判断，不需要改事件循环颗粒度。

### 1s 的主要风险
- 当前不是“大促峰值”时段，因此当前运行稳定只能证明“现状有可行性”，不能直接等价为“峰值绝对无阻塞”。
- 数据库写放大风险：
  - 每秒所有启用任务都可能触发一次运行态更新和样本写入。
  - 目前 `store.py` 采用“每次操作新连接”的模式，虽然有 `timeout=30`，但高频下会放大连接建立和写锁等待成本。
- 真实 Edge 并发风险：
  - 多个 `remote_edge` 任务会并发进入 `capture_once()`。
  - 页面恢复、截图、只读读取、页签定位都依赖真实浏览器状态；1s 下如果单次动作接近或超过 1s，会出现任务堆积和实际频率漂移。
- OCR 路径稳定性风险：
  - OCR 链路比只读链路更重，且还包含截图落盘与候选判断。
  - 对 OCR 任务统一压到 1s，会比只读任务更容易放大 CPU、磁盘和 SQLite 写入抖动。
- 全局设置读取成本：
  - 调度主循环每轮读取一次全局 `interval_seconds`，当前实现简单但非最优。
  - `AUDIT_FINDINGS.md` 已提示这一点在高频下会放大 SQLite 读取成本。

### 0.5s 的主要风险
- 不建议直接全平台启用。
- 原因：
  - OCR 任务可能在上一轮尚未真正完成时就进入下一轮到期判断，导致线程池堆积概率上升。
  - SQLite 写锁竞争会明显增加，尤其是在任务数较多时。
  - 真实 Edge 页签控制在 `0.5s` 节奏下更容易出现“页面仍在恢复/页面仍在刷新/读到旧 DOM”的抖动。
  - 即使模型层允许 `0.5`，当前系统并没有“背压限制、批量写入、调度并发上限、采样耗时监控”等配套保护。

## Proposed Changes
### 方案建议
- 采用“全平台正式采集统一到 1s”的稳态方案，不推进 `0.5s` 全量落地。
- 范围仅包含正式采集：
  - OCR 正式任务；
  - `screen_readonly` 正式任务。
- 不包含：
  - `frontend/app.js` 的 WebSocket 失败兜底轮询；
  - `frontend/edge.js` 的单页只读测试轮询；
  - 页面预览巡检轮询。

### 具体改动点
- `backend/services/shop_config.py`
  - 收口 `SCREEN_READONLY_FIXED_INTERVAL_SECONDS`，把所有已支持的 `screen_readonly` 平台统一固定为 `1.0`。
  - 目标是让天猫、唯品会也与京东、抖音、得物一致，不再依赖页面返回的 `screen.interval_seconds`。
- `backend/services/store.py`
  - 本次不建议重构数据库层，但建议至少评估一项低成本稳定性增强：
    - 给高频读取的全局设置增加调度侧缓存，避免每个 tick 都打 SQLite。
  - 若后续进入大促前压测，再考虑是否需要 WAL / busy_timeout / 写入批处理等更重的数据库优化。
- `backend/services/scheduler.py`
  - 保持 `screen_readonly` 任务优先走平台固定间隔。
  - 将 OCR 正式调度的统一全局间隔规划为 `1.0`。
  - 推荐同时补一个低风险优化：缓存全局 `interval_seconds` 设置值或降低每轮重复读取 DB 的频率，减少 1s 模式下的 SQLite 读取放大。
  - 推荐评估是否需要“到期任务数过多时的保护策略”，避免全部任务同一时刻并发突刺。
- `backend/main.py`
  - 检查并同步全局设置接口返回/保存默认值与文案，使 UI 默认建议值也落到 `1.0`。
- `frontend/app.js`
  - 仅同步“正式采样间隔”的默认显示值或提示语；不改看板断线后的 `2000ms` 兜底轮询。
- `frontend/config.js`
  - 检查各平台只读说明文案，统一描述为“正式链路按 1 秒轮询”。
- `data/shops_default.json`
  - 是否回写默认店铺 `interval_seconds = 1.0` 需要谨慎：
    - 若只想影响新建/重建任务模板，则建议同步。
    - 若担心影响现有 OCR 标定节奏，可仅改系统全局默认，不批量回写所有模板。

## Assumptions & Decisions
- 已确认本次范围是“仅正式采集”。
- 已确认优先目标是“绝对稳定”，不是追求极限实时性。
- 决策：推荐值为 `1s`，不推荐当前直接落地 `0.5s`。
- 决策：基于当前运行态，优先处理“口径统一”和“高频读设置优化”，而不是贸然引入更激进的 `0.5s`。
- 决策：不把前端 `app.js` 断线兜底轮询纳入本次实施。
- 决策：`screen_readonly` 页面自身返回的 `interval_seconds` 不再作为正式调度主依据；正式调度应以平台固定 `1s` 为准。
- 风险接受边界：
  - 如果压测/运行观察发现 OCR 任务在 1s 下仍有明显堆积，应允许单独把 OCR 保持为 `1.5s` 或 `2s`，而把大屏只读继续维持 `1s`。
  - 也就是说，“全平台统一”可以作为目标，但稳定性优先于绝对一致。

## Verification Steps
- 静态验证
  - 检查 `backend/services/shop_config.py` 中所有已支持只读平台是否都固定为 `1.0`。
  - 检查 `backend/services/scheduler.py` 是否仍存在只读任务回退到页面 `interval_seconds` 的路径；若保留，应确认固定平台不会走到该分支。
  - 检查 `backend/main.py` / 前端配置面板默认值是否与 `1s` 策略一致。
- 运行前基线记录
  - 记录当前启用任务数、任务类型分布、`last_sample_at` 刷新频率和 `last_reason_code` 分布，作为实施前对照。
- 运行验证
  - 启动系统后，选取 OCR 任务与大屏只读任务各至少 2 个，观察 3-5 分钟。
  - 核对 `last_sample_at` 更新频率是否接近 1 秒，没有出现大面积跳秒或卡顿。
  - 核对 `last_reason_code` 是否没有明显增加 `edge_action_timeout`、`edge_debug_disconnected`、`parse_failed`、`database is locked` 类异常。
- 稳定性验证
  - 观察 SQLite 是否出现明显写阻塞或任务状态延迟。
  - 观察真实 Edge 会话是否出现频繁断连、页签恢复失败、读到旧页签等现象。
  - 观察 CPU、磁盘和页面预览是否出现明显抖动。
- 回退预案
  - 若 1s 下 OCR 路径不稳定，优先回退 OCR 全局间隔，而不是回退大屏只读平台。
  - 若单个平台页面本身负载偏高，应允许该平台在 `shop_config.py` 中保留独立例外值。

## Recommended Outcome
- 推荐执行方案：统一“正式采集”到 `1s`，并同步补一处轻量调度优化，目标是把数据库读取放大风险降到可接受范围。
- 不推荐执行方案：直接把全平台正式采集统一到 `0.5s`。
