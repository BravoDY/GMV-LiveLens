# GMV-LiveLens 预警逻辑与采集频率设定修改计划

## 1. 当前状态分析
目前系统中存在以下两个待优化点：
1. **预警逻辑不精细**：`scheduler.py` 中的 `_judge` 函数在判断新提取的金额候选值时，仅仅是对比 `selected < last * 0.9` 或 `selected > last * 5` 就直接触发死局式的 `suspect` 状态。这导致即使是在双 11、618 等快速爆发期，正常的 5 倍以上跳变也会一直卡在预警状态；同时用户期望任何数值变小的情况（不仅限于跌去 10%）都应该预警。
2. **缺乏采集频率的 UI 配置入口**：虽然任务模型和后端 API 已经支持 `interval_seconds`（采集频率，最小 0.5 秒），但前端界面中 `intervalSeconds` 的输入框被隐藏在 `<section class="setup-hidden-fields">` 内部，用户无法直接在工作台上修改频率。

## 2. 修改方案与步骤

### 步骤 1：优化后端 `_judge` 预警判断与自愈逻辑
**修改文件**：`backend/services/scheduler.py`
**修改内容**：
- 重新定义 `suspect` 触发条件：
  - **下跌必定预警**：只要 `selected < last`，标记为异常跳变。
  - **跳变过大预警**：`selected > last * 5`（保留现有的 5 倍防 OCR 错误阈值），标记为异常跳变。
- **引入“多帧确认自愈”机制**：
  - 如果检测到上述异常跳变（无论是下跌还是剧增），我们将其 `status` 返回为 `suspect` 以便前端红框高亮告警。
  - **同时继续累加 `pending_count`**。要求连续确定的次数提高到 `max(3, confirm_count + 1)`。
  - 意味着：如果是偶尔一次的 OCR 识别故障，下一秒正常后预警会自动解除；如果是双 11 期间真实的 GMV 剧增，只要连续 3 次（或以上）采集到该大数值，系统会自动解除预警，将新值采纳为 `ok`。这完美兼顾了防错与快速爆发的诉求。

### 步骤 2：将采集频率输入框暴露到前端 UI
**修改文件**：`frontend/index.html`
**修改内容**：
- 找到隐藏域中的 `<input id="intervalSeconds" type="number" value="1" />`，将其移除。
- 在 `id="setupStepSave"`（“步骤 3. 保存并进入下一家”）的区块内，新增一个可见的输入框组合，让用户可以在保存前配置此数值。
- 新的 HTML 结构示例：
  ```html
  <label style="display:block; margin:12px 0;">
    采集频率 (秒)
    <input id="intervalSeconds" type="number" step="0.5" min="0.5" style="width: 80px; margin-left: 8px; background: rgba(255,255,255,0.05); border: 1px solid var(--line); color: var(--text-primary); border-radius: 4px; padding: 4px 8px;" />
  </label>
  ```
*(注：`frontend/config.js` 中保存和加载的逻辑已经具备处理 `intervalSeconds` 的能力，因此无需改动 JS。)*

## 3. 验证步骤
1. **重启后端服务**，并在浏览器打开看板。
2. **频率配置测试**：进入“采集配置”页，在“步骤 3”下方确认能看到“采集频率 (秒)”输入框。将其修改为 0.5 后保存任务，检查任务管理的“采一次”间隔或数据库记录是否生效。
3. **下跌预警测试**：修改数据库中的 `last_trusted_value` 为 25000，使用 OCR 扫一个 24000 的数字。观察看板是否变成红色的 `suspect`。连续扫 24000 多次后，观察它是否能自动恢复成 `ok`。
4. **爆发跳变测试**：`last_trusted_value` 设为 1000，扫一个 6000 的数字（大于 5 倍），观察是否进入 `suspect`，并且连续扫 3 次后自动采纳为 6000 (`ok`)。
