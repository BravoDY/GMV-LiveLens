## 1. 执行摘要

### 总体评价：**准商业级项目，工程化能力是最大短板**

这是一个**核心逻辑健壮、架构设计有亮点、但工程化规范严重不足**的项目。把它比作一辆车：发动机（OCR+Edge采集调度）和底盘（状态机+自动恢复）做工精良，但仪表盘（日志）、安全带（测试目录）、工具箱（lint/formatter）、说明书（环境文档）都处于临时拼凑状态。

| 判断维度 | 结论 |
|---|---|
| 是否达到商业项目标准 | ❌ 尚未完全达到，差距主要在**工程化**和**代码组织** |
| 是否需要重构 | ⚠️ 不需要架构重构，但需要**文件级拆分** 和**工程化补齐** |
| 最大短板 | **工程化成熟度**（无 lint/formatter、无 CI/CD、无 Docker、文件过大） |
| 最大优势 | **核心业务逻辑**（采集状态机、OCR评分、自动恢复、多层级降级） |
| 是否建议继续开发 | ✅ 可以，但必须先补齐工程化基础设施 |

### 一句话总结

> 项目的「能跑」部分做得非常好（65/65 测试通过），但「能维护」「能交付」「能协作」部分有显著差距。建议在启动新功能开发前，用 1-2 天补齐工程化基础设施。

### 3.3 数据流评估

```
shops.csv (GBK)
  → shop_config.load_shop_configs()               # 配置解析
  → store.init_db() → _ensure_shop_edge_sessions() # 会话初始化
  → store.sync_tasks_with_shop_configs()           # 任务同步
  → scheduler._run_loop() (每0.2s)                 # 采集心跳
      → capture_once() (asyncio.to_thread)
          → RemoteEdge.screenshot_page() (CDP)     # 截图
          → crop_by_ratio() → read_text()          # OCR
          → extract_candidates() → _judge()        # 评分+确认
          → store.update_task_runtime()            # 落库
  → broadcast_snapshot() → WS /ws/live             # 推送
  → 前端 renderSnapshot()                          # 展示
```

**评价：数据流清晰、单向、可追溯。** 这是项目的核心亮点。前端 state.snapshot 作为单一数据源，通过 WebSocket 驱动全量更新。

## 9. 是否建议重构

### 明确判断：**不需要架构重构，需要文件级拆分 + 工程化补齐**

| 选项 | 结论 |
|---|---|
| 不需要重构，只需局部优化 | ✅ **是** - 核心架构（分层/数据流/状态机）设计合理 |
| 需要局部重构 | ✅ **是** - 大文件拆分属于局部重构 |
| 需要核心模块重构 | ❌ 否 - OCR/采集/调度核心逻辑健壮 |
| 需要整体架构重构 | ❌ 否 - 架构方向正确 |

### 原因

1. **分层设计正确**：collectors/ → services/ → main.py 三层分离已在目录级别体现，只是 main.py 未继续拆分路由
2. **数据流清晰**：单向数据流（CSV→SQLite→Collector→WS→Frontend），状态管理单一
3. **核心逻辑优秀**：_judge 状态机、OCR 评分系统、自动恢复绑定这些"最难写的部分"质量很高
4. **改造风险可控**：文件拆分和工程化补齐不会改变业务逻辑
