## Summary

- 目标：优化 `任务管理` 卡片中的状态展示，让“暂停”具备和运行状态一样醒目的标签表现，并将 `pending_confirm` 等英文状态码翻译为中文，整体颜色风格随状态变化。
- 成功标准：
  - 卡片头部不再直接显示原始英文状态码。
  - “暂停”从灰色说明文字中抽离，变成独立且醒目的状态标签。
  - `pending_confirm` 翻译为易懂中文，并以突出样式展示。
  - 不同状态具备不同颜色层级，暂停状态与运行状态可同时被识别。

## Current State Analysis

- 任务卡片由 `frontend/app.js` 的 `renderManager()` 渲染，当前直接把 `task.status` 原样写入右上角标签。
  - 文件：`frontend/app.js`
  - 关键位置：`renderManager()` 中的 `<span class="status ${task.status}">${task.status}</span>`
- 卡片副标题中的启停状态仅以文本 `启用/暂停` 混在平台和采集模式描述里，没有单独视觉强调。
  - 文件：`frontend/app.js`
  - 当前表现：`${platform} · ${mode} · ${task.enabled ? "启用" : "暂停"}`
- 样式仅对 `ok` 和错误类状态做了基础颜色区分，`pending_confirm` 没有专门样式。
  - 文件：`frontend/styles.css`
  - 现有规则：`.status.ok` 与 `.status.suspect/.parse_failed/...`
- 卡片外层也只对 `ok` / 错误状态做边框颜色区分，未考虑“暂停”状态的卡片整体氛围。
  - 文件：`frontend/styles.css`

## Proposed Changes

### 1. 在 `frontend/app.js` 增加状态文案映射与展示逻辑

- 新增一个状态映射函数或常量表，把内部状态码转换为中文文案。
- 计划映射：
  - `ok` -> `正常`
  - `pending_confirm` -> `待确认`
  - `suspect` -> `疑似异常`
  - `parse_failed` -> `识别失败`
  - `window_not_found` -> `窗口未找到`
  - `needs_recalibration` -> `需要重标`
  - `page_not_found` -> `页面未找到`
  - 未知状态 -> 原值或 `未知状态`
- 在 `renderManager()` 中：
  - 将右上角标签从原始英文改成中文文案。
  - 为右上角标签生成更稳定的样式类，例如基于“业务状态”与“启停状态”分别命名。
  - 将“暂停”从副标题里拆出，新增一个独立徽标标签展示。
- 展示结构建议：
  - 右上角主状态标签：表示任务运行判定状态，如 `待确认`、`正常`、`识别失败`。
  - 标题区旁新增次状态标签：表示启停状态，如 `暂停中`。
  - 副标题保留 `平台 · 模式`，不再混入“暂停”文字，减少信息噪声。

### 2. 在 `frontend/styles.css` 增加状态标签视觉体系

- 保留现有深色科技风，增强标签层级。
- 为主状态标签增加专门配色：
  - `正常`：青绿色
  - `待确认`：黄色/琥珀色
  - `疑似异常/识别失败/窗口未找到/需要重标/页面未找到`：红色
- 为“暂停中”新增独立标签样式：
  - 推荐使用偏蓝灰或橙灰高对比样式，既明显，又不与错误态混淆。
  - 标签样式比现在的灰色元信息更醒目，例如使用圆角胶囊、边框、半透明背景和轻微 glow。
- 卡片整体状态联动：
  - 暂停任务的卡片外层可增加专门类，例如更柔和的边框/背景变化。
  - 若任务暂停，则保留业务状态标签，同时额外展示“暂停中”，实现“双状态并存”。

### 3. 调整卡片状态类命名，避免语义混淆

- 当前 `.manager-card ${task.status}` 只表达业务状态，不含启停语义。
- 计划让卡片类同时包含：
  - 业务状态类：如 `status-pending-confirm`、`status-ok`
  - 启停状态类：如 `is-paused`、`is-enabled`
- 这样 CSS 可以分别控制：
  - 业务状态标签颜色
  - 卡片是否处于暂停模式的整体视觉

## Assumptions & Decisions

- `pending_confirm` 的含义判定为：OCR 结果已识别出候选值，但尚未达到连续确认次数，因此不应直接写入最终可信值；中文展示定为 `待确认`。
- “暂停”与 `pending_confirm` 不是同一概念：
  - `pending_confirm` 是业务采集状态。
  - `暂停` 是任务启停状态。
- 因此最终设计采用“双标签”而不是二选一覆盖，确保用户能同时看到：
  - 当前任务是否暂停
  - 暂停前/当前记录的业务状态是什么
- 本次只改前端展示文案和样式，不修改后端状态机逻辑，也不改数据库字段。

- 重点目测确认：
  - 同一张卡片上，`暂停中` 和 `待确认` 同时出现时不拥挤。
  - 卡片边框/标签颜色在深色背景上对比足够明显，但不过度刺眼。
