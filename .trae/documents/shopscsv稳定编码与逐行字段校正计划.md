# shops.csv 稳定编码与逐行字段校正计划

## 概要
- 将 `data/shops.csv` 统一保存为稳定编码，消除当前文件内容乱码问题。
- 逐行校正 `platform`、`shop_name`、`default_page_url`、`keyword_hint`、`url_patterns`、`enabled`、`Target` 等字段，确保它重新成为项目唯一可信的店铺配置源。
- 修复完成后，前后端读取到的店铺命名、目标值、默认页面地址和匹配规则都必须稳定且可重复解析。

## 当前状态分析

### 1. `shops.csv` 文件本体已经是乱码文本
- 当前 `data/shops.csv` 的表头仍可读，但从第 2 行开始内容显示为乱码，例如：
  - `��è,�ٷ��콢��,...`
  - `����,�ٷ��콢��,...`
- 这说明问题不只是“读取方式不对”，而是文件当前保存状态已经不可作为稳定配置源继续使用。

### 2. 运行时完全依赖 `shops.csv`
- `backend/services/shop_config.py` 的 `load_shop_configs()` 会优先读取 `data/shops.csv`，只有 CSV 不存在或解析不到行时才会退回 `shops_default.json`。
- 当前 `data/shops_default.json` 实际内容是空数组 `[]`，不能作为恢复来源。
- 因此本次修复不能依赖现成备份，而必须直接修复 `data/shops.csv` 本身。

### 3. 当前解析逻辑对字段有明确要求
- `backend/services/shop_config.py` 中已定义：
  - `platform` 和 `shop_name` 不能为空，否则直接报错；
  - `(platform, shop_name)` 不能重复；
  - `Target` 会被转成整数；
  - `keyword_hint` 缺省会回退到 `成交金额`；
  - `default_page_url`、`url_patterns` 会直接影响页面绑定和任务匹配。
- 因此这次不能只修编码，必须同步校正每一行字段值，否则仍可能出现命名、绑定或目标异常。

### 4. 项目文档给出的当前业务事实
- `README.md` 标明 `data/shops.csv` 是店铺清单主数据源。
- 结合当前项目运行上下文与之前已确认的任务命名，当前有效店铺集合是：
  - 天猫 / 官方旗舰店
  - 天猫 / 儿童旗舰店
  - 天猫 / BLANC旗舰店
  - 天猫 / GOLF旗舰店
  - 京东 / 官方旗舰店
- 当前 CSV 共 5 条业务行，和现阶段系统实际使用的店铺数量一致。

## 推荐实现方案

### 1. 直接重写 `data/shops.csv` 为稳定编码
- **文件**：`data/shops.csv`
- **修改内容**：
  - 将文件整体重新保存为统一、稳定的编码格式，推荐 `UTF-8 with BOM` 或项目统一约定的单一编码；
  - 保留现有表头结构：`platform,shop_name,default_page_url,keyword_hint,url_patterns,url_must_contain,enabled,Target`。
- **原因**：
  - 当前文件内容本体已经乱码，继续依赖多编码兜底没有意义；
  - 需要把配置源恢复到“任何编辑器和运行时都能稳定读取”的状态。

### 2. 按业务事实逐行恢复 5 条店铺记录
- **文件**：`data/shops.csv`
- **修改内容**：
  - 逐行把乱码内容恢复为明确、可读、可校验的中文字段；
  - 基于当前系统已确认的业务店铺，恢复以下 5 条记录：
    - 天猫 / 官方旗舰店
    - 天猫 / 儿童旗舰店
    - 天猫 / BLANC旗舰店
    - 天猫 / GOLF旗舰店
    - 京东 / 官方旗舰店
  - 各行需同步校正：
    - `default_page_url`
    - `keyword_hint`
    - `url_patterns`
    - `enabled`
    - `Target`
- **原因**：
  - 这些列直接参与任务初始化、页面匹配、OCR 候选提示和目标值显示；
  - 若只修 `platform / shop_name`，其余字段仍可能继续导致看板、绑定和目标问题。

### 3. 以当前项目既有业务规则统一字段取值
- **文件**：`data/shops.csv`
- **修改内容**：
  - 天猫行统一校正为淘宝/天猫数据后台页面地址与对应 URL 模式；
  - 京东行统一校正为京东商家后台页面地址与对应 URL 模式；
  - `keyword_hint` 统一使用当前项目 OCR 主链路兼容的关键词；
  - `Target` 保持与当前业务目标一致，不额外改动业务数值。
- **原因**：
  - 同平台多个店铺应共享一致的默认页面和识别口径，减少后续维护偏差。

### 4. 校验解析结果与同步链路
- **文件**：`backend/services/shop_config.py`（只读验证，不要求改动，除非实现时发现必须补充编码约束）
- **修改内容**：
  - 先按现有解析逻辑验证新 CSV 能稳定解析；
  - 如确有必要，再补充“明确只接受单一稳定编码”或更明确的错误提示。
- **原因**：
  - 主要修复点应放在 `shops.csv` 数据本体；
  - 只有在实现时确认现有解析逻辑仍会产生歧义，才考虑最小化修改解析层。

### 5. 用同步链路把修正后的配置重新落到运行态
- **文件**：`backend/services/store.py`（执行时调用现有同步逻辑，不预期大改）
- **修改内容**：
  - 修复 `shops.csv` 后，执行现有 `sync_tasks_with_shop_configs()`；
  - 让数据库中的任务命名、目标值、排序和页面配置重新与 CSV 严格对齐。
- **原因**：
  - `shops.csv` 是源头，SQLite 是运行态；
  - 不把新 CSV 同步回数据库，前端仍可能继续看到旧数据。

## 假设与决策
- 本次“统一保存为稳定编码并逐行校正字段内容”指向的对象就是 `data/shops.csv`。
- `data/shops_default.json` 当前为空，不作为恢复数据来源。
- 当前有效业务范围是 5 家店铺，不额外新增或恢复历史店铺。
- 本次以“恢复为当前真实业务配置”为目标，不保留乱码原文。
- 除非实现时验证表明确有必要，否则优先只修 `shops.csv` 数据，不扩大到额外架构改造。

## 风险评估
- **数据风险**：由于 `shops_default.json` 为空，若逐行恢复内容填错，会直接影响任务初始化、页面绑定和目标展示。
- **兼容性风险**：如果现有某些历史任务依赖乱码时期的旧 `(platform, shop_name)`，修复后需要重新同步到新命名。
- **运行风险**：若编码格式与现有启动/编辑工具不一致，后续手工编辑仍可能再次写坏文件，因此需要明确单一编码约定。

## 验证步骤
1. 重新读取 `data/shops.csv`，确认文件内容直接可读，不再出现乱码字符。
2. 通过 `backend/services/shop_config.py` 的现有解析逻辑读取，确认返回 5 条配置，且每条的 `platform / shop_name / target / default_page_url / keyword_hint` 均正确。
3. 调用 `/api/shops`，确认接口返回的 5 条店铺配置与 CSV 完全一致。
4. 执行任务同步后，检查 `/api/tasks`，确认任务名称、目标值、排序已与新 CSV 对齐。
5. 刷新前端实时看板和采集配置页面，确认平台名、店铺名、目标值、绑定候选均正常。
6. 记录最终采用的文件编码约定，避免后续再次被错误编辑器写坏。
