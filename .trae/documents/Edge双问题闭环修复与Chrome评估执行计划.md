# Edge 双问题闭环修复与 Chrome 评估执行计划

## Summary
- 目标一：彻底修复“明明历史已登录且当前能启动 Edge，却仍被判成 `edge_session_not_ready` / 控制链路未恢复”的误导性状态与实际异常链路。
- 目标二：彻底修复“点击显示 Edge 后，隐藏在后台或屏幕外的 Edge 没有真实回到主屏前台”的窗口恢复问题。
- 目标三：按真实业务闭环验收，不以“接口返回 200”作为完成标准，而是以“显示 Edge -> 扫描页签 -> 生成预览”在真实店铺链路上反复通过作为完成标准。
- 附带判断：本轮不以“切换 Chrome”作为主修复方案；先把现有 Edge/Chromium 控制链路修稳，再决定是否抽象浏览器适配层。

## 当前需求理解
- 你当前要解决的不是单一报错，而是两个实际体验问题：
  - 历史已多次启动并登录的店铺会话，不应再被产品语义误导成“没有 session”。
  - “显示 Edge”按钮必须真正把隐藏窗口恢复到主屏前台，而不是只做后端动作或返回乐观结果。
- 成功标准不是“改几处文案”，而是：
  - 状态语义准确。
  - 窗口行为真实。
  - 采集配置链路可继续推进。
  - 经过反复自测后仍稳定。

## 当前项目结构判断
- `backend/main.py`
  - FastAPI API 入口。
  - 当前已承载 `start/show/hide/close`、`page-candidates`、`preview`、`test-ocr` 等 Edge 相关接口。
- `backend/collectors/remote_edge.py`
  - Edge 远程调试与 Playwright/CDP 连接核心。
  - 管理启动、显示、截图、页签扫描、健康检查、超时恢复。
- `backend/collectors/window_control.py`
  - Windows 下的 Edge 窗口发现、恢复、隐藏、关闭、进程诊断与前台激活。
- `frontend/core.js`
  - Edge 动作 API 调用封装，含 `startEdgeSession()`、`showEdgeSession()`、`startAndShowEdgeSession()`。
- `frontend/app.js`
  - 任务管理页按钮行为与显示/启动失败提示主入口。
- `frontend/config.js`
  - 采集配置工作台的页签扫描、恢复提示、重绑流程。
- `frontend/edge.js`
  - 配置页生成预览、页签失效、控制链路异常提示。
- `backend/services/scheduler.py`
  - 正式采集链路状态归类，需与手动配置链路的错误语义保持一致。

## Current State Analysis

### 1. 已确认已经做到的部分
- 单店铺 `show-edge` 当前已经不是旧版异步 `accepted`。
  - `backend/main.py` 当前已同步执行 `client.show_edge(...)` 并在失败时返回结构化 `409/500`。
- 前端任务管理页当前也已经等待 `showEdgeSession()` 的真实返回结果。
  - `frontend/app.js` 中 `show-edge` 和 `start-edge` 都会根据 `window_found`、`debug_available`、`result.last_error` 决定提示。
- `window_control.py` 当前已经做了多轮 `ShowWindow / Restore / move_into_view / activate / maximize`，并带结果复查。
- `remote_edge.py` 当前也已将健康状态拆成：
  - `edge_debug_unavailable`
  - `edge_debug_disconnected`
  - 若窗口已存在但调试端口未通，`show_edge()` 会优先尝试把窗口显示出来。

### 2. 仍然没彻底完成的真实问题
- `RemoteEdge._health()` 仍然只把“是否 connected”定义为“已有 `_browser` 且连接未断开”。
  - 这会导致：
    - 调试端口其实已通；
    - 当前页签也能被列出来；
    - 但因为 `_browser` 尚未重新建立或被动失效，健康状态仍显示 `connected=false`；
    - 前端和 API 层继续把它解释成“控制链路未恢复”。
- `start-edge` 路径仍存在 `500/Internal Server Error` 未完全收口。
  - 说明“调试端口是否可访问”“Playwright/browser 是否重连”“窗口是否恢复”“页签是否可用”这几层之间仍有状态错位。
- “显示 Edge”虽然已有同步返回，但还不能证明当前选中的窗口一定是最终用户想看的主窗口。
  - 当前 `window_control.py` 的窗口筛选主要基于 PID、可见性、是否最小化、是否屏幕外、面积大小。
  - 若同一进程下存在多个窗口、空白启动窗、过期缓存句柄、前台激活被系统阻止，仍可能出现：
    - 接口认为成功；
    - 用户主屏没有看到正确窗口；
    - 或拉回的是非目标窗口。
- 采集配置页已经能识别 `edge_debug_unavailable` / `edge_debug_disconnected`，但仍依赖后端健康状态是否准确。
  - 如果后端健康状态没有真实反映“可恢复连接”和“可继续页签扫描”，前端提示仍会显得反复横跳。

### 3. 根因判断
- 问题 1 的核心不是“历史 session 丢了”，而是：
  - Profile/登录态存在；
  - 调试端口可能已恢复；
  - 但 `_browser` / `_pages` / 健康检查缓存没有及时重建或校正；
  - 于是业务链路和健康状态产生错位。
- 问题 2 的核心不是“按钮没调用”，而是：
  - 窗口恢复成功的判定仍偏技术层；
  - 缺少“这是不是当前会话真正要显示的主窗口”的最终确认；
  - 以及启动窗口、后台窗、无标题窗、缓存 HWND、同 PID 多窗口场景的筛选仍需收口。

## 最佳实现方案

### 总体方案
- 不推翻现有 Edge 实现。
- 延续现有拆分方向，但把“会话健康状态”“浏览器重连”“窗口选择与显示验收”“前端提示一致性”“真实回归验证”五件事一次性收口。
- 先把当前 Edge 路线修稳，再决定是否做 Chrome/Chromium 抽象。

### 文件级改动方案

#### 1. `backend/collectors/remote_edge.py`
- 目标：修正健康状态与实际可用性错位，收口 `start/show` 链路。
- 计划改动：
  - 在 `_health()` 中补充“轻量重连判断”与“连接状态恢复策略”：
    - 当 `debug_available=True` 但 `_browser` 为空或已断开时，不立即固定判成 `edge_debug_disconnected`。
    - 先尝试一次受控的 `_ensure_browser()` 或更轻量的恢复逻辑，再决定最终 `connected` 与 `reason_code`。
  - 明确区分以下场景：
    - 调试端口未开。
    - 调试端口已开但 Playwright 重连失败。
    - Playwright 已连但当前无有效业务页。
    - 旧 `_pages` 缓存失效但浏览器可重建。
  - 收口 `_start_edge()` 的返回语义：
    - 如果端口可用但 `_ensure_browser()` 失败，返回明确 `reason_code`，避免落成裸 `500`。
    - 必要时在 `start_edge()` 成功后主动做一次浏览器连接预热，减少后续 `page-candidates`、`preview` 首次命中时才暴露断链。
  - 收口 `__show_edge()` 的窗口结果：
    - 把 `window_diagnostics`、选中的窗口 PID/HWND、恢复动作链保留到最终返回，便于前端提示和日志排查。
  - 如有必要，为窗口与调试链路增加少量结构化调试日志，便于真实回归时定位阶段失败点。
- 为什么这样设计：
  - 当前最大的语义错位来自健康状态与真实链路不一致，先修这里，前端提示和后续配置页行为才能稳定。

#### 2. `backend/collectors/window_control.py`
- 目标：把“显示成功”从“调用过 Win32 API”升级为“目标主窗口真实回到主屏前台”。
- 计划改动：
  - 加强窗口候选筛选：
    - 在 `edge_window_diagnostics()` / `find_edge_window()` 中补充更稳定的候选排序或过滤规则。
    - 优先选择真正可见、尺寸正常、标题有效、非工具窗、非异常启动窗的窗口。
  - 对缓存 `hwnd` 做更严格复验：
    - 避免复用已失效或已不代表目标窗口的缓存句柄。
  - 加强 `show_edge_window()` 的最终验收：
    - 不只检查 `visible/minimized/offscreen`；
    - 还要复查当前窗口是否仍为候选主窗口、是否被移动回主屏区域、是否在多窗口场景下选中了正确窗口。
  - 若系统焦点策略阻止前置：
    - 返回精确错误，而不是泛化为成功。
- 为什么这样设计：
  - 当前用户感知问题就是“你说显示成功，但我没看见”，所以必须把验收标准拉到用户可见层。

#### 3. `backend/main.py`
- 目标：统一 API 错误语义，消灭残留裸 `500`。
- 计划改动：
  - 检查并收口 `/api/edge-sessions/{session_id}/start` 的异常转换逻辑。
  - 对 `page-candidates`、`preview`、`test-ocr` 继续统一：
    - 所有 Edge 控制链路失败都返回结构化 detail。
    - 不再让前端从字符串猜测含义。
  - 若 `start` 后端已能识别“端口可用但连接未恢复”，则返回明确 `reason_code` 与 `recovery_hint`。
- 为什么这样设计：
  - 真实回归中最怕的是某一步重新掉回无语义 `500`，这样前端再完善也无从处理。

#### 4. `frontend/app.js`
- 目标：让任务管理页对 `start/show` 的反馈完全贴合真实后端返回。
- 计划改动：
  - 继续扩充 `edgeWindowErrorMessage()` 的映射，覆盖本轮新增或收口后的 `reason_code`。
  - 对“窗口已显示，但调试链路未恢复”“窗口已恢复，但当前页需要重新绑定”“系统选择了错误窗口”这类信息给出更明确提示。
  - 若后端返回窗口诊断信息，优先使用结构化 detail 生成提示，而不是只读 `last_error`。
- 为什么这样设计：
  - 任务管理页是用户第一触点，提示不准会让用户误判为“session 丢了”或“显示按钮失效”。

#### 5. `frontend/config.js`
- 目标：让采集配置页的页签扫描与恢复提示和任务管理页完全一致。
- 计划改动：
  - 继续统一 `edge_debug_unavailable` / `edge_debug_disconnected` / 目标页缺失 的恢复提示。
  - 若后端判定“当前会话可用但原绑定页失效”，优先引导重绑，而不是笼统要求重启。
  - 若后端判定“窗口已恢复但自动连接未恢复”，明确提示先等待再扫描，减少误操作。
- 为什么这样设计：
  - 你当前感受到的“明明启动过很多次还说未就绪”，很多时候发生在配置页的恢复提示阶段。

#### 6. `frontend/edge.js`
- 目标：让“生成预览”阶段复用同一套错误语义。
- 计划改动：
  - 与 `config.js` 对齐控制链路错误提示。
  - 对 `remote_page_not_found`、控制链路未恢复、截图超时分别给出不同指引。
- 为什么这样设计：
  - 否则用户会在任务管理页看到一种说法，在预览阶段又看到另一种说法，体验仍然割裂。

#### 7. `backend/services/scheduler.py`
- 目标：避免手动配置链路与正式采集链路状态名继续分裂。
- 计划改动：
  - 检查正式采集链路对 `edge_debug_unavailable`、`edge_debug_disconnected`、页签失效等场景的落库状态是否和本轮接口语义一致。
  - 若存在残留 `parse_failed` / 模糊错误，按本轮标准收口。
- 为什么这样设计：
  - 不然手动测试通过后，正式任务运行仍可能回到旧状态名。

## 风险评估
- 风险 1：`_health()` 若增加主动重连，可能引入额外阻塞时间。
  - 控制方式：使用轻量、有限次数、带超时的恢复逻辑，避免健康检查本身卡死。
- 风险 2：窗口筛选规则调整后，可能影响平台批量显示/隐藏的原有行为。
  - 控制方式：保留现有 PID 和会话目录匹配逻辑，在其基础上仅优化候选排序和验收。
- 风险 3：`start-edge` 路径若改为更积极的浏览器重连，可能暴露新的 Playwright/CDP 超时问题。
  - 控制方式：统一纳入结构化错误和超时恢复，不让它重新变成裸异常。
- 风险 4：前端提示文案调整后，若后端 detail 结构不稳定，仍会出现不一致。
  - 控制方式：先固定后端 detail 字段，再改前端映射。

## Chrome 方案判断

### 结论
- 当前阶段不建议把本轮问题的主解决方案改成“切换到 Chrome”。

### 判断依据
- 本仓库当前是 Edge 专用实现，不是浏览器抽象层：
  - 进程发现写死 `msedge.exe`。
  - 窗口匹配围绕 Edge 进程与 Edge 窗口。
  - README 也明确标注当前仅支持 Microsoft Edge。
- 你当前的两类问题里，真正核心的部分不是 Edge 品牌本身：
  - `edge_session_not_ready` 误判是状态建模与健康检查恢复问题。
  - “显示 Edge 不回前台”是 Win32 窗口恢复与前台激活问题。
  - 这两类问题换成 Chrome 依然会遇到同类根因。
- 外部公开信息也显示 Chromium/CDP 路线本身就存在类似问题：
  - `connect_over_cdp` 卡住既出现在 Edge，也出现在 Chrome。
  - 近版本 Chrome 对默认 Profile 的远程调试还有额外限制，反而会增加迁移复杂度。

### 本轮决策
- 本轮先修 Edge 路线，不在当前问题上切到 Chrome。
- 如果未来要支持 Chrome，应单独做“Chromium 浏览器适配层”专题，不应混进这次双问题修复中。

## Assumptions & Decisions
- 决策：以当前真实链路为准，不重复规划已经落地的同步 `show-edge` 改动。
- 决策：本轮必须优先修复 `start-edge` 500、`health.connected=false` 错位、窗口恢复验收不足这三个剩余核心问题。
- 决策：不新增复杂 UI，不改变用户主要操作路径，只修正行为与提示。
- 决策：Chrome 迁移不作为本轮主方案。
- 假设：当前可继续使用真实店铺会话做回归，并允许重启本地服务与反复进行真实点击验证。

## 执行计划

### 阶段 1：收口后端状态模型
1. 通读并核对 `remote_edge.py` 中 `start/show/health/ensure_browser/list_pages/screenshot_page` 的状态流。
2. 修正 `debug_available`、`connected`、`_browser`、`_pages` 之间的错位恢复逻辑。
3. 把 `start-edge` 剩余异常路径全部转换成结构化 `reason_code` 和 `recovery_hint`。

### 阶段 2：收口窗口恢复判定
1. 核查 `window_control.py` 中候选窗口选择、缓存 `hwnd` 复用、屏幕外恢复、前台激活逻辑。
2. 修正“返回成功但用户主屏没看到正确窗口”的剩余场景。
3. 对窗口恢复失败输出精确原因，确保后端/前端都能识别。

### 阶段 3：统一前端提示与闭环行为
1. 调整 `app.js` 的 `start/show` 提示。
2. 调整 `config.js` 与 `edge.js` 的控制链路恢复提示。
3. 校验任务管理页、采集配置页、预览页在同一场景下提示一致。

### 阶段 4：统一正式采集状态
1. 检查 `scheduler.py` 是否仍有残留模糊状态。
2. 确保手动链路与正式采集链路对相同问题使用一致状态名。

### 阶段 5：反复真实回归
1. 重启服务。
2. 对真实店铺执行“显示 Edge”。
3. 确认隐藏窗口是否真实回到主屏前台。
4. 扫描当前会话页签。
5. 若原页签失效，执行重绑。
6. 生成预览。
7. 如任一环节失败，回看日志和结构化 detail，继续修复，直到闭环连续通过。

## Verification Steps

### 1. 状态语义验证
- 场景 A：Profile 存在但调试端口未通。
- 场景 B：调试端口已通但浏览器连接尚未恢复。
- 场景 C：浏览器已可用但原绑定页签失效。
- 预期：
  - 三种场景提示不同；
  - 不再把它们笼统描述成“没有 session”。

### 2. 显示 Edge 验证
- 在真实店铺任务卡片点击“显示 Edge”。
- 预期：
  - 若成功，窗口真实回到主屏前台；
  - 若失败，返回具体原因而不是空泛失败。

### 3. 启动 Edge 验证
- 在当前会话调试端口断开或未恢复时点击“启动 Edge”。
- 预期：
  - 不再出现无语义 `500`；
  - 若仍失败，前端能看到结构化原因与恢复建议。

### 4. 配置链路验证
- 执行“显示 Edge -> 扫描页签 -> 选择页签/重绑 -> 生成预览”。
- 预期：
  - 能在至少一条真实店铺链路上完整通过；
  - 若原绑定页失效，也应能明确引导到重绑而不是误报未就绪。

### 5. 一致性验证
- 检查以下位置是否一致：
  - 后端日志
  - `/health` 结果
  - 任务管理页提示
  - 采集配置页恢复提示
  - 生成预览失败提示

### 6. 回归标准
- 只有以下条件同时满足才算完成：
  - `edge_session_not_ready` 类误导性表达不再影响真实使用。
  - 点击“显示 Edge”后窗口真实可见。
  - `start-edge` 不再出现未分类 `500`。
  - 至少一条真实店铺链路完成“显示 -> 扫描 -> 预览”闭环，并经过重复验证。
