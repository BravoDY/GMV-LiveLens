# 生产级大促战情屏升级计划

## 1. 目标与范围
基于 `D:\MyDownloads\GMV-LiveLens_生产级实时看板像素级修改文档.md` (v2.0) 进行实时看板的生产级响应式与细节优化。
**范围**：
- `frontend/index.html`（调整 Header 控件顺序结构）
- `frontend/styles.css`（补充响应式断点、优化 Header 与卡片布局）
- `frontend/dashboard.js`（增加目标空值的 `--` 兜底逻辑）

## 2. 冲突分析与解决策略 (核心)
在对比了这份 v2.0 的 MD 规范与目前项目状态后，存在以下两个**明确的冲突点**，需要进行合并处理：

1. **店铺排序逻辑冲突**：
   - **MD 规范 (32节)**：依然建议在前端写死一个 `SHOP_ORDER` 数组来做静态排序。
   - **解决策略**：**拒绝执行写死排序**。我们将继续保持目前已经跑通的、由后端 SQLite 提供 `sort_order`（基于 CSV 行顺序）的动态排序逻辑。
2. **“淘宝”与“天猫”的命名冲突**：
   - **MD 规范 (21节)**：平台映射和常量中仍包含对“淘宝”的处理和 `PLATFORM_ORDER = ["淘宝", "天猫", ...]`。
   - **解决策略**：**拒绝回退平台名称**。您在之前的对话中明确要求“将所有淘宝改为天猫”，我们已经在代码中全部替换为了“天猫”（Icon为“猫”），因此在应用规范时，我们将直接沿用“天猫”，忽略规范里的“淘宝”字眼。
3. **“采集全部”按钮位置**：
   - **MD 规范 (4.6节)**：建议不要放在中间，推荐做成小号或移走。
   - **解决策略**：结合您的上一个需求，我们将其保留在右侧控件区最左侧，并为其应用规范推荐的小号按钮样式（`collect-all-btn`），使其不挤占空间。

**可行性判断**：
**完全可行且非常有必要**。除了上述冲突点外，该文档中提到的 Header 防挤压（`nowrap`）、目标为空时显示 `--` 避免误读、以及补充 `1600px` 和 `1366px` 等响应式断点，都是当前看板急需的生产级优化。

## 3. 拟议的具体修改

### 3.1 frontend/index.html
- **Header 调整**：微调右侧 `.header-actions` 的结构，给“采集全部”按钮应用新的 `.collect-all-btn` 类。

### 3.2 frontend/styles.css
- **Header 响应式防挤压**：为 `.nav-item` 添加 `white-space: nowrap; word-break: keep-all;`，为 `.app-header` 添加 `1600px` 和 `1366px` 的媒体查询断点（缩小 padding、gap 和字体大小）。
- **Summary & Store Grid 响应式**：为 `.summary-grid` 和 `.store-grid` 增加媒体查询，确保在小屏下自动减少列数（如 `1366px` 下变 3 列，`980px` 下变 2 列）。
- **卡片内部优化**：增加 `.progress-fill.is-empty` 空状态样式；调整 `.store-card-header` 保证单字 Icon 与标题水平居中对齐；控制卡片最大高度不超过 190px。

### 3.3 frontend/dashboard.js
- **目标空值处理逻辑**：在 `taskToStoreViewModel`、`buildPlatformSummaries` 和 `buildTotalSummary` 之前，引入 `calcProgress` 和 `formatTarget` 辅助函数。
- **UI 更新**：当 `target <= 0` 时，进度和目标显示为 `--`，并为进度条添加 `is-empty` 类，避免出现 `¥0` 和 `0.0%` 的误导信息。

## 4. 验证步骤
1. 调整浏览器窗口大小，验证在宽屏和窄屏（1366px左右）下，顶部导航是否会折行，卡片列数是否会自动适应。
2. 制造一个目标为 `0` 或空的数据，验证卡片上是否正确显示 `目标 --` 和 `达成进度 --`，且进度条呈灰色空状态。
