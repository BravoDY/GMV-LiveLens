# 实时看板任务卡顶部边框像素闪光改造计划

## Summary

- 目标：按照你提供的规范文档，为“实时看板”中的店铺任务卡增加“GMV 数字变化后，顶部边框整体闪一下”的反馈效果。
- 成功标准：
  - 仅 `store-card` 店铺任务卡参与本次改造。
  - 只在该卡片 GMV 数值发生变化时触发一次，不循环、不扫光横移、不让整张卡片闪烁。
  - 顶部边框、闪光层、柔光层的主色不写死为文档中的橘红，而是跟随当前卡片实际平台色 `--platform-color`。
  - 不改变卡片尺寸、布局、GMV/目标/进度/更新时间等信息结构。
- 范围外：
  - 不改 `total-card` 全渠道总卡。
  - 不改 `platform-summary-card` 平台汇总卡。
  - 不改后端、WebSocket、OCR、调度器、数据结构。

## Current State Analysis

- `frontend/dashboard.js`
  - `buildDashboardViewModel()` 已通过 `_previousGmvValues.stores` 与当前店铺 GMV 对比，给变化的店铺卡片设置 `changed = true`。
  - `renderStoreCard()` 会在 `store.changed` 为真时给卡片附加 `is-flashing` 类，这已经满足“只在 GMV 变化时触发”的业务口径。
  - 当前店铺卡 DOM 只有卡片主体内容，没有文档建议的 `.top-bloom` 柔光层。
  - 当前实时看板采用整段 HTML 重渲染，因此每次 `is-flashing` 重新出现时，CSS 动画会自然重新播放；不需要额外引入 `void card.offsetWidth` 之类的手动重绘逻辑。

- `frontend/styles.css`
  - `.store-card` 已具备 `position: relative` 和 `overflow: hidden`，可以直接承载 `::before`、`::after` 和内部柔光层。
  - `.store-card::before` 当前就是顶部边框，但现状是“扫光感”实现：`background-position` 和 `background-size` 会移动，不符合你提供文档里的“整条同时亮一下、禁止横向扫光”的要求。
  - `.store-card.is-flashing::before` 目前绑定的是 `store-border-sweep-anim`，属于左右移动的流光方案，需要替换。
  - 平台颜色已经通过 `.platform-taobao` / `.platform-jd` / `.platform-dewu` / `.platform-douyin` / `.platform-vip` / `.platform-other` 统一映射到 `--platform-color`，这正好可以作为本次动态配色来源。

- `frontend/core.js`
  - `platformMeta` 已定义各平台品牌色，与实时看板卡片颜色体系一致。
  - 本次不需要修改 `platformMeta`，只需在样式层消费现有 `--platform-color`。

## Assumptions & Decisions

- 已确认本次作用范围是“仅店铺任务卡（`store-card`）”，不扩散到总卡和平台汇总卡。
- 已确认文档中“橘红色渐变”只作为视觉目标参考，不作为固定色值直接写死；实现时必须改为基于每张卡片当前的 `--platform-color` 推导出常态边框、闪光层和柔光层颜色。
- 已确认要保留“顶部整体闪一下”的反馈语义：
  - 不允许 `translateX`、`background-position` 横向位移；
  - 不允许跑马灯；
  - 不允许无限循环；
  - 不允许整卡背景整体闪烁。
- 已确认不改变看板布局，因此不机械照搬文档里的 `28px` 圆角和完整示例尺寸，而是在当前 `store-card` 的既有尺寸、圆角、内边距基础上做等价视觉落地。

## Proposed Changes

### 1. `frontend/dashboard.js`

- 在 `renderStoreCard()` 生成的店铺卡片内部，新增一个位于内容最前面的 `<div class="top-bloom"></div>`。
- 保持原有业务字段结构不变：
  - 头部、GMV 数值、目标、进度条、更新时间都维持当前 DOM 层级和顺序。
- 继续沿用现有 `store.changed ? ' is-flashing' : ''` 触发方式，不改变化检测判定。
- 不新增新的 JS 动画触发函数，因为当前重渲染机制已能确保每次数据变化重新播放动画。

### 2. `frontend/styles.css`

- 保留 `.store-card` 现有容器职责，并补充/校准支持新效果所需的层级样式：
  - 继续使用 `position: relative`、`overflow: hidden`。
  - 为正文内容建立明确的层级，让 `.top-bloom` 位于内容下方，不遮挡文字和进度条。

- 重构 `.store-card::before` 为“常态顶部边框”：
  - 保留顶部条存在，但从现有细边改为更接近规范文档的“带体积感的顶部高亮条”。
  - 使用基于 `--platform-color` 的渐变，而不是固定橘红。
  - 使用 `color-mix()` 从 `--platform-color` 推导两侧较暗、中部更亮的渐变层次，并保留适度发光。

- 新增 `.store-card::after` 作为“顶部整体闪光层”：
  - 默认隐藏；
  - 仅在 `is-flashing` 时播放一次；
  - 动画只使用 `opacity` 和轻微 `scaleX`，不做横向移动；
  - 颜色由 `--platform-color` 推导，中心高亮允许混入少量暖白/浅金来形成“刷新高光”，但整体仍以卡片自身平台色为主。

- 新增 `.top-bloom` 作为“顶部柔光扩散层”：
  - 默认隐藏；
  - 位于卡片顶部内容下方；
  - 使用基于 `--platform-color` 的径向渐变和模糊，形成局部柔光，不扩散到整张卡片。

- 替换当前店铺卡动画：
  - 删除或停用 `store-border-sweep-anim` 在店铺卡上的绑定。
  - 新增三组更符合规范的动画，例如：
    - `store-top-flash`
    - `store-top-border-pulse`
    - `store-top-bloom-pulse`
  - 动画节奏参考文档：闪光主峰值快速出现并在约 `0.72s` 内淡出，边框增强和柔光恢复可略长，但都只执行一次。

- 保持其他卡片不受影响：
  - `platform-summary-card.is-flashing::before` 维持当前行为。
  - `total-card.is-flashing::before` 维持当前行为。
  - 若现有公共动画命名会互相影响，则把店铺卡动画完全拆分成独立命名。

### 3. 样式落地细则

- 颜色策略：
  - 常态边框：用 `--platform-color` 做主色，两侧通过 `color-mix(in srgb, var(--platform-color) ... , #04111d)` 压暗。
  - 闪光层：用 `--platform-color` 与白色/暖色做少量混合，形成“瞬时提亮”而不是改成固定橘红。
  - 柔光层：仍以 `--platform-color` 为主，透明度和模糊半径要克制，避免变成整卡染色。

- 几何策略：
  - 参考文档中的“顶部条 + 更厚闪光层 + 稍内缩柔光层”三层结构。
  - 数值以现有 `store-card` 尺寸为基准微调，不机械照搬 `left: 22px/right: 22px/height: 10px`，但视觉关系保持一致：
    - 常态条最窄；
    - 闪光层更厚且略上浮；
    - 柔光层略内缩并位于顶部下方。

- 层级策略：
  - `::before` 位于内容下方、背景上方。
  - `::after` 高于 `::before`。
  - `.top-bloom` 低于正文。
  - 文字、数字、进度条保持可读，不被模糊层覆盖。

## Risk Assessment

- 视觉风险：
  - 若高亮层过强，可能让顶部像告警态而不是“刷新反馈”。
  - 解决方式：限制柔光范围，只作用于顶部边框区域，避免整卡泛光。

- 兼容风险：
  - 现有样式大量使用 `color-mix()`，说明项目已接受该能力；本次继续沿用，不引入额外依赖。
  - 需要注意 `.store-card.is-alert` 仍有独立脉冲动画，避免与新的顶部闪光在视觉上冲突过重。

- 交互风险：
  - 若层级处理不对，可能遮挡徽标或文字。
  - 解决方式：在计划实施时明确正文层 `z-index`，并让 `.top-bloom` / `::after` 都 `pointer-events: none`。

- 回归风险：
  - 若误改公共动画或公共伪元素规则，可能波及平台卡和总卡。
  - 解决方式：店铺卡动画和闪光层全部用 `.store-card` 作用域单独定义。

## Execution Plan

1. 修改 `frontend/dashboard.js`
   - 在 `renderStoreCard()` 的卡片内部插入 `.top-bloom` 节点。
   - 保持原有 `is-flashing` 判定和现有字段结构不变。

2. 修改 `frontend/styles.css`
   - 重写 `.store-card::before` 常态顶部边框。
   - 新增 `.store-card::after` 与 `.top-bloom` 样式。
   - 新增仅服务于店铺卡的 3 组关键帧动画。
   - 用新动画替换 `.store-card.is-flashing::before` 当前的横向扫光方案。

3. 自检样式作用域
   - 确认 `.platform-summary-card`、`.total-card` 未被本次规则命中。
   - 确认告警态 `.store-card.is-alert` 仍保留原有边框脉冲逻辑。

## Verification Steps

- 静态检查
  - 检查 `renderStoreCard()` 新增的 `.top-bloom` 仅存在于店铺任务卡。
  - 检查店铺卡动画不再依赖 `background-position` 或其他横向位移属性。
  - 检查颜色来源均基于 `--platform-color`，没有把橘红色作为固定主色写死。

- 页面验收
  - 打开“实时看板”，确认店铺卡布局、尺寸、文案、进度条均未变化。
  - 当某个店铺 GMV 更新时，只有对应卡片顶部边框整体高亮一次。
  - 高亮表现为顶部整条同时闪亮，有局部柔光，但没有左右扫光和无限循环。
  - 多次连续更新时，每次都能重新触发。
  - 未变化的店铺卡不闪。
  - 平台汇总卡和总卡保持当前表现不变。

- 视觉验收
  - 各平台卡片仍呈现自己的平台色风格。
  - 天猫、京东、得物、抖音、唯品会、其他平台的闪光效果均跟随各自卡片颜色，而不是统一变成橘红色。
