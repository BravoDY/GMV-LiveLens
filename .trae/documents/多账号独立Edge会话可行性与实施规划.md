## Summary

- 目标：评估并规划 `GMV-LiveLens` 对“天猫生意参谋等场景下，不同 Edge 会话登录不同账号并长期并行采集”的支持方案。
- 用户目标已确认：
  - 优先采用“每店铺独立 Edge 会话，多账号长期并行”的模式。
  - 登录方式接受“每个会话首次手动登录一次，后续长期复用”。
  - 重点担心阿里系风控，不希望因为独立环境或自动化接管而明显提升被判定为爬虫/自动化的风险。
- 结论：
  - 从浏览器底层原理看，该需求是可行的，且比“同一真实个人浏览器同时登录多个账号”更符合 Chromium/Edge 的会话隔离机制。
  - 从当前项目代码基础看，项目已经具备 70% 以上的底座：`edge_session`、独立 `user_data_dir`、独立 `debug_port`、任务到会话绑定、截图/OCR 主链路都已存在。
  - 真正需要补齐的不是“多账号是否能做”，而是“风险可视化、模式声明、交互约束、运维规则、少交互采集策略”。

## Current State Analysis

### 1. 当前项目已经具备的能力

- `backend/services/shop_config.py`
  - 现状：`load_shop_configs()` 会为每个店铺生成唯一 `edge_session_id`、唯一 `debug_port`，并把 `user_data_dir` 指向 `data/edge_profiles/<session_id>`。
  - 含义：架构上已经是“每店铺一套独立浏览器资料目录”。
- `backend/services/store.py`
  - 现状：`edge_sessions` 表中持久化 `session_id/name/platform/shop_name/debug_port/user_data_dir`。
  - 现状：`upsert_edge_session()` 对非 `default_real_edge` 会自动补默认独立 profile 目录。
  - 含义：项目已经把“浏览器会话”当成一等实体，而不是临时进程。
- `backend/collectors/remote_edge.py`
  - 现状：`RemoteEdge._start_edge()` 启动 Edge 时显式传入 `--user-data-dir` 与 `--remote-debugging-port`。
  - 含义：每个会话启动的是独立 Chromium profile，天然支持不同 cookie、不同本地存储、不同登录态并行共存。
- `backend/services/scheduler.py`
  - 现状：主采集链路在 `capture_once()` 中对 `remote_edge` 模式只做：
    - 定位已绑定页面
    - `page.screenshot()`
    - OCR 识别
  - 含义：当前定时采集并不依赖持续点击、输入、滚动、翻页等强交互自动化。
- `frontend/app.js`
  - 现状：任务操作按钮已经绑定 `edge_session_id`，并支持启动/显示/隐藏/关闭 Edge 会话。
  - 含义：用户交互层已经默认接受“任务绑定到某个独立 Edge 会话”的产品模型。

### 2. 当前项目对“真实个人环境”的限制

- `backend/services/store.py`
  - 现状：保留了一个 `default_real_edge` 会话，`user_data_dir` 为空。
- `backend/collectors/remote_edge.py`
  - 现状：当 `user_data_dir` 为空时，会走 `--profile-directory=Default` 的真实 profile 逻辑。
  - 限制：若普通 Edge 已运行，代码会拒绝启动真实 profile 调试，因为 Chromium 的真实 profile 调试端口必须由首个进程决定。
- 结论：
  - “复用个人真实环境”天生不适合“多账号并行”。
  - “独立店铺会话”反而更符合你当前明确接受的目标。

### 3. 技术底层原理判断

- Chromium/Edge 的账号隔离本质上依赖 profile（用户数据目录）隔离，而不是窗口隔离。
- 同一个 profile 下，目标站点一般只能维持一套站点 cookie / localStorage / sessionStorage / IndexedDB 语义上的主登录态。
- 多开多个窗口但共用同一 `user-data-dir`，通常不能稳定实现同站点多账号长期并行。
- 为每个账号创建独立 `user-data-dir`，才是 Chromium 体系下最标准、最底层、最稳定的多账号并行方案。
- 因此，从浏览器内核数据模型上判断：
  - “不同 Edge 登录不同账户”是可行且合理的。
  - “同一个真实个人浏览器同时稳定登录多个生意参谋账号”反而不符合内核层的数据隔离逻辑。

### 4. 风控风险判断

- 独立 profile 本身不等于爬虫，也不天然等于高风险。
- 真正更容易触发风控的因素通常是：
  - 无头浏览器特征
  - 高频自动点击、输入、翻页、批量请求
  - 非人类节奏的页面操作
  - 登录态频繁切换
  - 异常 IP / 代理 / 网络出口
  - 浏览器环境短时间内反复重建
- 当前项目对风控较有利的点：
  - 启动的是实际 Edge，而不是 headless 浏览器。
  - 采集主链路主要是截图 + OCR，不依赖站点接口直抓。
  - 已接受“每会话首次手动登录”，避免自动登录脚本本身触发高风险行为。
- 当前项目仍有潜在风险的点：
  - 使用远程调试端口接管浏览器，环境与纯手动打开仍不完全一致。
  - 若未来加入自动跳转、自动点页签、自动刷新、自动搜索、自动输入等操作，风险会明显上升。
  - 若频繁新建全新 profile，而不是长期复用同一店铺 profile，会增加环境不稳定和站点异常感知概率。
- 综合结论：
  - 在“长期复用独立 profile + 手动登录 + 只做截图/OCR + 少交互”的前提下，风险可控，且大概率低于“复杂自动化脚本采集”。
  - 不能承诺“绝对零风控”，但从工程策略上这是当前最稳妥方向。

## Proposed Changes

### 1. 明确产品模式与会话语义

- 文件：`backend/models.py`
- 计划：
  - 为 `EdgeSession` 增加模式语义，建议新增字段如：
    - `session_mode`: `isolated` | `real_profile`
    - `risk_level`: `low` | `medium` | `high`（可为展示层派生，非必需）
- 为什么：
  - 当前只有 `user_data_dir` 是否为空来隐式区分模式，业务语义不够清晰。
  - 明确模式后，后端校验、前端提示、后续扩展更稳定。
- 难度：低

### 2. 扩展会话存储结构，显式支持“独立店铺模式”

- 文件：`backend/services/store.py`
- 计划：
  - 扩展 `edge_sessions` 表结构，增加 `session_mode`、可选 `notes` / `last_login_hint` 等辅助字段。
  - 保持 `default_real_edge` 为兼容保留，但默认新增/同步的店铺会话全部标记为 `isolated`。
  - 在 `upsert_edge_session()` 中增加模式校验：
    - `isolated` 必须有独立 `user_data_dir`
    - `real_profile` 可允许空 `user_data_dir`
- 为什么：
  - 让“多账号并行能力”从隐式行为升级为显式产品能力。
  - 为后续 UI 风险提示、模式筛选、运维排障提供稳定数据基础。
- 难度：中

### 3. 保持采集主链路“少交互”，避免把项目演化成高风险自动化脚本

- 文件：`backend/services/scheduler.py`
- 文件：`backend/collectors/remote_edge.py`
- 计划：
  - 保持定时采集阶段默认只做 `ensure_page()` + `screenshot_page()`。
  - 把“自动打开页面”“页面恢复”“强制跳转”“新标签拦截”等逻辑明确限定为人工辅助功能，不作为高频调度默认动作。
  - 为后续可选刷新行为预留开关，但默认关闭。
- 为什么：
  - 你的首要约束是降低风控，不是提高自动化程度。
  - 页面交互越多，越接近高风险自动化行为。
- 难度：中

### 4. 在前端会话管理中增加模式说明与风险提示

- 文件：`frontend/app.js`
- 可能涉及文件：`frontend/index.html`、`frontend/styles.css`
- 计划：
  - 在 Edge 会话列表/任务编辑处显示：
    - 会话模式：独立店铺 / 真实个人环境
    - 会话目录路径
    - 调试端口
    - 推荐用途说明
  - 在独立店铺模式旁明确提示：
    - “首次手动登录一次，后续长期复用”
    - “建议长期固定同一账号对应同一会话，不要混用”
  - 在真实环境模式旁明确提示：
    - “不适合多账号并行”
    - “启动前需关闭全部 Edge”
- 为什么：
  - 你当前的困惑本质上是产品语义没有清楚表达，用户容易误以为“启动Edge”就是“启动我平时那个真实 Edge”。
- 难度：中

### 5. 增加风控友好的运维约束与状态校验

- 文件：`backend/main.py`
- 文件：`backend/collectors/remote_edge.py`
- 计划：
  - 新增或增强健康信息返回：
    - 当前会话模式
    - `user_data_dir`
    - 是否首次启动 / profile 是否为空壳
    - 是否已检测到业务页
  - 增加接口级错误提示：
    - 当独立会话尚未登录时，提示“请手动登录后再采集”
    - 当真实环境模式与已运行 Edge 冲突时，明确提示冲突原因
- 为什么：
  - 多账号能力本身可行，但真正影响使用体验的是“为什么这次能采集、那次不行”。
  - 健康状态必须对用户透明，避免误判成风控问题。
- 难度：中

### 6. 将“多账号能力”沉淀为明确的使用规范，而不是隐含知识

- 文件：`README.md`
- 可选文件：新增 `docs/` 或 `.trae/documents/` 说明文档
- 计划：
  - 补充《多账号 Edge 会话使用规范》：
    - 一个店铺/账号固定绑定一个会话
    - 首次人工登录，后续长期复用
    - 不随意删除对应 profile 目录
    - 不同账号不要混用同一会话
    - 非必要不要在调度中加入点击、输入、翻页
  - 补充《风控边界说明》：
    - 项目不是接口爬虫
    - 项目默认策略是页面截图/OCR
    - 若后续启用自动化交互，风险会提升
- 为什么：
  - 这是中小型商业项目里很关键的“运维制度化”内容，能显著降低后续误用成本。
- 难度：低

## Assumptions & Decisions

- 已确认的业务决策：
  - 优先目标是“每店铺独立 Edge 会话，多账号长期并行”。
  - 登录方式采用“每会话首次人工登录，后续长期复用”。
- 本次规划的底层判断：
  - 多账号并行依赖 profile 隔离，不依赖窗口隔离。
  - 独立 `user_data_dir` 是正确方向，不是偏门方案。
- 本次建议的风险边界：
  - 默认不做自动登录。
  - 默认不做高频页面交互。
  - 默认不走站点接口抓包/重放。
  - 默认不频繁创建全新 profile。
- 与当前代码兼容性的判断：
  - 当前数据库与会话模型具备演进空间。
  - 当前后端接口和调度架构不需要推倒重来。
  - 核心是“显式化模式 + 增强提示 + 加强约束”，不是重做浏览器内核接入层。

## Feasibility & Difficulty

- 可实现性：高
  - 现有项目已经把最难的浏览器多会话底座搭好了。
- 技术风险：中
  - 主要不是“做不出来”，而是“如何保持低交互、低风控”。
- 改造难度：中
  - 如果只做产品语义明确化、会话模式显式化、提示与约束补齐，难度不高。
  - 如果以后还要加入账号初始化流程、风控诊断、profile 导入导出，则复杂度会继续上升。
- 推荐实施顺序：
  - 第一阶段：明确模式、补 UI 提示、保留少交互采集策略
  - 第二阶段：增强健康状态、未登录检测、会话管理能力
  - 第三阶段：补充运维文档和排障机制

## Verification Steps

- 代码层验证：
  - 确认 `shop_config.load_shop_configs()` 生成的每个店铺都对应唯一 `edge_session_id`、唯一 `debug_port`、唯一 `user_data_dir`
  - 确认 `scheduler.capture_once()` 在 `remote_edge` 模式下主链路仍以截图/OCR为主
  - 确认 `RemoteEdge._start_edge()` 的启动参数仍保持真实 Edge + 独立 profile 模式
- 业务层验证：
  - 为两个不同店铺会话分别手动登录不同生意参谋账号
  - 重启项目后再次启动对应会话，确认各自登录态独立保留
  - 同时运行多个独立会话，确认不会互相串号
- 风险层验证：
  - 在不增加自动交互的前提下，持续观察登录稳定性、二次验证频率、异常退出率
  - 若出现风控异常，优先排查：
    - 是否混用了 profile
    - 是否频繁重建 profile
    - 是否引入了额外自动点击/跳转
    - 是否网络/IP 环境异常
