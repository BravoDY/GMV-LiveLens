## Summary
- 问题现象：
  - 用户已执行“一键初始化默认店铺”。
  - 已启动`天猫-官方旗舰店`对应 Edge。
  - 页面顶部显示当前可“生成预览”。
  - 实际点击后没有预览图，流程又跳回“步骤 1：重新选择页面”。
- 核心结论：
  - 这不是“预览按钮失效”，而是当前任务里保存的`page_id`已经失效，前端却仍把它当作可直接预览的有效绑定页。
  - 后端在真正截图时返回`remote_page_not_found`，前端收到后按现有严格策略回退到“重新选择页面”。
- 目标：
  - 让采集配置在进入任务时就能准确判断“当前绑定页是否仍有效”。
  - 只有在绑定页真实有效时才显示“生成预览”。
  - 如果绑定页已失效，应在进入页面时就直接进入“重新选择页面”状态，而不是先误导用户点击“生成预览”。

## Current State Analysis
- `frontend/config.js`
  - `setupFlowState()`当前只要发现`task.page_id`存在，就会把任务推进到“生成预览”阶段。
  - 它没有先验证这个`page_id`是否仍存在于当前 Edge 会话。
  - 这会导致“界面看起来可以预览，但实际预览一定失败”的假阳性状态。
- `frontend/edge.js`
  - `previewRemotePage()`会直接用`task.page_id`去请求`/api/edge-sessions/{session_id}/pages/{page_id}/preview`。
  - 若后端返回`remote_page_not_found`，前端就立刻切回“步骤 1：重新选择页面”。
  - 现有逻辑是“失败后纠正”，不是“进入前预判”。
- `backend/main.py`
  - `POST /api/edge-sessions/{session_id}/pages/{page_id}/preview`已会返回结构化`reason_code=remote_page_not_found`。
  - `GET /api/tasks/{task_id}/page-candidates`已能判断当前任务绑定页是否还存在，并返回：
    - `flow_state`
    - `next_action`
    - `recovery_hint`
  - 但前端当前并没有把这个接口作为“进入任务后的第一真相来源”。
- `backend/collectors/remote_edge.py`
  - `page_id`本质是运行时页面句柄，不是长期稳定标识。
  - 一旦发生以下任一情况，就可能失效：
    - 重启后端
    - 关闭并重开 Edge
    - 页面对象重建
    - 会话重新连接
  - `screenshot_page()`在找不到页时会抛`remote_page_not_found`，这正是当前跳回重选页面的直接触发点。
- `backend/services/store.py`
  - 任务会持久化`page_id/page_url/page_title`。
  - 其中`page_id`是临时运行时标识，却被当前前端当成“足够代表绑定仍有效”的判断依据，这是设计错位。

## Root Cause
### 根因 1：把`page_id`当成了长期稳定绑定依据
- 当前实现里，`page_id`被持久化后，前端把“有`page_id`”等同于“可直接预览”。
- 但`page_id`本质上只在当前运行时和当前页面对象生命周期内可靠。

### 根因 2：流程状态判断发生在真正校验之前
- 现状是：
  1. 前端进入任务
  2. 看见`task.page_id`非空
  3. UI进入“生成预览”
  4. 用户点击预览
  5. 后端才发现`page_id`已失效
  6. 前端再退回“重新选择页面”
- 这造成了明显的产品体验断层。

### 根因 3：当前任务进入采集配置时没有做“绑定页有效性预检查”
- 虽然后端已有`/api/tasks/{task_id}/page-candidates`，能判断当前页是否还在。
- 但前端在`loadTaskIntoConfig()` / `syncTaskContext()`阶段并没有优先利用这个结果来设定真实流程状态。

## Requirement Understanding
- 用户真正要的不是“报错后告诉我去重选”。
- 用户要的是：
  - 如果当前绑定页仍有效，界面就稳定显示“生成预览”。
  - 如果当前绑定页已经失效，界面一进来就明确告诉我“这页已经失效，需要重新选择页面”，而不是先误导我点预览。
  - 整条采集配置链路要清晰、可信、可预期。

## Best Implementation
### 总体原则
- 保持此前已经确认的产品方向：
  - 不自动偷偷恢复
  - 不自动新开页
  - 仍然由用户手工重新选择 OCR 页签
- 但要把“失效判断”前置，而不是等点击预览时再暴露。

### 推荐方案
- 把`/api/tasks/{task_id}/page-candidates`升级为“当前任务绑定状态真相接口”。
- 进入采集配置时，先用它判断：
  - 当前是否有绑定页
  - 当前绑定页是否真实存在
  - 当前是否应进入：
    - `waiting_page`
    - `page_selected`
    - `rebind_required`
- 前端工作台状态改为优先使用这份检查结果，而不是只看`task.page_id`。

## Proposed Changes
### 1. 进入采集配置时先做绑定页有效性检查
- 文件：`frontend/config.js`
- 变更：
  - 在`loadTaskIntoConfig()`或`syncTaskContext()`里，增加“当前任务页签候选预加载”逻辑。
  - 进入任务后自动请求`GET /api/tasks/{task_id}/page-candidates`。
  - 根据返回的`flow_state`决定当前工作台状态，而不是仅按`task.page_id`判断。
- 作用：
  - 避免页面错误显示“生成预览”。
  - 用户一进来就能看到真实状态。

### 2. 前端状态机改为以“绑定有效性”驱动，而不是以“page_id 非空”驱动
- 文件：`frontend/config.js`
- 变更：
  - `setupFlowState()`新增对当前候选状态的依赖。
  - 优先级调整为：
    1. 若`bindCandidates.flow_state === rebind_required`，直接显示“步骤 1：重新选择页面”
    2. 若无有效绑定，显示“步骤 1：选择页面”
    3. 若存在有效绑定，再进入“生成预览”
  - 不再使用“`task.page_id`非空就可以预览”的简化逻辑。
- 作用：
  - 修正当前产品误导。

### 3. 当前任务聚焦时自动渲染页签恢复卡片
- 文件：`frontend/config.js`
- 变更：
  - 若`page-candidates`返回`rebind_required`：
    - 直接渲染恢复提示卡片
    - 显示推荐操作：
      - 去任务管理确认后台页
      - 回来重新扫描
      - 重新手选页签
  - 同时自动渲染当前会话现有页签列表，减少用户多点一次“扫描”的成本
    - 这里不属于自动恢复，只是自动把可选列表展示出来
- 作用：
  - 用户进入页面后立刻知道怎么处理。

### 4. `previewRemotePage()`只保留最终兜底，不再承担主状态切换责任
- 文件：`frontend/edge.js`
- 变更：
  - 保留现有`remote_page_not_found`兜底分支。
  - 但产品目标改为：大多数情况下，用户在点击“生成预览”前就已经被前置判断拦住，不会再走到这个失败分支。
- 作用：
  - 让“点击预览失败 -> 再回退”从主流程变成异常兜底。

### 5. 明确区分“稳定绑定信息”和“运行时页签句柄”
- 文件：`backend/services/store.py`、`frontend/config.js`、`backend/main.py`
- 变更：
  - 继续保留`page_id`，但在前端与文案语义上，不再把它视为“稳定有效”的唯一依据。
  - 前端展示与状态流转应更多依赖：
    - 当前会话扫描结果
    - 当前绑定页是否还存在
    - `page_url / page_title / target_page_url`辅助匹配结果
- 作用：
  - 让系统设计更符合真实运行机制。

### 6. 优化“已启动 Edge 但没有预览”的错误提示文案
- 文件：`frontend/config.js`、`frontend/edge.js`
- 变更：
  - 把提示从单纯“页签已失效”升级为更可理解的说明：
    - “当前任务记录的是旧页签句柄，虽然 Edge 已启动，但这次会话里找不到原绑定页，因此不能直接生成预览。”
    - “请从当前会话现有页签中重新选择要 OCR 的页面。”
- 作用：
  - 用户能理解“为什么 Edge 明明开着，还是不能预览”。

## Assumptions & Decisions
- 已确认并沿用的决策：
  - 不自动恢复绑定
  - 不自动新建页签
  - 需要明确提示当前发生了什么以及用户下一步该怎么做
- 本次新增决策：
  - 将“绑定页有效性预检查”前置为进入采集配置时的默认动作
  - 将`page-candidates`作为当前任务流程状态的主依据
  - 允许自动展示当前会话的现有页签列表，但不自动替用户完成绑定

## Risk Assessment
- 可能影响的现有功能：
  - 进入采集配置时会多一次任务级页签候选请求。
  - 如果 Edge 会话未接通，页面会更早暴露“会话未就绪”状态。
- 兼容性风险：
  - 需要确保`scanBind()`的手工扫描逻辑与“进入时自动预检查”不冲突。
  - 需要避免自动预检查把用户主动选择的临时会话覆盖掉。
- 为什么风险可控：
  - 本次不改变底层抓图能力，只修正状态判断顺序和交互提示。

## Verification Steps
- 场景 1：任务保存了旧`page_id`，但当前会话没有原页
  - 进入采集配置时，应直接显示“步骤 1：重新选择页面”
  - 不应先显示“生成预览”
- 场景 2：任务绑定页仍有效
  - 进入采集配置时，应直接显示“步骤 2：生成预览”
  - 点击后应正常显示预览图
- 场景 3：Edge 已启动但当前后台打开的是别的页
  - 进入采集配置时，应展示当前会话页签列表
  - 用户可重新选择正确页签并重新绑定
- 场景 4：Edge 未启动
  - 进入采集配置时，应直接提示“会话未就绪”
  - 不应显示可直接预览
