# ddddocr 学习机制仍出现异常 OCR 的深层原因与闭环修复计划

## 1. 摘要

本问题不是单点的 `ddddocr` 模型精度问题，而是一个“模型输出 -> 预处理变体 -> 规则纠错 -> 候选评分 -> 调度确认 -> 历史样本沉淀”组成的系统性问题。当前仓库已经具备三类基础能力：样本自动沉淀、候选来源归因、离线异常分析；但“学习机制”仍主要停留在记录和观察层，没有形成足够强的运行时闭环，因此仍会出现“错得很异常，甚至错得很稳定”的 OCR 结果。

本计划的目标不是盲目继续强化 `ddddocr`，而是定位并修复以下深层原因：

1. 所谓“学习机制”并非在线学习，也不会自动更新 `ddddocr` 模型。
2. 当前 OCR 主链路允许 `joined_text` 聚合文本和强纠错结果参与高分竞争，容易放大局部错误。
3. `ddddocr` 对强二值化和结构弱文本天然更脆弱，某些店铺的图像特征会持续诱发稳定误识别。
4. 历史样本虽已保存，但尚未沉淀为“可自动限制线上策略”的结构化证据。
5. 调度器虽然已有二次确认机制，但在“低置信来源 + 连续重复错误”场景下，过去仍可能接受错误值。

最终方案以“先解释清楚，再补闭环，再做验证”为原则，在不破坏现有已跑通主链路的前提下，收紧 `ddddocr` 适用场景，并把学习机制从“样本记录型”升级为“规则反哺 + 决策护栏型”。

## 2. 需求理解

用户真正要解决的问题是：

- 为什么项目里已经加入了所谓的 `ddddocr` 学习机制，但线上采集时仍会产出特别异常的 OCR 数值；
- 需要找出真正的深层原因，而不是只给出“模型不准”“样本不够”这种表层结论；
- 需要形成后续可执行的改造与验证方案，确保当前已跑通项目不被破坏。

成功标准：

- 能清晰说明当前“学习机制”到底是什么、不是什麽；
- 能指出异常 OCR 是在哪一层被放大并最终进入可信值链路；
- 能给出具体到文件和行为的改造方案；
- 能定义验证办法，证明改造确实降低“异常 OCR 被接受”的概率。

## 3. 当前结构判断

### 3.1 OCR 主链路

- `backend/collectors/ocr_reader.py`
  - 负责 OCR 引擎选择、图像预处理、引擎执行、候选提取与打分。
  - 当前支持 `rapidocr`、`legacy_rapidocr`、`paddleocr`、`ddddocr`、`tesseract`。
  - `read_text()` 会遍历多个预处理变体和多个引擎，拿到全文本和细粒度结果。
  - `extract_candidates()` 会对 OCR 文本做形近字纠错，再用正则和打分逻辑提取金额候选。

### 3.2 可信值接受链路

- `backend/services/scheduler.py`
  - 负责截图、调用 `read_text()`、提取候选、选择首候选，再通过 `_judge()` 决定是否接受为可信 GMV。
  - 当前已存在 `confirm_count`、异常跳变判断、跨天处理、以及基于来源的附加确认护栏。

### 3.3 学习/样本沉淀链路

- `backend/services/scheduler.py`
  - `_save_ocr_dataset()` 会把识别裁剪图自动落到 `data/ocr_datasets/`。
- `backend/services/store.py`
  - `gmv_samples` 表记录 `ocr_text`、`candidates_json`、`selected_value`、`trusted_value`、`status`、`reason`、`screenshot_path`。
- `backend/tools/find_ocr_anomalies.py`
  - 会扫描历史 `gmv_samples`，统计内部异常字符、前缀异常字符、人工纠错驱动的新规则候选、店铺级异常聚集。

### 3.4 接口与可观测性

- `backend/main.py`
  - `/api/test-ocr` 是 OCR 测试入口；
  - `/api/tasks/{task_id}/manual-correction` 支持人工纠错；
  - `/api/tasks/{task_id}/samples` 可查看历史样本。
- `frontend/config.js`
  - 当前已具备展示候选来源信息的能力，可直观看到候选来自哪个引擎、哪个变体、何种来源、纠错次数多少。

## 4. 深层原因分析

### 4.1 “学习机制”被误解为模型会自动变准

当前仓库中的“学习机制”本质上是：

- 自动保存裁剪图到 `data/ocr_datasets/`；
- 自动记录 `gmv_samples` 历史；
- 允许人工纠错；
- 用 `find_ocr_anomalies.py` 做离线统计。

它不是：

- `ddddocr` 在线训练；
- 识别后自动增量微调模型；
- 自动把人工纠错样本回灌到 `ddddocr` 权重；
- 自动把新异常字符或店铺特征转成线上规则。

这意味着即便系统“记住了更多样本”，下一次识别也不会天然更准，除非有人把这些样本进一步转化为规则、实验配置或新模型。

### 4.2 预处理与引擎耦合问题会放大 ddddocr 的结构性误识别

在 `ocr_reader.py` 中，项目会生成多个预处理变体，例如：

- `color`
- `contrast_gray`
- `yellow_digits`
- `otsu`
- 可选的 `adaptive_binary`
- 可选的 `sharpen`
- 可选的 `invert_gray`

`ddddocr` 的特点决定了它对字符级验证码类文本更敏感，但对复杂背景、非标准金额排版、强二值化后的断裂数字更容易输出“结构上像数字、语义上很离谱”的字符串。深层风险包括：

- 强二值化把一个数字切裂成多个局部形状；
- 紫底黄字、描边、辉光、压缩噪点会让 `ddddocr` 过拟合局部轮廓；
- 没有检测框时，`ddddocr` 只能对整块图做单段分类，容易把杂质、单位、边缘高光一起吞进去。

因此，异常 OCR 并不一定是模型没“学会”，更常见的是图像输入形态已经把模型诱导进错误模式。

### 4.3 当前候选提取对“聚合文本”和“纠错后文本”仍有放大效应

`extract_candidates()` 当前会：

- 先对 `ocr_text` 和 `details[*].text` 做形近字替换；
- 同时从 `joined_text` 聚合文本和每条 detail 文本中提取候选；
- 统一进入 `_candidate_reason()` 打分。

这会带来几个深层问题：

1. 聚合文本会把多个引擎、多种变体、多段文本拼在一起，容易凭借“碰巧凑出的完整数字”产生高分假候选。
2. 字符纠错虽然能修复真实误识别，但也可能把本来就错的字符串修成“合法的大数字”。
3. 当前打分侧重金额结构、位数、货币符和相对历史值，但对“该值来自低置信组合路径”过去约束不够强。

结果是：系统并不是直接相信 `ddddocr`，而是在“错误输出 + 全局纠错 + joined_text 拼接 + 金额评分”组合后，间接把异常值放大。

### 4.4 历史样本记录了结果，但没有记录足够完整的失败因果

`gmv_samples` 当前保存：

- 原始 OCR 文本；
- 候选列表；
- selected/trusted/status/reason；
- 截图路径。

但仍缺少一些对“学习闭环”很关键的结构化信息，例如：

- 本次是否因为 `ddddocr` 单独命中而被提名；
- 各引擎各变体的原始排名与淘汰原因；
- 被接受前经历了几次确认；
- 同一错误是否在同一店铺重复出现；
- 当前店铺实验配置是否生效；
- 哪类来源在历史上最常导致人工纠错。

没有这些结构化证据，就很难把样本直接反推成“下次线上要限制什么”。

### 4.5 店铺差异大于全局差异，导致全局学习很难直接奏效

`find_ocr_anomalies.py` 当前已经开始按店铺聚类异常，说明问题分布并不均匀。深层含义是：

- 有些店铺的主题色、字号、描边、单位摆放方式，会稳定诱发同一类误识别；
- 另一些店铺用同样的全局 OCR 规则却没有问题；
- 如果把所有异常都通过全局字符替换修复，容易伤及正常店铺；
- 因此真正有效的“学习”应优先是店铺级策略学习，而不是全局无差别强化。

### 4.6 调度器确认机制过去能防“随机错”，但不一定防“稳定错”

`_judge()` 的连续确认逻辑本来用于防止偶发抖动，但 OCR 世界里更危险的是：

- 某个错误值在相同截图、相同店铺、相同预处理下会连续多次出现；
- 它看起来很稳定，于是容易通过确认机制；
- 一旦被收为可信值，后续又会反向影响“非递减”“相近值”判断。

也就是说，连续确认并不天然等于可信，尤其在 `ddddocr` 这类可能稳定输出错误模式的场景中更是如此。

## 5. 风险评估

### 5.1 现有功能风险

- 若直接全局削弱 `ddddocr`，可能会影响极少数本来依赖它识别成功的店铺。
- 若过度增加确认门槛，可能导致实时性下降，出现更多 `pending_confirm`。
- 若过度扩大全局纠错库，可能把正常字符误改成数字，制造新的异常值。

### 5.2 兼容性风险

- `gmv_samples` 当前结构较轻，若后续新增更细字段，需要保证历史兼容读取。
- `/api/test-ocr` 与前端候选展示若继续增加调试字段，需要保持旧字段不删除。

### 5.3 数据风险

- 误接受异常 OCR 会污染 `last_trusted_value`，并影响后续连续确认逻辑。
- 若只保留图片样本、不保留行为上下文，样本再多也无法形成强学习闭环。

### 5.4 性能风险

- OCR 已经是多引擎、多变体串行执行；若再无控制地加更多变体，会放大单次采集耗时。
- 更合理的方向是减少 `ddddocr` 参与面，而不是继续叠更多试错变体。

## 6. 推荐实现方案

推荐方案分为“解释层、约束层、回灌层、验证层”四部分，优先顺序从高到低如下。

### 6.1 第一优先级：收紧 ddddocr 的默认参与条件

**修改文件**

- `backend/collectors/ocr_reader.py`

**改什么**

- 继续把 `ddddocr` 限制在低风险变体上，默认仅允许 `color`、`contrast_gray`、`sharpen` 这类相对稳态输入；
- 禁止 `ddddocr` 默认参与 `yellow_digits`、`otsu`、`adaptive_binary`、`invert_gray` 这类更容易结构性失真的变体；
- 保留 `ocr_experiment_profiles` 店铺级放开口子，仅对确有证据的店铺局部开启。

**为什么这样设计**

- 问题核心不是“让 `ddddocr` 试更多图”，而是“避免它在易错图形态上胡乱给出高置信文本”。
- 先减输入噪声，再谈学习，比盲目扩展预处理更稳。

### 6.2 第二优先级：把候选来源风险正式纳入接受判定

**修改文件**

- `backend/services/scheduler.py`
- 可能补充 `backend/collectors/ocr_reader.py`

**改什么**

- 把以下来源持续视为低置信，需要额外确认：
  - `engine == "ddddocr"`
  - `source_kind == "joined_text"`
  - `correction_count >= 2`
  - 强二值化变体叠加纠错
  - 缺少 `currency / separator / thousand_sep` 等结构信号
- 对“低置信来源 + 异常跳变”组合执行更严格确认；
- 对“连续重复但低置信”的候选，不因稳定就直接视为可信。

**为什么这样设计**

- 这是在运行时拦截“错得很稳定”的关键一步，能直接回应用户看到的异常值问题。

### 6.3 第三优先级：把学习机制从“看报告”升级为“能反哺规则”

**修改文件**

- `backend/tools/find_ocr_anomalies.py`
- `backend/collectors/ocr_reader.py`
- 视需要补充 `.trae/documents` 中对应说明文档

**改什么**

- 将异常分析脚本的输出重点转向三类结论：
  - 建议新增到 `OCR_CHAR_REPLACEMENTS` 的候选字符；
  - 建议新增到 `OCR_CURRENCY_ALIASES` 的前缀字符；
  - 建议使用店铺级实验配置而非全局修复的异常聚集店铺。
- 明确“何时更新全局规则、何时只做店铺级实验”的判定标准。

**为什么这样设计**

- 真正的学习闭环不是保存更多图片，而是把高频、可复现、低风险的异常模式转成线上可执行策略。

### 6.4 第四优先级：补足样本上下文，让后续学习有抓手

**修改文件**

- `backend/services/store.py`
- `backend/services/scheduler.py`
- `backend/main.py`

**改什么**

- 评估是否为 `gmv_samples` 增加更细的上下文字段，至少考虑：
  - `selected_candidate_engine`
  - `selected_candidate_variant`
  - `selected_candidate_source_kind`
  - `selected_candidate_correction_count`
  - `required_confirms`
  - `accepted_after_confirms`
- 若不改表结构，则至少保证现有 `candidates_json` 与 `reason` 能足够表达这些信息，并在接口层稳定返回。

**为什么这样设计**

- 否则以后只能知道“错了”，却不知道“为什么这次被接受”。

### 6.5 第五优先级：补接口与前端可观测性，帮助人工定位异常链路

**修改文件**

- `backend/main.py`
- `frontend/config.js`

**改什么**

- `/api/test-ocr` 持续返回 `suggested_candidate` 及来源归因；
- 前端测试区明确展示候选的 `engine / variant / source_kind / correction_count / reason`；
- 让人工一眼能看出异常值到底是 `ddddocr` 本体、聚合文本拼接、还是纠错放大导致。

**为什么这样设计**

- 如果没有可观测性，后续每次出现异常 OCR 都只能重复猜。

## 7. 文件级改造清单

| 文件 | 改动内容 | 原因 |
|---|---|---|
| `backend/collectors/ocr_reader.py` | 收紧 `ddddocr` 默认变体；保留店铺级实验口；补齐候选来源字段与纠错计数使用边界 | 避免 `ddddocr` 在高风险预处理上输出结构性异常文本 |
| `backend/services/scheduler.py` | 强化 `_candidate_guard_reason()` 和 `_judge()` 对低置信来源的确认门槛 | 阻止“稳定错误”被连续确认后收为可信 |
| `backend/tools/find_ocr_anomalies.py` | 输出更聚焦的规则候选和店铺级异常聚集结论 | 让历史样本真正反哺线上规则 |
| `backend/services/store.py` | 评估是否补充 `gmv_samples` 结构化上下文或至少统一表达现有候选归因 | 给后续学习闭环提供可复盘证据 |
| `backend/main.py` | 稳定暴露 `suggested_candidate` 与调试字段；人工纠错链路保持可追踪 | 让接口层成为排查与验证入口 |
| `frontend/config.js` | 展示候选来源和修复信号 | 帮助人工快速判断异常来源 |

## 8. 假设与决策

### 8.1 已确认假设

- 当前项目中的“学习机制”并不是模型在线学习。
- 异常 OCR 的深层问题已经跨越模型层，包含规则层、评分层和调度接受层。
- 当前项目以稳定、低回归风险为优先，不能为了追求理论最优识别率破坏主链路。

### 8.2 关键决策

- 决策 1：不把“继续喂更多样本给 `ddddocr`”作为主解法。
- 决策 2：优先建设“来源归因 + 调度护栏 + 规则回灌”闭环。
- 决策 3：优先做店铺级实验收敛，而不是全局大范围放开强预处理。
- 决策 4：验证重点放在“减少异常值被接受”，而不是只看单次 OCR 文本是否变化。

### 8.3 不推荐但更简单的方案

- 方案：完全移除 `ddddocr`。
  - 不推荐原因：太粗暴，可能误伤少数确实依赖它的场景，也无法解释现有异常是如何被系统接受的。
- 方案：无限扩展字符替换库。
  - 不推荐原因：会把局部问题误升级成全局规则，增加误替换风险。
- 方案：单纯提高 `confirm_count`。
  - 不推荐原因：只能拖慢接受速度，无法识别“稳定错误”。

## 9. 执行计划

1. 先基于当前代码确认 `ddddocr` 的默认参与变体、候选来源、调度护栏、样本沉淀字段是否完整对齐。
2. 在 `ocr_reader.py` 明确收紧 `ddddocr` 场景边界，并保留店铺级实验口，避免全局硬编码扩散。
3. 在 `scheduler.py` 强化低置信来源识别和确认门槛，阻止稳定错误穿透到可信值。
4. 在 `find_ocr_anomalies.py` 形成“全局规则候选 / 店铺级实验候选 / 待观察字符”三类明确输出。
5. 通过 `main.py` 与前端测试面板稳定暴露候选归因，确保异常能被人肉快速定位。
6. 评估是否增强 `gmv_samples` 记录字段；若成本过高，则至少保证现有 `candidates_json + reason` 足够支持复盘。
7. 完成静态检查、脚本验证、接口冒烟和主链路验证，确认没有破坏现有采集流程。

## 10. 验证步骤

### 10.1 静态与结构检查

- 检查 `ocr_reader.py`、`scheduler.py`、`find_ocr_anomalies.py`、`main.py` 是否存在语法或导入错误。
- 检查新增字段或返回结构是否与当前接口兼容。

### 10.2 样本级验证

- 使用历史异常截图或 `data/ocr_samples/` 中样本，验证：
  - 候选是否带有 `engine / variant / source_kind / correction_count`；
  - `ddddocr` 是否不再默认跑高风险变体；
  - 明显异常候选是否会被标注为低置信来源。

### 10.3 脚本级验证

- 运行 `backend/tools/find_ocr_anomalies.py`，确认：
  - 能正常读取数据库；
  - 能输出新增规则候选；
  - 能输出店铺级异常聚集；
  - Windows 终端下不再因 emoji 编码导致失败。

### 10.4 接口级验证

- 调用 `/api/test-ocr`，确认返回中包含：
  - `suggested_candidate`
  - `candidates[*].engine`
  - `candidates[*].variant`
  - `candidates[*].source_kind`
  - `candidates[*].correction_count`

### 10.5 主链路冒烟验证

- 触发一次真实或近真实 `capture_once` 链路，确认：
  - 正常候选仍可被接受；
  - 低置信异常候选会进入更严格确认，而不是直接写入可信值；
  - `gmv_samples` 与任务运行态仍正常更新。

## 11. 回归关注点

- 不能破坏原本依赖 `rapidocr` 或 `paddleocr` 的正常店铺。
- 不能导致所有任务大量停留在 `pending_confirm`。
- 不能让人工纠错、样本查看、测试 OCR 接口失效。
- 不能因为新增调试字段而破坏前端现有渲染。

## 12. 预期结论

本次改造完成后，系统会把“学习机制”从“我保存了很多图，所以应该更聪明”改为“我知道哪些来源危险、哪些规则可回灌、哪些店铺该局部收敛，因此异常 OCR 更难穿透到可信值”。这才是对当前问题最稳妥、最接近商业项目可维护形态的解决路径。
