# 任务唯一约束永久修复计划

## Summary
- 目标：从根上杜绝同一店铺出现多条实时 GMV 任务，不再依赖“同步后清理重复”兜底。
- 用户偏好：不需要兼顾历史数据，优先选择“永远解决问题”的方案。
- 结论：应以 **`capture_tasks(platform, shop_name)` 数据库唯一约束** 作为最终方案，而不是仅靠代码层去重，也不是优先把 `edge_session_id` 设为唯一。
- 成功标准：
  - 数据库层无法再插入第二条同 `platform + shop_name` 的任务。
  - 前端保存、启动同步、`/api/tasks`、`/api/shops/init` 仍正常工作。
  - 当前“天猫 / 儿童旗舰店”只保留一条任务，之后也不会再次出现重复。

## Current State Analysis
- 当前 `capture_tasks` 建表语句没有任何唯一约束，见 `backend/services/store.py` 中 `CREATE TABLE capture_tasks`。
- 当前防重主要依赖两层：
  - `shops.csv` 读取层会拦截重复 `(platform, shop_name)`，见 `backend/services/shop_config.py`；
  - `sync_tasks_with_shop_configs()` 当前已加上“同步后去重”，见 `backend/services/store.py`。
- 但这两层都不是数据库硬约束，仍存在两个根本风险：
  - 任何绕过同步流程的写入口，只要直接 `INSERT`，数据库仍会接受重复任务。
  - 当前 `upsert_task()` 在 `payload` 不带 `id` 时仍会走纯插入路径，见 `backend/services/store.py`；如果将来前端上下文丢失或出现异常调用，理论上仍可能制造重复。
- 现有前端保存入口 `saveTask()` 是 `POST /api/tasks`，并依赖 `editingTaskId` 作为更新锚点，见 `frontend/config.js`。这说明现有保存流不是天然的“按店铺幂等 upsert”。
- 现有 `edge_session_id` 来自 `platform + shop_name` 的派生规则，见 `backend/services/shop_config.py`。在当前业务模型下，它通常和店铺一一对应，但它更像“技术会话标识”，不是最稳的业务主键。
- 业务事实已经非常明确：
  - 你关注的是“每个店铺唯一的一条实时 GMV 任务”；
  - 不需要保留一店多任务的历史兼容；
  - 因此数据库唯一键应当跟随**店铺业务身份**，而不是只跟随运行时 Edge 会话。

## Proposed Changes
### 1. 数据库层：为 `capture_tasks(platform, shop_name)` 增加唯一约束
- 文件：`backend/services/store.py`
- 变更内容：
  - 在初始化/迁移阶段，先执行“重复任务清理”。
  - 清理完成后，创建唯一索引，例如：
    - `CREATE UNIQUE INDEX IF NOT EXISTS idx_capture_tasks_platform_shop_unique ON capture_tasks(platform, shop_name)`
  - 将唯一索引创建放在 `init_db()`/迁移链路中，确保服务启动即可收敛库结构。
- 为什么选这个：
  - `platform + shop_name` 才是你真正关心的“一个店铺一条任务”业务主键。
  - 即使未来 Edge 会话命名规则调整，只要店铺身份不变，唯一约束仍成立。

### 2. 数据清理：保留主任务，删除重复任务后再上唯一索引
- 文件：`backend/services/store.py`
- 变更内容：
  - 复用或加强现有 `_dedupe_tasks_for_configs()`。
  - 在创建唯一索引前，先全量扫描 `capture_tasks` 的 `(platform, shop_name)` 重复组并清理。
  - 主任务保留策略继续使用当前已落地的可信排序逻辑：
    - 优先保留当前配置匹配的任务；
    - 优先保留已有 `page_id` / 最近活跃 / 状态更完整的任务；
    - 删除其余重复项及关联样本。
- 为什么这样做：
  - 唯一索引建立前必须先消除现有冲突，否则迁移会失败。
  - 用户已明确“不用管历史数据”，因此可以直接删除重复项，无需做复杂样本迁移。

### 3. 写入口：把 `upsert_task()` 改为按业务主键幂等
- 文件：`backend/services/store.py`
- 变更内容：
  - 当 `payload` 未显式携带 `id` 时，先按 `(platform, shop_name)` 查询现有任务。
  - 若存在同店铺任务，则改为更新该任务，而不是直接插入。
  - 只在“该店铺不存在任何任务”时才新建。
- 为什么必须做：
  - 单靠唯一索引虽然能阻止脏写，但可能把用户正常保存变成 `IntegrityError`。
  - 正确做法是：数据库硬约束负责兜底，应用层按同一主键主动做幂等更新。

### 4. API 层：处理唯一约束冲突并返回可理解错误
- 文件：`backend/main.py`
- 变更内容：
  - 包装 `POST /api/tasks`、可能的重绑/恢复保存路径，捕获 SQLite 唯一约束异常。
  - 将底层数据库报错转成明确的业务提示，例如“该店铺任务已存在，已自动按当前店铺更新”或“同店铺任务唯一约束冲突”。
  - 理想状态下，大多数路径应在 `store.upsert_task()` 内部已转成更新，不应频繁冒到接口层。
- 为什么要做：
  - 防止未来异常路径把数据库原始报错直接暴露给前端。

### 5. 同步链路：保留现有去重，但降级为迁移/兜底层
- 文件：`backend/services/store.py`
- 变更内容：
  - `sync_tasks_with_shop_configs()` 继续保留当前去重逻辑，但其角色从“主要防线”改为“迁移清理 + 启动兜底”。
  - 返回值继续保留 `deleted_duplicates` / `deduped_shops`，便于观察启动时是否发现历史脏数据。
- 为什么这样设计：
  - 最终防线是唯一索引。
  - 同步链路去重仍有价值，可用于历史脏库自动修复与运维诊断。

### 6. 不选择 `edge_session_id` 作为唯一主约束
- 原因：
  - `edge_session_id` 是技术标识，来源于命名规则，理论上可变。
  - `platform + shop_name` 更贴合业务语义，也直接对应你在界面上看到的“店铺任务”。
  - 若未来需要，可将 `edge_session_id` 保持代码级一致性，但不作为本次唯一主键的唯一选择。

## Assumptions & Decisions
- 决策：永久修复方案选 `UNIQUE(platform, shop_name)`，不选单独 `UNIQUE(edge_session_id)`。
- 决策：不保留一店多任务的历史兼容能力。
- 决策：现有重复任务先清理再建索引，接受直接删除重复项。
- 决策：应用层 `upsert_task()` 必须同步改成按店铺幂等更新，避免唯一索引上线后正常保存报错。
- 假设：当前前端业务永远只需要每店铺一条“最新实时 GMV”任务，这与用户最新要求一致。
