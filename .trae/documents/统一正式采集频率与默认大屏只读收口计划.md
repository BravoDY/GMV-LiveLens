# 统一正式采集频率与默认大屏只读收口计划

## Summary
- 目标：把顶部“正式采集频率”下拉栏作为全部正式任务的统一调度频率入口，并让采集配置页在支持的平台上默认选中“大屏只读”，同时保持历史 OCR 任务兼容。
- 成功标准：
  - 顶部频率下拉变更后，OCR 与 `screen_readonly` 正式任务都按同一全局频率运行。
  - 采集配置页进入支持只读的平台任务时，默认显示“大屏只读”；不支持的平台仍默认 `OCR识别`。
  - 历史已真实使用过 OCR 的任务不被静默改成只读。
  - 前后端默认值一致，不再出现 `0.5 / 1 / 2` 多处默认值互相冲突。

## Current State Analysis
- `frontend/app.js`
  - `setupGlobalSettings()` 已把 `#globalInterval` 写入 `/api/settings`，前端入口已存在。
  - 当前读取逻辑依赖 `/api/settings` 返回值；若后端为空，最终展示受接口默认值影响。
- `backend/services/scheduler.py`
  - 调度器主循环已通过 `_global_interval_seconds()` 读取全局频率。
  - `screen_readonly` 分支也已改为使用 `global_interval`，说明“下拉栏直连正式调度器”主链路基本成立。
- `frontend/config.js`
  - 已有 `SCREEN_READONLY_SUPPORTED_PLATFORM_KEYS`、`defaultValueSourceForTask()`、`taskHasPersistedOcrSelection()`，说明“支持平台默认大屏只读、历史 OCR 兼容”逻辑已有实现基础。
  - `renderSetupWorkbench()` 与 `loadTaskIntoConfig()` 会对 `#valueSourceSelect` 赋默认值，但仍存在 UI 最终显示不稳定的风险点，需要做一轮完整收口。
  - 保存任务时 `interval_seconds` 取自顶部 `#globalInterval`，配置页保存与全局频率已部分打通。
- `frontend/index.html`
  - 顶部已存在“正式采集频率”下拉。
  - 采集配置区文案已描述“支持平台默认大屏只读”和“顶部频率作用于全部正式任务”。
  - 静态资源版本号已多次提升，说明此前存在浏览器缓存导致前端行为与代码不一致的问题。
- `backend/main.py`
  - `SettingsPayload.interval_seconds` 默认值已是 `1.0`。
  - 但 `GET /api/settings` 读取 `store.get_setting("interval_seconds", "0.5")`，与 payload 默认值不一致。
- `backend/services/shop_config.py`
  - `screen_readonly_supported()` 保留，平台支持名单齐全。
  - `ShopConfig.interval_seconds` 字段默认值仍为 `2`，CSV/JSON 模板解析默认也仍回退到 `2`，与“统一正式采集频率”目标不一致。
- `data/shops_default.json`
  - 默认模板里的 `interval_seconds` 仍为 `2.0`，会让新任务模板与顶部全局频率默认口径不一致。

## Assumptions & Decisions
- 决策：顶部频率下拉栏覆盖全部正式任务，包括 OCR 与 `screen_readonly`；测试态只读轮询不纳入本次范围。
- 决策：默认“大屏只读”只对已支持只读的平台生效，即 `天猫 / 京东 / 唯品会 / 抖音 / 得物`；其他平台保持 `OCR识别`。
- 决策：历史 OCR 任务以“已保存 OCR 选择或已有 OCR 运行痕迹”为保留条件，不做数据库批量迁移。
- 决策：统一默认正式采集频率口径为 `1.0s`，因为当前项目已形成“1 秒稳定优先”的结论；`0.5s` 继续仅作为用户可选项，不作为默认值。
- 假设：当前 MCP 浏览器中看到的“仍显示 OCR”大概率与静态资源缓存或赋值时序有关，而不是后端任务值错误；执行阶段需要以本地强刷和实际 DOM 行为复核。

## Proposed Changes

### 1. 收口正式采集频率的默认值来源
- `backend/main.py`
  - 把 `GET /api/settings` 的 `interval_seconds` 默认回退从 `0.5` 改为 `1.0`。
  - 原因：避免页面首次加载时把顶部下拉错误回填成 `0.5秒`。
- `backend/services/shop_config.py`
  - 把 `ShopConfig.interval_seconds` 默认值从 `2` 调整为 `1`。
  - 把 `load_shop_configs()` 中 `interval_seconds` 的回退值从 `2` 调整为 `1`。
  - 原因：新初始化任务、CSV 缺省值、运行时默认口径统一到 `1.0s`。
- `data/shops_default.json`
  - 将默认模板中的 `interval_seconds` 从 `2.0` 批量调整为 `1.0`。
  - 原因：保证新生成/重建任务模板与全局默认一致。
- `frontend/index.html`
  - 保留下拉可选项，但把初始选中项与统一默认值对齐到 `1秒`。
  - 原因：在 `/api/settings` 尚未返回前，也能先展示合理默认值，减少“闪成 0.5 秒”的感知。

### 2. 收口“顶部频率下拉栏 -> 正式调度器”的产品表达
- `frontend/app.js`
  - 保持 `setupGlobalSettings()` 的 POST 行为，但补充对空值/非法值的更稳妥处理，确保始终提交数值型频率。
  - 如有必要，在保存成功后触发轻量状态刷新，减少用户切换频率后误以为未生效。
- `frontend/config.js`
  - 保持 `saveTask()` 使用 `#globalInterval` 作为 `interval_seconds` 来源，确保任务保存时和调度器全局口径一致。
  - 检查是否还存在其它旧入口写死 `interval_seconds`；若存在，一并改成走顶部全局值或仅作为历史展示字段。
- `frontend/index.html`
  - 仅保留“正式采集频率”的统一语义，避免用户误解成只对 OCR 生效。

### 3. 收口配置页默认“大屏只读”的判定与显示时序
- `frontend/config.js`
  - 复核并加固 `defaultValueSourceForTask()` 的判定顺序：
    - 已保存 `screen_readonly` 的任务，始终返回 `screen_readonly`。
    - 已保存 `ocr` 且存在 OCR 历史痕迹的任务，保持 `ocr`。
    - 支持只读的平台新任务/未沉淀 OCR 痕迹任务，默认返回 `screen_readonly`。
    - 不支持的平台默认返回 `ocr`。
  - 复核 `renderSetupWorkbench()`、`loadTaskIntoConfig()`、`valueSourceSelect change` 三条链路，避免同一任务切换视图或异步刷新后又把下拉覆盖回 `ocr`。
  - 若存在 `task` 尚未同步完成就先渲染工作台的问题，执行阶段将补强“任务已切换后再绑定默认值”的时序控制。
  - 原因：当前逻辑虽已具备，但浏览器实测仍有“显示 OCR”的现象，需要确保 UI 与任务值最终一致。
- `frontend/index.html`
  - 保持“当前采集方式”说明文案，但同步静态资源版本号，确保浏览器强制拉取最新 `config.js` / `app.js`。

### 4. 保持历史任务兼容，不做破坏性迁移
- `backend/services/store.py`
  - 不改数据库结构，不批量重写已有 `value_source`。
  - 仅在必要时确认读取/序列化链路继续返回规范化后的 `screen_readonly` / `ocr`。
- 执行原则
  - 历史已稳定跑 OCR 的任务继续尊重原选择。
  - 只有配置页默认展示与新保存行为调整，不改旧数据本体。

## Execution Plan
1. 先统一默认值来源，修正后端设置接口、模板默认值和前端下拉初始值。
2. 再复核并收口配置页 `valueSourceSelect` 的默认判定与渲染时序。
3. 提升前端静态资源版本号，确保本地浏览器拿到最新脚本。
4. 做一次静态检查和针对最近改动文件的诊断检查。
5. 用本地服务做两轮验证：频率联动验证、默认采集方式验证。

## Verification Steps
- 接口验证
  - 打开首页后检查 `/api/settings` 返回的 `interval_seconds` 默认值为 `1.0`。
  - 切换顶部“正式采集频率”为 `2秒`，确认再次读取 `/api/settings` 已持久化为 `2.0`。
- 调度验证
  - 在已有 OCR 任务和只读任务同时存在时，观察两类任务的 `last_sample_at` 均随全局频率变化。
  - 至少验证一次从 `1秒 -> 2秒 -> 1秒` 的切换，确认只读任务也同步变慢/恢复。
- 配置页验证
  - 进入支持只读的平台任务（如天猫/京东/抖音/得物），确认 `当前采集方式` 默认显示“大屏只读”。
  - 进入不支持只读的平台任务，确认仍默认显示 `OCR识别`。
  - 找一个已有 OCR 历史痕迹的支持平台任务，确认默认仍显示 `OCR识别`，证明兼容逻辑生效。
- 缓存验证
  - 强制刷新页面后，确认首页引用的是新的静态资源版本号。
  - 若 MCP 浏览器仍显示旧值，再用 DOM 快照核对 `#valueSourceSelect.value` 与 `#globalInterval.value` 的真实值。
- 回归检查
  - 采集配置页保存任务仍可成功保存。
  - 任务管理与实时看板中的“数据源”标签仍正确显示 `大屏只读` / `OCR识别`。
