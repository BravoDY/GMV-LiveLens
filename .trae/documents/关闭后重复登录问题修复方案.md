## Summary
- 问题现象：
  - 用户选择某个店铺任务，登录后台后，关闭再打开仍反复要求输入账号密码。
  - 用户确认不是单一关闭方式触发，而是“项目关闭按钮 / 隐藏 / 手动点 X”多种方式都遇到过。
- 核心判断：
  - 按当前项目设计，店铺独立会话应使用固定 `user_data_dir` 持久保存登录态，本不应该频繁重新登录。
  - 当前现象说明“登录态持久化链路”实际不稳定。
- 用户目标：
  - 希望即使点击项目里的“关闭Edge”，后续再次打开也尽量保留登录态。

## Current State Analysis
- `backend/services/shop_config.py`
  - 每个店铺会生成固定 `edge_session_id`。
  - 每个店铺会固定映射到 `data/edge_profiles/<session_id>`。
  - 这说明架构上本来就是“一店一资料目录，一店一登录态”。
- `backend/services/store.py`
  - `edge_sessions` 表里会持久化：
    - `session_id`
    - `user_data_dir`
    - `session_mode`
  - 对独立店铺环境会强制使用固定目录。
  - 这进一步说明数据库层也支持长期复用登录态。
- `backend/collectors/remote_edge.py`
  - 启动独立店铺会话时，使用：
    - `--user-data-dir=<固定目录>`
  - 理论上 Cookie、LocalStorage、平台登录态都应落在这个目录。
  - 但关闭逻辑 `close_edge()` 会调用 `_close_edge()`。
- `backend/collectors/window_control.py`
  - 当前真正的关闭实现是：
    - `taskkill /PID ... /T /F`
  - 即“强制杀整个 Edge 进程树”。
  - 这类关闭方式存在高风险：
    - 浏览器来不及优雅退出
    - Cookie / Session / IndexedDB / 平台票据未稳定落盘
    - 平台后台会把这类异常中断视作不可信会话
- `README.md`
  - 文档明确写了：
    - `data/edge_profiles/<session_id>/` 是该账号的持久化登录态
    - 不要随意删除
  - 所以当前现象不是产品预期，而是实现与预期不完全一致。

## Root Cause Analysis
### 根因 1：关闭机制过于粗暴
- 当前项目“关闭Edge”本质是强杀进程树。
- 这不是正常浏览器关闭路径。
- 对需要严格会话落盘的平台后台来说，强杀很可能造成：
  - 登录态未落盘
  - 会话票据损坏
  - 下次打开触发重新登录

### 根因 2：系统缺少“优雅关闭 -> 超时后兜底强杀”的分层策略
- 当前没有看到：
  - 先尝试通过 CDP / Page / Browser 优雅关闭
  - 等待资料目录稳定写入
  - 失败后才执行强杀
- 现在是直接走高破坏性关闭。

### 根因 3：缺少 profile 持久化诊断信息
- 当前虽然有 `profile_initialized`，但没有更细的诊断：
  - 本次会话使用的是哪个实际目录
  - 关闭前目录是否已有 Cookie / Network / Local Storage 相关文件
  - 关闭后目录是否仍存在并持续更新
- 所以出了问题时，用户只能体感“为什么又掉登录了”，系统不能解释。

### 根因 4：不同关闭路径体验不一致
- 用户明确说“都有”，说明不只是项目按钮。
- 除了强杀问题，还可能存在：
  - 手动点 X 后项目没有及时正确重连到原资料目录
  - 某些平台登录策略对远程调试 + 无扩展环境更敏感
  - 页面登录发生在错误会话里，导致你以为登录的是这个店，其实登录在别的 Edge 会话
- 这类情况需要通过更明确的会话展示和诊断信息来收束。

## Requirement Understanding
- 用户真正要的不是一句“理论上应该保存登录态”。
- 用户要的是：
  - 为什么现在没有保存住
  - 项目层面怎么修，才能让关闭再打开尽量不掉登录
  - 如果还有平台侧不可控因素，系统应该怎么提示和规避

## Best Implementation
### 总体策略
- 将“关闭Edge”改造成两段式：
  1. 优雅关闭
  2. 失败或超时后再兜底强杀
- 在关闭前后增加最小必要的 profile 诊断。
- 在前端明确提示：
  - “关闭Edge”属于高影响动作，但系统会尽量保留登录态
  - “隐藏Edge”仍是最稳妥的长期保活方式
- 考虑到平台风控风险，本次不能直接全量替换旧关闭逻辑，必须采用“先诊断、再灰度、后切换”的分阶段落地方案。

### 方案原则
- 不改变“每店铺固定 profile 目录”的主架构。
- 不引入新的浏览器引擎或复杂持久化方案。
- 重点修正关闭机制和可观测性。
- 所有可能影响平台登录态或触发平台风控的动作，都必须满足：
  - 可单店铺验证
  - 可观察关闭方式
  - 可确认是否走了强杀
  - 可快速回退到旧逻辑

## Safe Rollout Strategy
### Phase 0：只加诊断，不改关闭行为
- 目标：
  - 先确认当前每个店铺会话到底是不是在稳定复用同一个 `user_data_dir`
  - 先确认“重新登录”主要发生在哪种关闭结果后
- 要做的事：
  - 给 health / close 返回结果加 profile 诊断字段
  - 给 close 返回结果加“本次是否强杀”的显式标记
- 原因：
  - 这是最低风险阶段，不改变任何浏览器行为，不主动触发新的平台风控路径

### Phase 1：增加“优雅关闭尝试”，但默认仍保守兜底
- 目标：
  - 只在单店铺会话上验证：优雅关闭后再次打开是否更容易保住登录态
- 要做的事：
  - 在 `close_edge` 中增加“优雅关闭尝试”
  - 保留超时后强杀兜底
  - 返回结果中明确记录：
    - 是否尝试优雅关闭
    - 是否最终被迫强杀
- 执行方式：
  - 先对单个店铺手工验证，不做平台批量全量切换
- 原因：
  - 优雅关闭本身比强杀更接近真实用户行为，理论上更安全，但仍需确认不会影响会话重连和窗口控制

### Phase 2：前端交互再切换默认认知
- 目标：
  - 在验证过优雅关闭确实更稳后，再修改前端文案与默认使用建议
- 要做的事：
  - 将“关闭Edge”文案升级为“尽量保留登录态的关闭”
  - 将返回结果中的关闭方式展示给用户
- 原因：
  - 先验证真实效果，再改变产品承诺，避免先说大话后落空

### Phase 3：文档与最终默认策略收口
- 目标：
  - 让 README、产品说明、实际行为三者一致
- 要做的事：
  - 若验证成功，更新文档，把“优雅关闭优先”写成正式策略
  - 若验证发现某些平台仍容易掉登录，则保留“隐藏优先，关闭谨慎”的平台级说明

## No-Risk Boundaries
- 本轮计划中，以下动作不允许直接一步到位全量替换：
  - 不直接删除当前强杀路径
  - 不直接把所有平台的“关闭Edge”默认改为全新逻辑且无回退
  - 不在缺少关闭结果诊断的情况下就宣称“关闭后一定保留登录态”
- 本轮允许做的低风险动作：
  - 增加诊断字段
  - 增加优雅关闭尝试
  - 保留强杀兜底
  - 先单店铺验证

## Proposed Changes
### 1. 第一步先补关闭诊断与 profile 诊断
- 文件：`backend/collectors/remote_edge.py`、`backend/main.py`
- 变更：
  - 优先补齐返回字段，而不是先改关闭行为：
    - `close_mode`
    - `graceful_attempted`
    - `force_kill_used`
    - `profile_initialized`
    - `profile_path`
    - 关键目录状态摘要
- 目的：
  - 先把问题看清楚，保证执行落地可验证

### 2. 将关闭机制改为“优雅关闭优先，强杀兜底”
- 文件：`backend/collectors/remote_edge.py`
- 变更：
  - 在 `_close_edge()` 中先尝试优雅关闭当前受控浏览器：
    - 先枚举当前上下文 / 页面
    - 尝试关闭页面或浏览器连接
    - 等待短时间，让 Edge 自行退出
  - 若调试端口仍存在，再进入现有 `kill_edge_process_tree()` 兜底路径
- 目的：
  - 给浏览器正常写盘和平台会话收尾留出机会

### 3. 为关闭结果增加“关闭方式”与“是否走强杀兜底”诊断字段
- 文件：`backend/collectors/remote_edge.py`、`backend/main.py`
- 变更：
  - 在 `RemoteEdgeCloseState` 中增加类似字段：
    - `close_mode`
    - `graceful_attempted`
    - `force_kill_used`
  - API 返回时一起透出
- 目的：
  - 后续用户能知道这次关闭到底是“正常退出”还是“强杀退出”

### 4. 增加 profile 持久化诊断
- 文件：`backend/collectors/remote_edge.py`
- 变更：
  - 新增轻量诊断方法，用于检查：
    - 当前 `user_data_dir` 是否存在
    - 目录是否有实际内容
    - 关键文件最近修改时间
  - 将诊断结果挂到 health 或 close 返回结果中
- 目的：
  - 判断“是不是根本没写进去”
  - 区分“会话没保存”与“平台主动让你重新登录”

### 5. 前端提示语升级
- 文件：`frontend/app.js`、`frontend/edge.js`
- 变更：
  - 第一阶段先增加“关闭结果展示”，不先改默认承诺：
    - 本次关闭是否优雅完成
    - 是否最终使用强杀
  - 第二阶段再增加更准确的关闭说明：
    - 系统将先尝试优雅关闭，尽量保留登录态
    - 若系统被迫强杀，会给出提示
  - 对“隐藏Edge”补充说明：
    - 隐藏仍是最稳妥的长期保活方式
- 目的：
  - 先让用户看到真实关闭结果，再逐步切换产品认知

### 6. 会话信息展示更明确
- 文件：`frontend/edge.js`、`frontend/config.js`
- 变更：
  - 在会话状态里清楚展示：
    - 当前店铺会话名称
    - 当前资料目录路径
    - 是否已初始化 profile
  - 若 profile 目录为空或未初始化，给出明确提醒
- 目的：
  - 避免用户在错误会话里登录后，以为登录态会复用到当前任务

### 7. 文档与使用策略同步更新
- 文件：`README.md` 或项目文档区
- 变更：
  - 待单店铺验证通过后，再补充“登录态保留策略”说明：
    - 推荐长期使用 `隐藏Edge`
    - `关闭Edge`已优化为优雅关闭优先
    - 若平台仍频繁要求二次登录，需结合平台风控判断
- 目的：
  - 避免在验证完成前过早更新对外承诺

## Assumptions & Decisions
- 已确认决策：
  - 用户要求“关闭后再打开也尽量保留登录态”
  - 不是只接受“隐藏即可”
  - 用户要求这一步执行计划必须尽量规避平台风控，不能冒进
- 本次方案选择：
  - 保留当前固定 profile 目录架构
  - 先补诊断，再灰度改关闭机制，最后再调整默认产品承诺
  - 整体采用“可观测、可回退、单店铺先验证”的落地策略

## Risk Assessment
- 可能影响的现有功能：
  - 关闭 Edge 的耗时会略有上升，因为会先尝试优雅关闭
  - 某些异常卡死情况下，最终仍会回退到强杀
  - 若未做灰度直接切换，可能引入“窗口没关净 / 调试端口残留 / 会话状态不一致”新问题
- 兼容性风险：
  - 需确保优雅关闭不会误伤其它店铺会话
  - 需避免浏览器连接对象关闭后造成调试状态残留异常
- 不可完全承诺的边界：
  - 即便项目侧实现优雅关闭，某些平台仍可能因自身安全策略要求重新登录
  - 但本次至少能把“项目自己导致登录态不稳”的问题显著降低，并让问题可诊断

## Execution Order
1. 先做关闭结果诊断与 profile 诊断，不改关闭逻辑默认行为。
2. 单店铺验证并观察：
   - 使用的 profile 是否稳定
   - 当前关闭是否总是强杀
3. 再引入“优雅关闭尝试 + 强杀兜底”。
4. 只对单店铺手工验证，不做平台批量切换。
5. 验证通过后，再修改前端默认文案和 README。

## Verification Steps
- 先决要求：
  - 所有验证先在单店铺执行，不同时覆盖多个平台店铺
  - 不做批量并发关闭验证
- 场景 1：独立店铺会话首次登录后，点击项目“关闭Edge”再重新打开
  - 期望：大多数情况下保持登录态，不再频繁要求重新登录
- 场景 2：独立店铺会话首次登录后，点击“隐藏Edge”再显示
  - 期望：登录态保持不变
- 场景 3：手动点 X 关闭后再由项目重新打开
  - 期望：若 profile 正常落盘，应尽量保持登录态
- 场景 4：关闭结果诊断
  - 期望：能看到本次是否走了优雅关闭、是否最终使用强杀、profile 是否已初始化
- 场景 5：回退验证
  - 若优雅关闭导致异常残留，应能快速退回“旧关闭逻辑 + 诊断保留”的状态
