# ddddocr异常OCR深因排查与修复计划

## Summary
- 目标：找出“项目里已经加入 ddddocr 学习机制，但仍会出现特别异常 OCR”的深层原因，并形成一套低风险、可验证的修复路线。
- 结论预判：当前项目中的“学习机制”并不是在线训练或运行时自适应，主要只是样本沉淀、人工纠错和离线异常统计；因此它并不会直接提升下一次识别质量。
- 成功标准：
  - 能明确解释为什么“学过了”仍会继续出现异常 OCR。
  - 能区分问题来自模型、预处理、候选提取、评分还是调度确认。
  - 能给出不破坏现有主链路的分阶段修复方案与验证方式。

## Current State Analysis
- `ddddocr` 只是多个可选 OCR 引擎之一，接入点在 `backend/collectors/ocr_reader.py`。
- 当前所谓“学习机制”实际包含三部分，但都没有直接改变 ddddocr 的运行时模型：
  - 样本沉淀：`/api/test-ocr` 与调度器都会把裁剪图保存到 `data/ocr_datasets`。
  - 人工纠错：`/api/tasks/{task_id}/manual-correction` 会把人工值写回 `last_trusted_value`，并追加历史样本。
  - 离线异常监控：`backend/tools/find_ocr_anomalies.py` 只统计高频异常字符，提示人工去扩充纠错规则。
- 核心事实 1：项目里没有发现 ddddocr 训练、增量学习、微调、权重更新、热加载模型等实现。
- 核心事实 2：`manual_correction` 只更新任务运行态和历史样本，不会反向更新 OCR 模型、引擎优先级、字符映射或候选评分逻辑。
- 核心事实 3：`find_ocr_anomalies.py` 是离线分析工具，只输出报告，不会自动把新异常字符并入运行时规则。
- 核心事实 4：`_run_ddddocr()` 只有 `classification()` 文本输出，没有检测框，也没有真实置信度；当前代码给它固定分数 `0.5`。
- 核心事实 5：默认 `auto` 引擎顺序是 `rapidocr -> legacy_rapidocr -> paddleocr -> ddddocr -> tesseract`，因此很多“异常 OCR”不一定是 ddddocr 产出的，也可能在 ddddocr 还没轮到之前，候选就已被前序引擎提前命中。
- 核心事实 6：运行时真正决定值的是“预处理变体 + OCR 原文 + 纠错替换 + 正则抽取 + 候选评分 + 调度连续确认”，而不是单一模型。

## Deep Root Causes
### 1. “学习”没有闭环到运行时
- 文件：`backend/main.py`、`backend/services/scheduler.py`、`backend/tools/find_ocr_anomalies.py`
- 深因：
  - 样本被保存了，但没有自动标注闭环。
  - 人工纠错被保存了，但没有自动生成新规则或新模型。
  - 异常字符被统计了，但没有自动升级到 `ocr_reader.py` 的运行时特征库。
- 结果：
  - 系统知道“曾经识错过”，但下一次识别时仍按旧模型、旧规则工作。

### 2. ddddocr 本身在本项目里是“弱观测引擎”
- 文件：`backend/collectors/ocr_reader.py`
- 深因：
  - `_run_ddddocr()` 只返回整段文本，没有 box，没有多候选，没有真实 score。
  - 当前统一给 `ddddocr` 写死 `0.5`，这对异常排查和引擎对比几乎没有诊断价值。
- 结果：
  - 一旦 ddddocr 识别出异常文本，系统只能在后续正则和评分阶段“猜”，无法精细判断其可信度。

### 3. 预处理是静态模板，不是按店铺/颜色/异常模式自适应
- 文件：`backend/collectors/ocr_reader.py`
- 深因：
  - 当前只固定尝试 `color / contrast_gray / yellow_digits / otsu` 四类变体。
  - 这些预处理没有根据平台、店铺、背景色、字体风格、最新失败样本动态调整。
- 结果：
  - 某些特殊场景会长期卡在“同一种坏切图 + 同一种坏预处理 + 同一种异常 OCR”循环里。

### 4. 候选提取与纠错规则仍偏“字符级补丁”，不是“模式级约束”
- 文件：`backend/collectors/ocr_reader.py`
- 深因：
  - 当前主要靠 `OCR_CHAR_REPLACEMENTS` 和金额正则做补救。
  - 这对单字符形近字有效，但对粘连、断裂、错位、分组裂变、局部缺字的异常文本能力有限。
  - `_normalize_amount_values()` 虽然做了部分重组，但仍可能把错误分段重建成“看起来合法但实际很离谱”的金额。
- 结果：
  - 学习机制即便发现了新的异常字符，也只是继续堆字符替换，不能根治“结构性误识别”。

### 5. 调度确认能拦截部分异常，但也可能放大“稳定错误”
- 文件：`backend/services/scheduler.py`
- 深因：
  - `_judge()` 的连续确认逻辑主要防跳变、防下降、防跨天误判。
  - 但如果 OCR 连续多次稳定地识别出同一个错误值，系统仍可能在达到确认次数后接受它。
- 结果：
  - 错误一旦“稳定”，学习机制又没有反哺运行时，最终仍可能被可信化。

### 6. 异常归因可能被“引擎顺序”误导
- 文件：`backend/collectors/ocr_reader.py`
- 深因：
  - `read_text()` 是按引擎顺序和预处理变体逐个尝试；一旦 `extract_candidates()` 命中，就提前返回。
  - 所以用户看到“最后异常值”，不代表就是 ddddocr 造成的；很可能是前序引擎文本经过纠错和正则后被选中了。
- 结果：
  - 如果不先补齐“每次建议值来自哪个引擎、哪个预处理”的诊断信息，单独强化 ddddocr 学习机制也可能治错方向。

## Proposed Changes
### 1. 先补齐异常归因链路，再动学习机制
- 文件：`backend/collectors/ocr_reader.py`、`backend/main.py`、`backend/services/store.py`
- 计划：
  - 给每个候选值保留来源信息：`engine / variant / raw_fragment / normalized_value / reason`。
  - 将最终 `suggested_value` 的来源一起写入样本记录，便于区分究竟是谁产生了异常值。
- 原因：
  - 在没搞清“异常值来自哪个引擎和哪个预处理”前，强化 ddddocr 学习机制风险很高。

### 2. 把“学习机制”从记录型升级为规则反哺型
- 文件：`backend/tools/find_ocr_anomalies.py`、`backend/collectors/ocr_reader.py`
- 计划：
  - 不改变模型训练前提下，先做最稳的闭环：把人工确认过的高频异常模式转成可审查的规则候选。
  - 规则候选包括：
    - 新增字符替换建议
    - 新增前缀货币别名建议
    - 新增按店铺特定黑名单/白名单模式建议
- 原因：
  - 这是不动外部模型的最低风险升级，且能真正影响下一次运行时识别。

### 3. 给 ddddocr 增加“只在合适场景出手”的约束
- 文件：`backend/collectors/ocr_reader.py`
- 计划：
  - 不再把 ddddocr 仅当作通用后备引擎。
  - 增加场景判定，例如只在某些特征明显偏验证码/艺术字/扭曲数字时提高其优先级，或在特定店铺实验开启。
- 原因：
  - ddddocr 擅长的并不等于当前所有 GMV 场景都适合。
  - 若异常主要来自前序引擎，则不应错误地把问题都归到 ddddocr 身上。

### 4. 把预处理从固定模板升级为“可按店铺实验”
- 文件：`backend/collectors/ocr_reader.py`、可能新增配置读写点在 `backend/services/store.py`
- 计划：
  - 为失败率高的店铺引入可切换预处理策略集，而不是所有店铺共用一套。
  - 保持默认不变，只对明确问题店铺开启额外预处理。
- 原因：
  - 这样能降低对已跑通店铺的风险，同时验证“异常是否来自预处理不匹配”。

### 5. 给调度层增加“稳定错误防接受”信号
- 文件：`backend/services/scheduler.py`
- 计划：
  - 在 `_judge()` 前增加额外约束：
    - 若候选来源持续来自低置信组合，哪怕值稳定，也不直接升为可信。
    - 若候选文本命中过多纠错替换或异常模式，抬高确认门槛。
- 原因：
  - 防止“同一个错误值连续出现三次，就被系统认可”为真值。

### 6. 用真实样本基准而不是主观感觉评估学习效果
- 文件：`backend/tools/benchmark_ocr_engines.py`、`data/ocr_datasets`、`data/ocr_samples`
- 计划：
  - 基于真实异常样本建立一个“异常 OCR 回归集”。
  - 评估维度至少包括：
    - 原文是否被正确抽取
    - 候选排序是否正确
    - 最终建议值是否正确
    - 连续确认后是否仍会误接收
- 原因：
  - 只有这样才能判断“学习机制是否真的提升了系统结果”，而不是只看单次 OCR 文本。

## Assumptions & Decisions
- 决策：本次优先把问题定义为“系统级 OCR 决策链异常”，而不是先假设 ddddocr 模型本身有问题。
- 决策：第一阶段不直接引入 ddddocr 训练/微调，先做运行时可闭环的低风险改造。
- 决策：学习机制要以“能影响下一次运行时识别”为标准，否则只能算日志和样本积累。
- 决策：修复顺序按“归因 -> 规则反哺 -> 预处理分层 -> 调度护栏 -> 回归基准”推进。
- 假设：用户更关心“减少特别异常 OCR 落到最终 GMV”而不是单纯提升某个引擎的字符识别率。

## Verification Steps
### A. 归因验证
- 对异常样本执行 `/api/test-ocr`，确认可以看到最终建议值对应的 `engine / variant / raw_fragment / candidate reason`。
- 确认能区分异常是来自 ddddocr，还是来自 rapidocr / paddleocr 的早期命中。

### B. 学习闭环验证
- 选取一批人工纠错过的高频异常样本。
- 验证这些样本生成的规则候选，能在不改模型的情况下提升下一次识别结果。

### C. 调度保护验证
- 构造“错误但稳定”的 OCR 样本序列。
- 验证系统不会仅因为连续出现就错误接受。

### D. 店铺差异验证
- 选取一到两个异常店铺做店铺级预处理实验。
- 验证只改善目标店铺，不破坏其他已跑通店铺。

### E. 回归验证
- 用历史异常样本集重跑 benchmark。
- 确认改造后“异常最终值”显著下降，且主链路吞吐和已有正常识别不明显退化。
