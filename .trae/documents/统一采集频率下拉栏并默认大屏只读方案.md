# 统一采集频率下拉栏并默认大屏只读方案

## Summary
- 目标 1：让顶部“采集频率”下拉栏直接控制全部正式采集任务，而不是只影响 OCR / 全局设置的一部分逻辑。
- 目标 2：把“采集配置”中的默认采集方式改为“大屏只读”，但仅对当前已支持大屏只读的平台生效；不支持的平台继续默认 OCR。
- 关键收口：前面已实现的“部分平台正式链路固定 1 秒”需要改为“由顶部频率下拉栏统一驱动”，否则用户改下拉栏时，大屏只读任务不会跟着变化。

## Current State Analysis
- 顶部“采集频率”控件位于 `frontend/index.html`，实际元素是 `#globalInterval`。
  - 当前可选值有 `0.5 / 1 / 2 / 5 / 10 / 30 / 60` 秒。
  - `frontend/app.js` 中 `setupGlobalSettings()` 会读取 `/api/settings` 并在变更时 POST `/api/settings`。
- 后端正式采集调度位于 `backend/services/scheduler.py`。
  - 非 `screen_readonly` 任务使用全局设置 `interval_seconds`。
  - `screen_readonly` 任务当前优先读取 `shop_config.screen_readonly_fixed_interval_seconds(task.platform)`。
- 平台固定只读频率位于 `backend/services/shop_config.py`。
  - 当前已支持平台都被固定成 `1.0` 秒。
  - 这与用户新需求“顶部下拉栏覆盖全部任务”冲突。
- 采集配置中的采集方式选择位于 `frontend/index.html` 的 `#valueSourceSelect`，选项包含 `ocr` 和 `screen_readonly`。
  - 当前前端逻辑在 `frontend/config.js` 中通过 `selectedValueSource()` 和 `renderSetupWorkbench()` 处理。
  - 当前默认值来源仍然偏向 `task.value_source || "ocr"`，并未对“支持只读的平台默认选大屏只读”做显式收口。
- 平台只读支持判断已存在于 `frontend/config.js` 与 `backend/services/shop_config.py`。
  - 当前已支持只读的平台包括：天猫、京东、唯品会、抖音、得物。

## Proposed Changes
### 1. 让顶部频率下拉栏统一控制全部正式采集
- 修改 `backend/services/shop_config.py`
  - 移除或停用 `SCREEN_READONLY_FIXED_INTERVAL_SECONDS` 对正式调度的强固定作用。
  - 保留“平台是否支持大屏只读”的判断，不再在这里决定正式调度频率。
- 修改 `backend/services/scheduler.py`
  - 保留现有全局 `interval_seconds` 缓存优化。
  - 调整 `screen_readonly` 任务的下次执行时间计算逻辑，使其与 OCR 一样读取全局 `interval_seconds`。
  - 若仍需保留只读测试接口中的 `screen.interval_seconds` 展示字段，只保留为观测信息，不参与正式调度决策。
- 修改 `frontend/config.js`
  - 更新各平台大屏只读提示文案，把“正式链路固定按 1 秒轮询”改为“正式链路跟随顶部采集频率设置”。
- 修改 `frontend/index.html`
  - 如有必要，调整“采集频率”标签或说明文字，让用户明确该下拉栏控制的是“全部正式采集频率”。

### 2. 把采集配置默认方式改为“大屏只读”
- 修改 `frontend/config.js`
  - 调整 `selectedValueSource()` / 当前任务初始化逻辑：
    - 若当前任务已有明确 `value_source`，优先尊重已保存值，避免误改历史任务。
    - 若当前任务还未明确配置或首次进入配置页，则：
      - 已支持只读的平台默认选 `screen_readonly`
      - 不支持只读的平台默认选 `ocr`
  - 同步更新 `valueSourceHint`、步骤提示、保存校验逻辑，保证默认切到大屏只读时，界面文案与流程提示一致。
- 修改 `frontend/index.html`
  - 如有必要，调整 `#valueSourceSelect` 的默认选项顺序或默认选中表现，使“支持只读的平台默认大屏只读”在 UI 上更直观。

### 3. 保证现有任务兼容
- 修改 `frontend/config.js`
  - 仅改变“新进入配置流程时”的默认展示，不强制改写已保存任务的 `value_source`。
  - 避免用户打开历史 OCR 任务后，被前端默认值误切成大屏只读。
- 修改 `backend/services/scheduler.py`
  - 不批量改数据库里已有任务的 `value_source` 或 `interval_seconds` 字段。
  - 统一只从全局设置读取正式采样频率，减少历史字段差异对正式调度的影响。

## Assumptions & Decisions
- 已确认顶部“采集频率”下拉栏后续要覆盖全部正式任务，包括 OCR 和大屏只读。
- 已确认“默认大屏只读”只针对已支持的大屏只读平台生效。
- 已确认不需要新增第二个只读专用频率下拉栏，避免 UI 复杂化。
- 决策：历史任务若已保存为 OCR，则继续尊重历史值，不做静默迁移。
- 决策：大屏只读页面返回的 `interval_seconds` 继续作为页面观测数据保留，但不再作为正式调度主依据。
- 决策：本次只收口“正式采集调度”和“配置页默认行为”，不改前端单页只读测试轮询，也不改看板断线兜底轮询。

## Verification Steps
- 静态验证
  - 检查 `frontend/index.html` 中的 `#globalInterval` 说明是否与“控制全部正式采集”一致。
  - 检查 `backend/services/scheduler.py` 是否不再对 `screen_readonly` 平台走固定 `1s`。
  - 检查 `frontend/config.js` 是否对支持只读的平台默认返回 `screen_readonly`，对不支持平台默认返回 `ocr`。
  - 检查已存在任务若有 `value_source`，前端是否仍优先显示已保存值。
- 运行验证
  - 在顶部将采集频率切换为 `2 秒`，观察至少 1 个 OCR 任务和 1 个大屏只读任务，确认两类任务都随之变慢。
  - 再切回 `1 秒`，确认两类任务都恢复为 1 秒级刷新。
  - 进入采集配置页：
    - 天猫/京东/唯品会/抖音/得物等支持只读的平台，默认采集方式应显示为“大屏只读”
    - 若存在不支持只读的平台，默认应显示为 `OCR识别`
  - 打开一个历史 OCR 任务，确认页面不会被默认值静默改成大屏只读。
- 回归验证
  - 核对任务保存流程、大屏只读保存流程、OCR 框选保存流程均不被破坏。
  - 核对顶部下拉栏修改后，`/api/settings` 返回值与实际正式调度表现一致。
