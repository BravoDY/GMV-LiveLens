## Summary
- 目标：在`采集配置`里，以“当前任务”为中心展示其绑定的`Edge 会话`下所有可见页签，由用户手动选择其中一个页签作为该任务长期使用的 OCR 页面。
- 成功标准：
  - 当前任务进入采集配置后，能看到所属会话的多个页签，而不是只依赖旧`page_id`。
  - 用户可以手动选择任意一个当前页签，并立即绑定到该任务。
  - 绑定后，预览、OCR 测试、正式采集都继续使用该任务绑定页。
  - 若绑定页签失效，不自动偷偷恢复，不自动新开页；界面必须明确提示“发生了什么、你该怎么操作”，并允许用户重新扫描后再次手动选择。
  - `采集配置`整体流程从“初始化 -> 进入当前店铺 -> 选择页签 -> 生成预览 -> 框选 -> 测试 -> 保存 -> 下一家 -> 失效重绑”必须形成统一链路，不再让用户自行猜测当前处于哪一步。
  - 交付时除代码方案外，还需要生成一份独立文档`采集配置配置图`，采用“双版本”结构：流程图/思维导图 + 页面产品结构图。
- 明确不做：
  - 不处理“启动 Edge 后自动关闭非目标页签”。
  - 不改变你当前“我自己手动关其它页签，只保留需要页签”的使用习惯。
  - 不引入大幅 UI 改版，只做采集配置区域的最小必要调整。

## Workflow Goals
- 本轮不再把`采集配置`看成单个绑定动作，而是看成完整的“配置向导”。
- 必须解决的核心问题：
  - 当前任务是谁。
  - 当前应该做哪一步。
  - 下一步按钮是什么。
  - 如果失败，是哪一环失败。
  - 用户回到哪里继续操作。
- 最终目标是让你之后即使隔很久再回来看，也能通过“采集配置配置图”与页面实际布局，立即知道整条链路怎么走。

## Requirement Understanding
- 需求本质：
  - 天猫/京东后台操作过程中，可能因为点击导致当前会话出现多个页签。
  - 你希望在`采集配置`里能直接看到“当前任务所属浏览器会话”的多个页签，并由你手动指定“哪一个页签用于 OCR”。
  - 这个选择应直接写回任务，后续预览与正式采集默认都使用它。
  - 如果后面这个页签失效，不做自动恢复，而是清晰告诉你问题原因，并给你二次选择能力。
  - 你还希望`采集配置`所有流程操作都有非常清晰的链路，不只是功能能用，而是页面本身就像导航面板。
  - 你需要额外产出一份`采集配置配置图`文档，作为后续一目了然的操作说明和产品结构图。

## Best Implementation
### 总体设计
- 采用“单任务、单会话、手工绑定”的最小改造方案。
- 保留现有工作台和任务流转，不重做整体界面。
- 弱化`/api/shops/match`在采集配置里的主角色，改为以“当前任务页签列表 + 明确手选”作为主交互。
- 把现有三块区域重新定义为一条连续链路：
  - 左侧顶部：当前任务状态与下一步导航。
  - 左侧中部：当前任务页签选择与失效恢复。
  - 左侧底部：保存与切换下一家。
  - 右侧：预览、框选、测试结果。

### 目标用户链路
1. 进入`采集配置`
   - 系统自动聚焦当前待配置任务。
   - 明确显示当前店铺、当前会话、当前绑定状态、下一步动作。
2. 初始化阶段
   - 若尚未初始化默认店铺，只允许执行“初始化任务”。
   - 初始化完成后自动进入第一家待配置任务。
3. 页面选择阶段
   - 用户在任务管理中确保正确 Edge 页已打开。
   - 回到采集配置，点击“扫描当前会话页签”。
   - 系统列出该任务所属会话全部可见页签，并给出辅助排序/匹配提示。
   - 用户手动选择一个页签并绑定任务。
4. 预览与框选阶段
   - 绑定成功后自动生成预览，失败则给出明确提示。
   - 用户在右侧截图中拖拽框选 GMV 区域。
5. 测试与保存阶段
   - 用户先点“测试识别”。
   - 确认结果后点击“保存并进入下一家”。
   - 系统保存后自动切到下一家待配置任务。
6. 失效恢复阶段
   - 若绑定页签失效，界面进入“需重选页面”状态。
   - 系统只提示，不自动恢复，不自动新开页。
   - 用户去任务管理确认后台页，再回来重新扫描并重新绑定。

## Assumptions & Decisions
- 已确认决策：
  - 启动后出现多少页签不再作为本轮处理重点。
  - 其它页签由你手动关闭，只保留需要页面。
  - 采集配置必须能看到当前任务所属浏览器会话的多个页签。
  - 手动选择后，页签直接绑定到任务长期生效。
  - 页签失效时不自动恢复，必须给出明确提示和二次选择能力。
  - `采集配置配置图`采用双版本：流程图/思维导图 + 产品结构图。
- 本次实现选择：
  - 保留现有页面结构，不做大改版。
  - 新增“任务级页签候选接口”，而不是继续复用批量接口硬凑。
  - 正式采集改为严格模式，不自动新开页。
  - 通过“状态机 + 配置图文档”把当前产品链路长期固化下来。

## Risk Assessment
- 可能影响的现有功能：
  - `remote_edge.ensure_page()`相关调用语义会更严格，需避免误伤其它启动/开页流程。
  - 采集任务在页签关闭后，将更快暴露为`remote_page_not_found`，不再被自动补页掩盖。
  - 采集配置文案和按钮可见性会更严格依赖状态，需回归验证“进入下一家”“重扫”“保存”等按钮出现时机。
- 兼容性风险：
  - 如果其它地方仍依赖`/api/shops/bind`做批量绑定，需要保持原接口不删不改，仅在采集配置中切换到新接口。
  - 若前端错误处理依赖字符串判断，需要同步改成优先读结构化`reason_code`。
- 为什么风险可控：
  - 本次改动集中在“采集配置绑定链路”和“严格失效处理”，不触碰任务管理整体布局。

3. 在`backend/services/scheduler.py`切换到严格查找策略，页签失效时直接记录失败。
4. 在`frontend/config.js`梳理采集配置状态机与工作台提示语，统一“当前步骤/下一步/异常恢复”。
5. 在`frontend/config.js`把扫描/绑定交互改成“当前任务页签列表 -> 单选 -> 直接绑定任务”。
6. 在`frontend/edge.js`完善预览失败与重选提示，保证只提示不自动恢复。
7. 在`frontend/index.html`做最小文案和结构补强，必要时补少量样式。
8. 输出`.trae/documents/采集配置配置图.md`，沉淀流程图与产品结构图。
9. 联调“初始化 -> 进入当前任务 -> 扫描 -> 绑定 -> 预览 -> 框选 -> 测试 -> 保存 -> 下一家 -> 页签失效重绑”完整链路。

## Verification Steps
- 接口验证
  - 打开某个任务对应的 Edge 会话，确保有多个页签。
  - 调用任务级页签候选接口，确认能返回该会话全部页签及当前任务绑定信息。
- 前端验证
  - 进入采集配置，当前任务能看到多个页签。
  - 手动选择任意页签后，点击绑定，任务的`page_id/page_url/page_title/edge_session_id`被更新。
  - 绑定后立即能生成预览。
  - 当前步骤、下一步提示、按钮显隐与实际流程一致。
- 采集验证
  - 启动调度后，正式采集继续使用刚绑定的页签。
  - 同一页签内正常跳转 URL 时，只要页对象未失效，采集仍可继续。
- 失效验证
  - 手动关闭已绑定页签或让其彻底失效。
  - 预览时应收到明确提示，不自动新建页签。
  - 调度采集应记录`remote_page_not_found`，并要求人工重新扫描绑定。
- 回归验证
  - 任务管理页面现有 UI 不被大改。
  - 平台级/任务级启动、显示、隐藏、关闭接口行为不受本轮方案影响。
  - `采集配置配置图`文档与真实页面流程一致，不出现“文档说法”和页面行为不一致的情况。
