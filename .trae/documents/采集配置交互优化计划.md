## Summary

- 目标：优化 `采集配置` 页中的全链路 UI 交互，降低“快速初始化”后的人工切换成本，让用户在现有单页结构内完成更连贯的配置流程。
- 用户偏好已锁定：
  - 优先做“全链路简化”
  - 保留“单页”形态，不改成重型向导
  - 自动化程度为“中等自动化”：系统负责推进上下文和默认选择，框选与保存仍由人工确认
- 成功标准：
  - 用户完成 `初始化 -> 会话 -> 绑定 -> 预览 -> 框选 -> 保存` 时，不需要自己反复判断下一步在哪
  - 初始化后可以直接进入“下一家待配置店铺”的上下文
  - 页面能明确显示哪些店铺“待绑定”、哪些“待标定”、哪些“已完成”
  - 保留现有能力边界，不引入自动登录、自动框选、自动保存

## Current State Analysis

- 当前 `采集配置` 左侧是三段分离结构：
  - `frontend/index.html`
  - 区块依次为：`快速初始化`、`扫描并绑定 Edge 标签页`、`任务配置`
- 当前“快速初始化”只负责批量创建或修正任务：
  - `frontend/app.js`
  - `initShops()` 仅调用 `POST /api/shops/init`
  - 初始化成功后只刷新任务列表，没有继续推进到下一步配置动作
- 当前“扫描并绑定”与“任务配置”之间没有自动承接：
  - `frontend/app.js`
  - `confirmBind()` 保存绑定后，只提示“请逐一进入任务管理 -> 编辑 -> 截取预览 -> 框选 GMV 区域 -> 保存”
  - 这导致用户完成绑定后又要跳去 `任务管理` 再回到 `采集配置`
- 当前“编辑任务”虽然能回填基础字段，但不会自动恢复到可直接操作的配置上下文：
  - `frontend/app.js`
  - `loadTaskIntoConfig(task)` 只回填平台、店铺、会话、间隔等字段，并提示“重新截取预览并框选后保存”
- 当前“真实 Edge 调试模式”能力已经齐全，但操作入口分散：
  - `frontend/app.js`
  - 已具备：会话管理、启动调试、打开页面、刷新页面、预览页面
  - 问题不在能力缺失，而在用户必须自己手动串联顺序
- 当前后端已经具备初始化、匹配候选页、批量绑定、页面预览、保存任务等关键接口：
  - `backend/main.py`
  - 现有接口可复用：
    - `GET /api/shops`
    - `POST /api/shops/init`
    - `GET /api/shops/match`
    - `POST /api/shops/bind`
    - `GET /api/edge-sessions`
    - `POST /api/edge-sessions/{session_id}/pages/{page_id}/preview`
    - `POST /api/tasks`
- 当前“待配置状态”没有专门抽象：
  - `backend/services/store.py`
  - `task_to_dict()` 虽返回 `page_id`、坐标比例等字段，但前端没有把这些字段组织成“待绑定 / 待标定 / 已完成”的工作清单
- `data/shops.csv` 当前只提供平台、店铺、关键词、URL 模式等配置，不包含一套可直接自动完成标定的数据：
  - `data/shops.csv`
  - 这意味着本次优化应聚焦“减少人工串联”，而不是追求全自动配置

## Proposed Changes

### 1. 将左侧配置区改造成“单页工作台”，但不改成独立向导

- 文件：
  - `frontend/index.html`
  - `frontend/styles.css`
- 改动内容：
  - 保留现有单页布局和预览区域
  - 将 `快速初始化`、`扫描并绑定`、`任务配置` 重组为一条连续工作台
  - 在左侧顶部新增“配置进度条/状态摘要区”，显示：
    - 总店铺数
    - 待绑定数
    - 待标定数
    - 已完成数
    - 当前处理店铺
  - 将“扫描并绑定”区域文案改成与当前上下文绑定，而不是全局说明
- 为什么这样做：
  - 用户明确希望保留单页形态
  - 现有页面的主要问题是步骤断裂，不是必须拆成多页
- 实现方式：
  - 在 `frontend/index.html` 中新增一个工作台头部区域和“当前任务卡”
  - 在 `frontend/styles.css` 中补充轻量流程状态样式，不做整页视觉重构

### 2. 新增前端“配置工作流状态机”，由系统决定下一步处理对象

- 文件：
  - `frontend/app.js`
- 改动内容：
  - 在现有 `state` 基础上新增工作流状态，例如：
    - `setupQueue`
    - `currentSetupTaskId`
    - `setupSummary`
    - `lastBindSessionId`
  - 新增一组纯前端 helper，用于从 `state.snapshot.tasks` 中计算：
    - 待绑定：`capture_mode === remote_edge` 且 `page_id` 为空
    - 待标定：`page_id` 已存在，但仍使用初始化默认裁剪参数
    - 已完成：已有 `page_id`，且裁剪参数已脱离默认初始化值
  - 新增统一入口：
    - `refreshSetupWorkbench()`
    - `focusSetupTask(taskId)`
    - `focusNextPendingTask()`
- 决策说明：
  - 本次不新增独立数据库字段
  - “待标定”先基于现有任务字段与默认初始化值比较来判断，避免扩库和迁移
  - 如后续发现误判，再在执行阶段评估是否下沉为后端汇总接口
- 为什么这样做：
  - 目前每一步做完后，用户都要自己重新定位上下文
  - 通过前端状态机可以在不重构后端模型的前提下，显著减少重复点击

### 3. 调整“快速初始化”后的默认行为：初始化成功后自动进入下一家待处理店铺

- 文件：
  - `frontend/app.js`
- 改动内容：
  - `initShops()` 成功后不只刷新任务列表，还要顺序执行：
    - `loadTasks()`
    - `refreshEdgeSessions()`
    - `refreshSetupWorkbench()`
    - 自动聚焦第一家待处理店铺
  - 在消息区明确反馈：
    - 创建多少家
    - 修正多少家
    - 当前已跳转到哪家店铺继续配置
- 默认跳转规则：
  - 优先第一家 `待绑定`
  - 若没有待绑定，则跳到第一家 `待标定`
  - 若都没有，提示“初始化已完成，无待处理项”
- 为什么这样做：
  - 这是对“快速初始化”最直接的体验增益
  - 用户点击一次后，系统立即给出下一步，而不是只给一个统计数字

### 4. 把“扫描并绑定”改成“针对当前店铺的绑定动作”，减少全量扫描心智负担

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
  - `frontend/styles.css`
- 改动内容：
  - 原有 `bindTable` 保留，但默认围绕“当前店铺/当前会话”展开
  - 页面增加上下文显示：
    - 当前店铺
    - 当前 Edge 会话
    - 候选页面数
    - 当前最佳匹配是否为自动匹配
  - `scanBind()` 的触发方式调整为：
    - 默认以“当前店铺对应的 edge_session_id”作为当前会话
    - 若用户主动切换会话，允许覆盖
  - `confirmBind()` 成功后，不再提示用户跳去 `任务管理`
  - 改为自动执行：
    - 将刚绑定的任务加载到配置表单
    - 自动选择绑定后的页面
    - 自动触发页面预览
    - 将用户停留在框选区域，直接进入“拖拽框选 -> 测试识别 -> 保存”
- 约束：
  - 自动预览只在本次绑定结果里存在唯一选中页面时触发
  - 若没有候选页或用户手工留空，仍保持人工选择
- 为什么这样做：
  - 当前最大断点就在这里，绑定完成后流程被人为切断

### 5. 为“当前任务配置”增加显式的上下文和快速跳转操作

- 文件：
  - `frontend/index.html`
  - `frontend/app.js`
  - `frontend/styles.css`
- 改动内容：
  - 在 `任务配置` 区上方增加“当前处理店铺”信息块，显示：
    - 平台 / 店铺名
    - 当前状态
    - 当前会话
    - 当前页面标题或 URL
  - 新增轻量操作按钮：
    - `下一家待配置`
    - `重新扫描本店铺页面`
    - `清空当前标定并重来`
  - 保存成功后不切回 `实时看板`
  - 改为：
    - 刷新任务
    - 刷新工作台状态
    - 自动跳到下一家待处理店铺
    - 若没有下一家，再提示本轮配置完成
- 关键决策：
  - 修改 `saveTask()` 结束行为，取消当前的 `switchView("dashboard")`
  - 这是本次体验优化的必要变更，否则用户每保存一次都会被踢出当前工作流

### 6. 保留现有真实 Edge 能力，但让“店铺会话”和“当前任务”自动联动

- 文件：
  - `frontend/app.js`
- 改动内容：
  - 当 `focusSetupTask(taskId)` 执行时，自动：
    - 选择该任务的 `edge_session_id`
    - 回填平台、店铺、关键词、确认次数、安全边距等字段
    - 刷新当前会话的健康状态与页面列表
  - 当用户在 `edgeSessionSelect` 中切换会话时：
    - 若当前处于某家店铺工作流中，提示这是“临时改绑”
    - 不直接丢失当前任务上下文
  - 若当前任务已有 `page_id`，则优先在页面下拉中选中已有页面
- 为什么这样做：
  - 当前会话选择器与任务表单属于两套平行控件，缺少主从关系
  - 自动联动后，用户的操作对象会更明确

### 7. 只做必要的后端补强，不重做接口体系

- 文件：
  - `backend/main.py`
  - `backend/services/store.py`
- 改动内容：
  - 保持现有初始化、匹配、绑定、预览、保存接口为主
  - 仅在以下场景补充后端辅助能力：
    - 若执行阶段验证发现前端难以稳定判断“默认裁剪值”，则新增一个轻量 setup summary 接口，返回每个任务的配置阶段状态
    - 若 `shops/bind` 需要支持“绑定后返回更完整任务对象”，则扩展响应体，避免前端二次查找
- 当前默认方案：
  - 先不预设新增 API
  - 执行时优先通过现有 `GET /api/tasks`、`GET /api/edge-sessions` 和 `GET /api/shops/match` 组合完成
- 为什么这样做：
  - 当前后端能力已经较完整，本次优化重点在前端交互串联
  - 避免为 UI 优化引入不必要的接口膨胀

## Assumptions & Decisions

- 本次优化以 `真实 Edge 调试页面` 主链路为核心，`兼容：已打开窗口` 只保持原有可用，不作为主要简化对象。
- `快速初始化` 的职责仍然是“从 `shops.csv` 同步任务”，不扩展为自动打开页面或自动登录。
- 不改动 `data/shops.csv` 结构；当前 CSV 编码问题属于数据源治理，不纳入本次交互优化范围。
- 保存任务后继续停留在 `采集配置` 工作流，而不是切回 `实时看板`，这是本次方案的核心交互决策。
- 不新增数据库字段；优先通过现有任务字段和初始化默认值推导配置阶段。
- 所有自动推进都必须可见、可中断：
  - 自动选择会话可以手动改
  - 自动选择候选页面可以手动改
  - 自动预览失败时给出明确错误，并停留在当前任务
- 不引入复杂拖拽式步骤组件，不做多页 wizard，不新增全局浮层式引导。

## Verification Steps

- 页面结构验证：
  - 打开 `采集配置`，确认左侧已变为连续工作台，而不是三个互相割裂的区块
  - 确认能看到配置摘要、当前店铺、待处理数量
- 初始化链路验证：
  - 点击 `一键初始化默认店铺`
  - 确认页面除了显示创建/修正数量外，还会自动定位到第一家待处理店铺
- 绑定链路验证：
  - 在当前店铺上下文中扫描候选页面
  - 确认绑定成功后不再要求用户跳去 `任务管理`
  - 确认系统会自动选中绑定页面并尝试预览
- 标定链路验证：
  - 在预览图上框选区域，执行 `测试识别`
  - 点击 `保存并启用`
  - 确认页面不会跳到 `实时看板`
  - 确认系统自动跳到下一家待处理店铺
- 边界场景验证：
  - 无候选页面时，能明确提示并停留在当前店铺
  - 自动预览失败时，不丢失当前任务上下文
  - 用户手动切换会话或页面时，不会导致表单状态异常
- 回归验证：
  - `任务管理 -> 编辑`
  - `任务管理 -> 重绑页面`
  - `window_capture` 模式预览和保存
  - WebSocket 看板刷新

## Implementation Order

1. 调整 `frontend/index.html` 的工作台结构，占位新增摘要区、当前任务区、下一步操作区。
2. 在 `frontend/app.js` 中新增工作流状态、状态计算 helper、当前任务聚焦逻辑。
3. 修改 `initShops()`、`scanBind()`、`confirmBind()`、`loadTaskIntoConfig()`、`saveTask()`，把流程串起来。
4. 在 `frontend/app.js` 中补充“下一家待配置”“重新扫描本店铺页面”等事件处理。
5. 在 `frontend/styles.css` 中补足工作台和状态样式，确保视觉层级清晰。
6. 如果执行时发现前端阶段判断不稳，再最小化补充 `backend/main.py` / `backend/services/store.py` 的辅助汇总能力。
