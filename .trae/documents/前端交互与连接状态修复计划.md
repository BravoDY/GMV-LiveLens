## Summary

- 目标：修复当前前端改版后出现的两个核心回归问题：
  - 页面上多数或全部按钮无法正常点击/触发动作
  - 顶部连接状态长期停留在“连接中”
- 范围：按“热修复 + 前端整理”处理，优先恢复功能，再补足与本次回归直接相关的前端状态治理和可用性细节。
- 约束：本次优先保持现有 UI 结构，不整体回退样式改版；仅修复会导致功能中断、状态误导、明显不可用的前端逻辑。

## Current State Analysis

### 1. 项目实际结构

- 前端入口：`frontend/index.html`
- 前端公共层：`frontend/core.js`
- 看板渲染：`frontend/dashboard.js`
- Edge 会话交互：`frontend/edge.js`
- 配置工作台：`frontend/config.js`
- 全局事件 / WebSocket / 调度器 / 初始化：`frontend/app.js`
- 后端入口与实时接口：`backend/main.py`
- 采集状态判定：`backend/services/scheduler.py`
- Edge 健康检查：`backend/collectors/remote_edge.py`

### 2. 已确认的现状

- `frontend/index.html` 中顶部状态文案默认写死为“连接中”，只有 `frontend/app.js` 的 `connectWs()` 成功运行后才会改成“实时连接”“轮询连接”或“连接失败”。
- `frontend/app.js` 当前把 WebSocket 地址写死为 `ws://${location.host}/ws/live`，没有根据 `http/https` 自动切换 `ws/wss`，也没有连接超时降级控制。
- `frontend/app.js` 在初始化阶段负责绑定几乎所有按钮事件；如果前置脚本未正常加载或执行被中断，页面会表现为“样式正常、按钮都不工作、连接状态停在初始文案”。
- `frontend/core.js` 的 `showEdgeSession()` 附近存在高风险异常点：回退提示语字符串包含未转义双引号，需在执行阶段优先验证其是否导致浏览器脚本解析/执行中断。
- `backend/main.py` 的 `/ws/live` 会在连接成功后立即推送一次快照，但前端当前缺少：
  - 协议自适配
  - 首次连接超时兜底
  - WS 成功后停止轮询兜底
  - 更清晰的连接状态流转
- `backend/services/scheduler.py` 会写入 `remote_page_not_found`，但 `frontend/core.js` 的 `taskStatusMeta` 只有 `page_not_found`，导致前后端状态枚举不完全一致。
- `frontend/config.js`、`frontend/edge.js` 内部有多处按钮会在异步请求期间被禁用；当前禁用态视觉提示较弱，用户容易误判为“按钮坏了”。

### 3. 对当前问题的判断

- “按钮都点不动”和“一直连接中”很可能不是两个独立故障，而是同一类前端初始化回归导致的连锁表现。
- 优先怀疑链路：
  1. `frontend/core.js` 或前置脚本存在语法/运行时异常
  2. 导致 `frontend/app.js` 未完整执行
  3. 事件监听未绑定，`connectWs()` 也未正常进入后续状态流转
  4. 页面只保留 HTML 初始态，因此顶部一直显示“连接中”
- 即便前端初始化问题不是唯一根因，`frontend/app.js` 当前 WebSocket 策略也存在真实缺陷，执行阶段应一并修复，避免后续再次卡在“连接中”。

## Proposed Changes

### A. 修复前端初始化阻断与按钮失效

#### 目标文件

- `frontend/core.js`
- `frontend/app.js`
- 必要时补充 `frontend/edge.js`、`frontend/config.js`

#### 修改内容

- 在 `frontend/core.js` 中优先修复 `showEdgeSession()` 回退文案的字符串问题，确保公共脚本可被浏览器稳定解析执行。
- 执行阶段在浏览器控制台 / 诊断中确认是否还有其他会阻断初始化的 JS 错误，并一并清理。
- 复核 `frontend/app.js` 的 `setupEvents()`、`setupSchedulerUi()`、`setupFullscreen()`、`connectWs()` 初始化顺序，确保任一子模块失败时不会让整个页面完全失活。
- 对关键点击入口做最小容错：
  - 顶部 Tab
  - 启动采集按钮
  - 全屏按钮
  - 任务管理卡片操作按钮
  - 配置页工作台按钮

#### 设计原因

- 当前所有主要交互均依赖前端初始化链路；优先恢复初始化，比局部修补某个按钮更稳妥。
- 保持现有模块拆分，不把逻辑重新堆回一个文件，符合项目现有结构。

### B. 修复连接状态长期停留“连接中”

#### 目标文件

- `frontend/app.js`
- 可选：`backend/main.py`（仅在执行阶段确认确有必要时）

#### 修改内容

- 将 WebSocket 地址从固定 `ws://` 改为基于当前页面协议自动选择 `ws://` 或 `wss://`。
- 为首次 WebSocket 连接增加超时控制；若在限定时间内未 `open`，主动关闭并切换到轮询兜底。
- 为 `ws.onerror`、`ws.onclose`、首次连接失败、重连中、轮询中、连接失败等状态建立更明确的 UI 文案流转。
- 在 WebSocket 成功连通时清理轮询定时器，避免 WS 与轮询并行更新。
- 保留当前后端 `/ws/live` 设计，不先改协议结构，优先做前端治理。

#### 设计原因

- 这属于本次故障直接暴露出的连接治理缺陷，修复成本低，收益高。
- 先修前端协议与超时控制，比贸然调整后端广播机制风险更低。

### C. 补齐状态映射与界面反馈

#### 目标文件

- `frontend/core.js`
- `frontend/styles.css`

#### 修改内容

- 在 `taskStatusMeta` 中补齐后端真实会返回的状态，如 `remote_page_not_found`，避免 UI 直接显示原始状态码。
- 梳理按钮禁用态的视觉和提示文案，降低“实际上在等待请求，但用户误以为按钮失效”的概率。
- 仅做与当前故障直接相关的可用性修正，不扩展到整套 UI 重做。

#### 设计原因

- 这类问题会放大用户对“前端坏了”的感知，属于低风险高价值整理项。

### D. 执行阶段验证与回归检查

#### 目标文件

- `frontend/core.js`
- `frontend/app.js`
- `frontend/edge.js`
- `frontend/config.js`
- 必要时 `backend/main.py`

#### 修改内容

- 用浏览器或诊断工具确认前端无新增语法错误。
- 验证以下核心场景：
  - 页面加载后顶部状态不再永久卡在“连接中”
  - Tab 可切换
  - `启动采集`、`全屏` 可点击
  - 配置页中的关键按钮可触发
  - 任务管理区按钮可触发对应动作
  - WebSocket 失败时会降级到轮询并显示正确状态

#### 设计原因

- 当前问题属于回归缺陷，重点不是补大量测试，而是做针对性可用性验证。

## Assumptions & Decisions

- 决策：采用“热修复 + 前端整理”，不整体回退现有 UI 改版。
- 决策：优先修复前端初始化链路，因为它最可能同时解释“按钮不可点击”和“连接中不变化”。
- 决策：WebSocket 问题先修前端协议与状态治理，不先大改后端推送机制。
- 假设：当前截图中“按钮都不可点击”主要指顶部导航、顶部操作、任务/配置页交互整体异常，而不是某一两个按钮单独失效。
- 假设：后端服务当前至少能正常返回首页和静态资源，否则不会出现现有 UI 画面。
- 风险注意：若执行阶段发现不是 JS 阻断，而是浏览器覆盖层或运行期异常导致事件未触发，需要把诊断重点转向：
  - `frontend/styles.css` 中布局覆盖
  - `frontend/app.js` 事件委托
  - 浏览器控制台运行时错误

## Verification Steps

### 执行阶段验证顺序

1. 打开页面并查看浏览器控制台，确认是否存在前端脚本解析/运行错误。
2. 检查 `frontend/core.js` 修复后，`frontend/app.js` 初始化是否完整执行。
3. 验证顶部 `实时看板 / 采集配置 / 任务管理` Tab 点击可正常切换。
4. 验证顶部 `启动采集`、`全屏`、连接状态条按预期变化。
5. 验证任务管理中任务按钮和平台级 Edge 按钮可正常触发。
6. 断开或模拟 WS 不可用时，确认状态会从“连接中”转为“轮询连接”或“连接失败”，不再永久卡死。
7. 检查最近编辑文件的诊断结果，确保未引入新的前端语法/类型问题。

### 验收标准

- 用户可正常点击主要按钮并触发对应动作。
- 页面初始化后连接状态能进入明确终态，不再无限停留于初始“连接中”。
- WebSocket 不可用时页面仍能通过轮询显示可用数据。
- 任务状态文案不再出现明显的前后端枚举错位。
