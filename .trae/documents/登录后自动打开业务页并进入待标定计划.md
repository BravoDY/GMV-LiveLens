# 登录后自动打开业务页并进入待标定计划

## Summary
- 目标：把采集配置工作台中的“待登录 -> 打开业务页 -> 待标定”链路进一步自动化，但采用低风险触发方式，避免在增强自动化时引入新的误跳页、误绑定、状态错乱问题。
- 已确认的产品决策：采用“登录后点按钮”的方式，而不是在 `启动Edge/显示Edge` 后无条件自动跳页。
- 交付标准：
  - 用户在 Edge 中手动完成登录后，只需在工作台点击一次按钮，系统就会尝试打开目标业务页、重扫页签、自动恢复绑定，并尽量进入“待标定”阶段。
  - 如果目标业务页未打开成功，系统要给出明确的失败原因，不得误判为“已完成”或“待标定”。
  - 新增功能完成后，必须对成功链路、失败链路、边界链路做反复回归验证。

## Current State Analysis

### 1. 现有能力已具备的部分
- 后端 `RemoteEdge` 已支持通过 `launch_url` 打开目标页：
  - `backend/collectors/remote_edge.py`
  - `_normalize_launch_url()`
  - `_ensure_launch_page()`
  - `_open_page()`
- 单店铺 `start/show` 接口已经会把 `launch_url` 透传到后端：
  - `backend/main.py`
  - `/api/edge-sessions/{session_id}/start`
  - `/api/edge-sessions/{session_id}/show`
- 当前自动恢复绑定也已具备基础能力：
  - `backend/main.py`
  - `auto_restore_edge_session_task_bindings()`
  - 会在 `start/show` 成功后根据现有页签自动恢复绑定
- 前端工作台已经能区分：
  - `待登录`
  - `待切业务页`
  - `待标定`
  - `已完成`
  对应文件：
  - `frontend/core.js`
  - `frontend/config.js`

### 2. 当前缺口
- 现在“待登录”场景只会提示：
  - 先在 Edge 中完成登录
  - 再点击“重新扫描”
- 也就是说，系统虽然知道当前是登录页，但还没有提供一个“登录后继续自动化”的专门动作。
- 当前用户需要手动做多步操作：
  1. Edge 里完成登录
  2. 手动打开目标业务页
  3. 回工作台点“重新扫描”
  4. 如果页签变化，还要重新确认绑定
- 这条链仍有较多人工步骤，也容易造成状态理解混乱。

### 3. 风险点
- 目标页打开成功 ≠ 已真正进入业务页：
  - 可能仍停在登录中间页
  - 可能跳到其它业务页
  - 可能被站点重定向
- 若在“登录后自动继续”里没有做严格校验，容易出现：
  - 错绑非目标页
  - 提前进入“待标定”
  - UI 提示和真实页状态再次不一致

## Proposed Changes

### 1. 新增后端“登录后继续”受控动作
- 文件：`backend/main.py`
- 建议新增接口：
  - `POST /api/tasks/{task_id}/resume-after-login`
- 职责：
  - 读取当前任务与其 `edge_session_id`
  - 取 `target_page_url`
  - 调用当前会话执行“打开目标业务页”
  - 重新获取当前页签列表
  - 运行现有：
    - `reconcile_edge_session_task_runtime()`
    - `auto_restore_edge_session_task_bindings()`
  - 返回结构化结果：
    - 是否成功发起目标页打开
    - 当前页签列表中是否出现目标业务页
    - 是否自动恢复绑定成功
    - 恢复后的任务状态
    - 若失败，返回明确 `reason_code / recovery_hint`
- 设计原因：
  - 这条链应做成任务级接口，而不是前端串多次请求拼装，便于统一收口异常与状态写回。

### 2. 在 `RemoteEdge` 中补一个显式“确保目标业务页”动作
- 文件：`backend/collectors/remote_edge.py`
- 调整方向：
  - 在不破坏现有 `start/show` 的前提下，暴露一个更明确的任务级操作入口，例如：
    - `ensure_launch_page`
    - 或新增包装方法供 `main.py` 调用
- 要求：
  - 只接受合法 `http/https` 目标 URL
  - 如果当前已经存在目标业务页，不重复开页
  - 如果当前仍是登录页，尝试导航到目标 URL
  - 导航后要等待短时间并重新扫描页签，不能只发起一次 `goto/open` 就宣称成功
- 设计原因：
  - 当前 `_ensure_launch_page()` 更像内部辅助动作；本次需要一个可控、可诊断、可测试的显式动作。

### 3. 将“自动继续”严格限定为低风险策略
- 文件：`backend/main.py`, `backend/collectors/remote_edge.py`
- 规则：
  - 仅在以下前提下执行：
    - 当前任务 `capture_mode == remote_edge`
    - 当前任务存在 `edge_session_id`
    - 当前任务存在 `target_page_url`
    - 当前任务当前状态为 `edge_login_page_bound` 或 `edge_page_bound`
  - 不做的事：
    - 不自动输入账号密码
    - 不自动点击站内未知按钮
    - 不在 `启动Edge/显示Edge` 后无条件跳页
- 设计原因：
  - 把自动化控制在“导航到目标业务 URL + 自动恢复绑定”范围内，降低新增 BUG 的概率。

### 4. 前端工作台新增“登录后自动继续”按钮
- 文件：`frontend/config.js`
- 调整内容：
  - 在 `setupFlowState()` 为 `login_required` 时，新增一个主按钮，例如：
    - `已登录，打开业务页并自动继续`
  - 点击后调用新的后端接口：
    - `/api/tasks/{task_id}/resume-after-login`
  - 根据结果自动执行：
    - 刷新任务快照
    - 刷新页签候选
    - 若已恢复到目标业务页，则自动进入“待标定”
    - 若仍是登录页/其它页，则展示明确失败提示
- 设计原因：
  - 交互上仍保持“用户手动完成登录后再点一下”，但后续步骤自动串起来。

### 5. 工作台步骤提示同步更新
- 文件：`frontend/config.js`
- 调整内容：
  - `login_required` 场景下的文案调整为：
    - 在 Edge 中完成登录后，点击“已登录，打开业务页并自动继续”
  - 当自动继续成功：
    - 若命中目标页且未标定：进入 `待标定`
    - 若命中目标页且已有非默认框选：进入 `已完成`
  - 当自动继续失败：
    - 区分：
      - 目标页未打开
      - 已打开但未命中目标页
      - 已命中但自动恢复绑定失败
- 设计原因：
  - 文案必须反映真实状态转换，避免再次出现“状态已变，但提示还停留在旧步骤”。

### 6. 前端事件与提示收口
- 文件：`frontend/config.js`, 必要时 `frontend/app.js`
- 调整内容：
  - 增加新的按钮点击事件处理
  - 统一成功/失败提示
  - 刷新后自动把当前任务重新载入配置面板
- 设计原因：
  - 避免后端状态变了，但前端仍停留在旧快照或旧按钮状态。

## Assumptions & Decisions
- 决策 1：采用“登录后点按钮”的触发方式，不在 `start/show` 后自动跳页。
- 决策 2：自动化范围限定为：
  - 打开目标业务页
  - 重扫页签
  - 自动恢复绑定
  - 自动推进到 `待标定/已完成`
- 决策 3：不自动执行站点登录动作，不自动处理账号密码。
- 决策 4：成功判定必须基于页签真实状态，而不是仅以“导航请求已发出”作为成功。
- 假设：现有 `target_page_url` 配置对相关任务是可用的；若为空，则接口直接返回可诊断错误，不做隐式猜测。

## Verification Steps

### 1. 代码级验证
- 检查新增后端接口是否：
  - 只在合法任务状态下执行
  - 能返回结构化结果而非裸异常
- 检查前端按钮是否只在 `待登录` 场景出现
- 检查成功后是否会刷新任务与页签上下文，而不是停留旧状态

### 2. 成功链路验证
- 场景 A：当前任务为 `edge_login_page_bound`，用户已在 Edge 中完成登录
  1. 点击“已登录，打开业务页并自动继续”
  2. 系统打开 `target_page_url`
  3. 自动重扫页签
  4. 自动恢复绑定
  5. 任务状态进入：
     - `待标定`，如果仍是默认框选
     - `已完成`，如果已有非默认框选

### 3. 失败链路验证
- 场景 B：用户未真正完成登录，仍停在登录页
  - 点击自动继续后，不得误进入 `待标定/已完成`
  - 应保持 `待登录` 或返回明确失败提示
- 场景 C：目标 URL 打开后跳到了非目标页
  - 应进入 `待切业务页`，不得误判成 `待标定`
- 场景 D：目标页已打开，但自动恢复绑定失败
  - 必须返回明确提示，允许用户继续手动重扫/重绑

### 4. 回归验证
- 反复测试以下不应被破坏：
  - 单店铺 `启动Edge / 显示Edge / 隐藏Edge / 关闭Edge`
  - 自动恢复绑定
  - 登录页/业务页状态识别
  - 工作台摘要统计与切店下拉标签
- 至少覆盖：
  - 官方旗舰店
  - 儿童旗舰店
  - 一个未绑定页签的店铺
  - 一个已完成标定的店铺

### 5. 验收标准
- 用户完成登录后，只需点击一次工作台按钮，即可尽量自动推进到“待标定”。
- 任一失败场景都不能把任务误判成“已完成”或“待标定”。
- 原有 Edge 控制链与工作台状态一致性不被破坏。
