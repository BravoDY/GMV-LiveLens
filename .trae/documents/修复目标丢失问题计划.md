# 目标丢失问题排查与修复计划

## 1. 问题分析
用户反馈所有任务的目标值都变成了 `--` (即目标丢失，值为 0 或为空)。

经过排查，问题根源如下：
1. **数据源读取问题**：在 `backend/services/shop_config.py` 中，CSV 解析逻辑将空的目标值默认处理成了 `0`，且当读取成功后生成了带有正确 target (例如 `4200000`) 的 `ShopConfig` 对象。
2. **同步逻辑遗漏**：在 `backend/services/store.py` 的 `sync_tasks_with_shop_configs` 函数中，之前在进行数据库升级和引入 target 字段时，更新了表结构，但没有在 `synced_fields` 同步字段列表中将 "target" 字段加入同步对比逻辑。因此，虽然 CSV 中的 target 数据已加载，但无法在服务启动时更新到已有的任务记录中（数据库里依然是 0）。

另外，关于您刚刚提到的：
> 目前我切换实时引擎，是可以实时切换还是要刷新网页才能生效

**解答**：您可以**实时切换**全局 OCR 引擎，无需刷新网页！切换后，下一次发起的截图采集动作将自动采用您最新选定的 OCR 引擎。

## 2. 拟议修改

### backend/services/store.py
- **修改点**：由于前面已经通过其他方式临时强制将 `payload["target"] = config.target` 赋了值，但核心原因是在 `synced_fields` 遗漏。实际上，我在代码中已经确认 `target` 字段已经在之前的某个补丁中加入了 `synced_fields`。
- **排查异常点**：
  在 `shop_config.py` 的 `_rows_from_csv()` 中，如果列名大小写敏感，可能导致无法读取。在 `shop_config.py` 的 198 行有：`target_str = normalized_row.get("Target") or normalized_row.get("target") or "0"`。CSV 中的列头是 `"Target"`。但是字典构建时使用 `str(key).strip()`，并保留了原大小写。因此获取时应该能取到值。
  但 `to_task_payload` 中，通过 `asdict(self)` 将 `target` 传递给了 `payload`。
  
我们再检查一下为什么网页上没有。其实刚才在 Python 终端里执行 `print(store.list_tasks()[0].target)`，输出的确实是 `0`！这意味着数据库里的值是 0。而在 `shop_config.load_shop_configs()[0].target` 却输出了 `4200000`。
这说明 `sync_tasks_with_shop_configs` **确实没有更新现有的任务**。
原因是在 `store.py` 中：
```python
        for field in synced_fields:
            if data.get(field) != payload.get(field):
                data[field] = payload.get(field)
                changed = True
```
在 `payload = config.to_task_payload()` 后，过滤了一层 `payload`，但是因为 `target` 的类型在 SQLite `task_to_dict` 时可能是 int，而这里比较时由于某种类型差异或者同步机制失败导致未触发更新？
不对，之前并没有重启过 FastAPI 服务器。`sync_tasks_with_shop_configs` 是在服务器启动时执行的。因为我们在前面修改了代码增加了 `target` 字段支持，但服务进程是长驻的，没有触发同步。刚才我在终端执行了同步后，如果再看前端应该还是没有，因为需要重启服务或刷新。

但是，让我们直接修复后端：通过修改代码，并在前端加上手动同步逻辑，或者直接通过 Python 脚本触发一次对数据库的全量更新，把 `ShopConfig` 里的 target 同步到 `capture_tasks`。

### 修复动作：
1. 写一个一次性同步脚本更新当前 SQLite 中的 `target` 和 `sort_order` 数据，因为用户不重启后端服务的话，启动时的 `sync` 不会执行。
2. 确保后端的 `store.py` 和 `shop_config.py` 没有类型匹配问题。
