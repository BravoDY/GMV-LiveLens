# 修复页签切换失效与视图串页问题计划

## Summary
- 目标：修复点击 `采集配置`、`数据治理`、`任务管理` 时仍然呈现实时看板内容的问题，恢复四个页签各自独立展示。
- 成功标准：
  - 点击 `实时看板` 时只显示实时看板；
  - 点击 `采集配置` 时只显示采集配置；
  - 点击 `数据治理` 时只显示数据治理；
  - 点击 `任务管理` 时只显示任务管理；
  - 不再出现“实时看板内容和其他页面串在一起”的情况。

## Current State Analysis
- `frontend/core.js`
  - `switchView(view)` 仍然使用 `.view` / `.view.active` 控制显示状态，逻辑本身没有被破坏。
  - 当前实现会对所有 `.view` 节点切换 `active`，说明点击行为和页签切换入口仍在。
- `frontend/app.js`
  - 页签点击事件依旧调用 `switchView(button.dataset.view)`，并在切到 `config-flow` 时补充执行 `renderConfigFlowPage()`。
  - 从代码形态看，点击后“不跳转”的根因不是事件绑定丢失。
- `frontend/styles.css`
  - 旧的通用规则仍然存在：`.view { display: none; }` 与 `.view.active { display: block; }`。
  - 最近新增的实时看板 UI 样式在后半段引入了 `dashboard-view`、`dashboard-top-frame`、`dashboard-brand-section` 等新容器。
  - 现象与代码结合判断：问题更像是实时看板新容器/布局影响了其他 `.view` 的显示边界，或把本应只属于实时看板的结构视觉上扩展到了整页。
- `frontend/index.html`
  - 当前 `main` 下仍是四个并列 `.view` 区块：`dashboard`、`config`、`config-flow`、`manager`。
  - DOM 结构本身没有把多个页面真的嵌套进实时看板，但最新浏览器快照出现了同时看到 `数据治理` 内容的现象，说明执行后的显示层出了问题。

## Recommended Implementation Plan
- 修改 `frontend/styles.css`
  - 复核并收敛所有最近新增的实时看板样式作用域，只允许它们作用于 `#dashboard.dashboard-view` 内部。
  - 重点检查并修复：
    - `body.dashboard-page`
    - `.dashboard-view`
    - `.dashboard-top-frame`
    - `.dashboard-brand-section`
    - `.store-grid`
    - `.brand-area-layout`
  - 目标是确保这些样式不会影响 `#config`、`#config-flow`、`#manager` 的可见性与布局。
- 复核 `frontend/index.html`
  - 检查是否需要微调实时看板专属容器的包裹方式，使其与其他 `.view` 的层级边界更清晰。
  - 仅在必要时做最小结构修正，不改变四个视图的既有 DOM 入口。
- 复核 `frontend/core.js`
  - 若 CSS 修复后问题仍存在，再补一层防御性处理，例如确保切换时目标 `.view` 以外的页面不会残留异常类名或内联显示状态。
  - 优先保持现有 `switchView()` 简洁实现，不主动加复杂逻辑。

## File-by-File Changes
- `frontend/styles.css`
  - 收紧实时看板新样式的作用域；
  - 修复 `.view` 切换后其他页面仍被视觉挤入当前页的问题；
  - 保持实时看板的双品牌区布局，但不能牵连其它页面。
- `frontend/index.html`
  - 如有必要，只做最小结构调整，确保实时看板容器边界清晰。
- `frontend/core.js`
  - 仅当样式层不足以修复时，才加最小的页签切换防御性逻辑。

## Assumptions & Decisions
- 决策：优先判定为样式/容器作用域问题，而不是事件逻辑问题，因为 `switchView()` 与页签点击绑定仍然存在且结构清晰。
- 决策：先修 `styles.css`，再看是否需要动 `index.html` 或 `core.js`；避免直接修改 JS 导致问题被掩盖。
- 假设：当前“采集配置 / 数据治理 / 任务管理都显示成实时看板”是同一个根因导致的连带现象，修复 `.view` 显示隔离后可整体恢复。
- 假设：此次修复不应影响已接入的 `brand` 分组和双品牌区域数据逻辑，只修复展示边界与页签切换体验。

## Risks
- 如果只修视觉样式但遗漏某个后置选择器，可能出现“看起来好了，但某一页签仍被局部样式污染”的残留问题。
- 如果误改 `.view.active` 或公共 `main` 布局，可能引入四个页面同时不可见或页面高度塌陷的问题。
- 如果过度修改实时看板容器，可能破坏刚完成的双品牌区域布局。

## Verification Steps
- 静态检查
  - 检查 `frontend/styles.css`、`frontend/index.html`、`frontend/core.js` 是否有新增诊断错误。
- 页面切换验证
  - 打开实时看板，确认只显示实时看板。
  - 点击 `采集配置`，确认只显示采集配置工作台和预览区。
  - 点击 `数据治理`，确认只显示数据治理内容，不再混入实时看板。
  - 点击 `任务管理`，确认只显示任务管理内容。
- 回归验证
  - 切回 `实时看板`，确认顶部 Summary 与下方左右双品牌区域仍然存在。
  - 确认 `采集配置 / 数据治理 / 任务管理` 页面未被双品牌区域样式压缩或覆盖。
