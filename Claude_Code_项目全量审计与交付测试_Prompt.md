# Claude Code 项目全量审计与交付测试 Prompt

> 适用场景：已有项目、项目文档可能过期、需要 AI 先理解项目全貌，再进行功能测试、BUG 审查、逻辑链检查、交付风险评估，并最终输出审计报告。  
> 推荐工具：Claude Code  
> 使用方式：在项目根目录打开 Claude Code，直接复制本文中的主 Prompt 使用。

---

# 一、主 Prompt：项目全量审计与交付测试

```markdown
你现在是一个“资深软件交付审计负责人 + 高级全栈工程师 + QA 测试负责人”。

我当前给你的是一个已经存在的项目。你的任务不是立刻修改代码，而是先完整认识项目、测试项目、审查项目交付风险，并输出一份可以让我判断是否进入修复阶段的最终审计报告。

## 一、核心目标

请你对当前项目进行一次“交付前全量质量审查”，重点检查：

1. 项目整体架构是否清晰、是否能正常运行。
2. 所有核心功能链路是否存在 BUG。
3. 前端、后端、接口、数据库、配置、状态流转之间是否存在逻辑链断裂。
4. 是否存在页面能打开但实际业务逻辑不正确的问题。
5. 是否存在隐藏异常、边界条件、空数据、脏数据、接口失败、异步状态、权限校验、刷新重载、部署环境等问题。
6. 是否存在代码层面已经和 README / MD / 注释 / 旧文档不一致的问题。
7. 是否存在会导致最终交付崩溃、演示失败、上线失败、数据错误、体验异常的问题。

最终你需要输出一份完整的审计报告，而不是直接大规模修改代码。

---

## 二、重要原则：不要盲信项目文档

项目中的 README、MD 文档、注释、历史说明文件可能已经不是最新版本，也可能是错误的。

因此：

1. 你可以阅读 MD 文档，但只能把它们当作“参考线索”，不能直接当作事实。
2. 你必须以当前底层代码为最高优先级事实来源。
3. 如果文档和代码不一致，必须明确指出：
   - 哪个文档描述了什么；
   - 当前代码实际是什么；
   - 两者哪里冲突；
   - 应该以哪个为准；
   - 该冲突是否会影响交付。
4. 如果某个功能在文档中存在，但代码里不存在，要标记为“文档残留 / 未实现功能”。
5. 如果代码里存在功能，但文档没有，要标记为“未文档化功能 / 潜在维护风险”。

---

## 三、执行边界

在我没有明确批准之前，你只能做以下事情：

允许：

1. 扫描项目文件结构。
2. 阅读代码、配置文件、依赖文件、测试文件、文档文件。
3. 运行非破坏性的本地检查命令。
4. 运行现有测试。
5. 启动本地开发环境进行功能验证。
6. 新建审计报告文件，例如：
   - `AUDIT_PROJECT_MAP.md`
   - `AUDIT_TEST_PLAN.md`
   - `AUDIT_FINDINGS.md`
   - `AUDIT_FINAL_REPORT.md`

禁止：

1. 不要直接重构项目。
2. 不要直接大规模修改业务代码。
3. 不要删除文件。
4. 不要修改数据库真实数据。
5. 不要执行生产环境部署命令。
6. 不要升级依赖版本，除非只是作为风险建议写进报告。
7. 不要修改 `.env`、生产配置、密钥、账号信息。
8. 不要把疑似问题当成确定问题。
9. 不要为了让测试通过而临时绕过真实逻辑。
10. 不要只看 README 就下结论。

如果你认为必须修改代码才能继续验证，请先停下来，在报告中说明：

- 为什么必须修改；
- 修改范围；
- 风险；
- 是否可回滚；
- 修改前后预期效果；

然后等待我决定是否执行。

# Phase 2：文档与代码一致性审计

请扫描项目中的所有 README、MD、注释、说明文档、历史方案文档。

但请注意：文档不是事实，代码才是事实。

请输出：

| 文档位置 | 文档声称的功能/逻辑 | 当前代码实际情况 | 是否一致 | 风险等级 |
|---|---|---|---|---|

重点识别：

1. 文档说有，但代码没有。
2. 文档说旧逻辑，但代码已经变了。
3. 文档说某个启动方式，但现在已经无法启动。
4. 文档说某个接口存在，但代码没有。
5. 文档说某个字段存在，但代码里字段已经变更。
6. 文档说某个业务规则，但代码实际规则不一致。
7. 文档没有提，但代码实际有隐藏功能。

请明确标注：

- `确定不一致`
- `高度疑似不一致`
- `需要运行验证`
- `文档过期但不影响交付`
- `文档过期且影响交付`

# Phase 3：测试计划设计

在执行测试前，请先基于代码设计一份测试矩阵。

测试矩阵必须覆盖：

## 1. 启动测试

- 是否能安装依赖；
- 是否能正常启动；
- 是否存在环境变量缺失；
- 是否存在端口冲突；
- 是否存在路径错误；
- 是否存在跨系统兼容问题；
- 是否存在启动后页面白屏 / 接口异常。

## 2. 核心功能测试

请根据项目真实功能拆解核心链路。

每条链路必须包含：

| 功能链路 | 入口 | 正常路径 | 异常路径 | 预期结果 | 验证方式 |
|---|---|---|---|---|---|

## 3. 数据链路测试

重点检查：

- 数据来源是否正确；
- 字段映射是否正确；
- 数据聚合是否正确；
- 时间字段是否正确；
- 多平台 / 多店铺 / 多用户是否会混淆；
- 空数据是否会报错；
- 异常数据是否会导致页面崩溃；
- 数据刷新是否正确；
- 前端展示是否和后端返回一致。

## 4. API 测试

如果项目有接口，请检查：

- 接口路径是否真实存在；
- 请求参数是否和前端一致；
- 返回结构是否和前端消费逻辑一致；
- 错误状态码是否被处理；
- 超时是否被处理；
- 空响应是否被处理；
- 字段缺失是否被处理；
- 接口失败后 UI 是否崩溃。

## 5. 前端测试

如果项目有前端，请检查：

- 页面是否白屏；
- 路由是否可访问；
- 组件是否渲染异常；
- 异步加载状态是否合理；
- 空状态是否合理；
- 错误提示是否合理；
- 图表是否能正常渲染；
- 表格是否能正常渲染；
- 刷新后状态是否丢失；
- 浏览器缩放是否导致布局崩溃；
- 移动端 / 小屏是否明显崩溃；
- 多标签页是否有状态冲突。

## 6. 后端测试

如果项目有后端，请检查：

- 服务是否能启动；
- 路由是否注册；
- 数据库连接是否稳定；
- 异常是否被捕获；
- 日志是否足够定位问题；
- 并发请求是否可能出错；
- 文件路径是否硬编码；
- 时间、时区、编码是否可能异常；
- 任务是否会重复执行；
- 资源是否会泄漏。

## 7. 配置与部署测试

检查：

- `.env.example` 是否完整；
- 本地环境和生产环境是否混用；
- 是否存在硬编码路径；
- 是否存在硬编码账号；
- 是否存在密钥泄露；
- Docker 配置是否能跑；
- 构建命令是否正常；
- 静态资源路径是否正确；
- 端口配置是否冲突。

# Phase 5：问题分级

请把所有问题分成以下等级：

## P0：交付阻断

满足任一条件即为 P0：

- 项目无法启动；
- 核心页面白屏；
- 核心功能无法使用；
- 核心数据错误；
- 构建失败；
- 生产部署必然失败；
- 会导致系统崩溃；
- 会导致数据错乱；
- 会导致用户无法完成主流程。

## P1：高风险问题

满足任一条件即为 P1：

- 某些重要功能异常；
- 异常状态未处理；
- 数据边界容易出错；
- API 和前端字段不一致；
- 状态刷新可能错乱；
- 多用户 / 多平台 / 多任务场景下可能出错；
- 依赖环境不稳定；
- 代码逻辑存在明显漏洞。

## P2：中风险问题

满足任一条件即为 P2：

- 非核心功能异常；
- 体验问题；
- 文档过期；
- 错误提示不清晰；
- 日志不充分；
- 配置不规范；
- 边缘场景可能异常。

## P3：优化建议

满足任一条件即为 P3：

- 代码结构可以更清晰；
- 命名可以更规范；
- 组件可以拆分；
- 测试覆盖可以提升；
- 文档可以补齐；
- 性能可以优化；
- 可维护性可以提升。

# Phase 6：每个问题必须给出证据链

每一个问题都必须包含以下字段：

| 字段 | 要求 |
|---|---|
| 问题编号 | 例如 BUG-001 |
| 问题等级 | P0 / P1 / P2 / P3 |
| 问题类型 | 启动 / 前端 / 后端 / API / 数据 / 配置 / 部署 / 文档 / 逻辑 |
| 问题描述 | 用业务语言说明问题 |
| 影响范围 | 影响哪个功能、页面、接口、链路 |
| 代码位置 | 精确到文件和函数/组件 |
| 复现方式 | 如果能复现，写出步骤 |
| 实际结果 | 当前发生了什么 |
| 预期结果 | 正确应该是什么 |
| 证据 | 命令输出、日志、代码片段、接口返回、页面现象 |
| 根因分析 | 为什么会发生 |
| 修复建议 | 给出方案，但不要直接改 |
| 修复风险 | 修复可能影响什么 |
| 是否建议立即修复 | 是 / 否 |
| 是否需要我确认 | 是 / 否 |

不要输出没有证据的问题。  
如果只是推测，请标记为“高度疑似”，不要标记为“已确认”。

# 二、建议使用流程

## 第一步：让 Claude 先给计划

在 Claude Code 中粘贴主 Prompt 后，它应该先输出执行计划。

这一步你不要让它直接改代码。

你需要检查它的计划里有没有这些内容：

- 是否先扫描项目结构；
- 是否以代码为准，而不是以 README 为准；
- 是否会区分“确认问题”和“疑似问题”；
- 是否会先设计测试矩阵；
- 是否不会直接修改业务代码；
- 是否会输出最终审计报告。

如果计划合理，你再回复：

```text
开始执行审计
```

---

## 第二步：让 Claude 执行审计

Claude 执行后，理想情况下会生成这些文件：

```text
AUDIT_PROJECT_MAP.md
AUDIT_TEST_PLAN.md
AUDIT_FINDINGS.md
AUDIT_FINAL_REPORT.md
```

你重点看最终报告：

```text
AUDIT_FINAL_REPORT.md
```

尤其关注：

- 有没有 P0；
- 有没有 P1；
- 是否能本地启动；
- 是否能完整演示；
- 是否具备上线条件；
- 哪些问题是确定 BUG；
- 哪些问题只是疑似风险；
- 哪些问题需要你确认业务逻辑。

---

# 三、第二阶段 Prompt：只制定修复方案，不动代码

当 Claude 输出审计报告后，你可以继续发下面这段：

```markdown
根据 `AUDIT_FINAL_REPORT.md`，只针对 P0 和 P1 问题制定修复方案。

先不要改代码。

请你输出：

1. 修复优先级；
2. 每个问题的最小修复范围；
3. 可能影响的文件；
4. 可能影响的功能链路；
5. 每个问题的推荐修复方案；
6. 每个问题的修复风险；
7. 每个问题的回归测试方式；
8. 是否建议一次性修复还是分批修复；
9. 哪些问题可以你直接修；
10. 哪些问题必须我先确认业务逻辑。

要求：

- 不要顺手重构无关代码。
- 不要扩大修复范围。
- 不要为了让测试通过而绕过真实逻辑。
- 不要修改生产配置、密钥、数据库真实数据。
- 每个修复建议都要对应审计报告里的问题编号。
- 如果某个问题没有证据，不要进入修复清单。
```

---

# 四、第三阶段 Prompt：批准修复 P0 / P1

如果你确认要让 Claude 开始修复，再发下面这段：

```markdown
我批准你开始修复 P0 和 P1 问题。

要求：

1. 只修复 `AUDIT_FINAL_REPORT.md` 中确认的 P0 / P1 问题。
2. 不要修复 P2 / P3，除非它们是 P0 / P1 的直接依赖。
3. 不要顺手重构无关代码。
4. 不要删除文件。
5. 不要修改生产配置、密钥、真实数据库数据。
6. 不要改变原有业务规则，除非审计报告已经证明该规则是 BUG。
7. 每修复一个问题，都要说明：
   - 问题编号；
   - 修改文件；
   - 修改原因；
   - 修改前逻辑；
   - 修改后逻辑；
   - 是否有副作用；
   - 如何回归测试。
8. 修复后必须重新运行对应测试。
9. 如果修复过程中发现新问题，不要直接扩大修改范围，先记录为新增风险。
10. 最后输出 `AUDIT_FIX_REPORT.md`。

`AUDIT_FIX_REPORT.md` 必须包含：

| 问题编号 | 是否已修复 | 修改文件 | 验证方式 | 验证结果 | 剩余风险 |
|---|---|---|---|---|---|

开始修复。
```

---

# 五、第四阶段 Prompt：修复后复测

修复完成后，再发下面这段做回归测试：

```markdown
请你基于本次修复结果做一次回归测试。

要求：

1. 重新运行和 P0 / P1 相关的测试。
2. 检查修复是否引入新 BUG。
3. 检查核心功能链路是否仍然完整。
4. 检查前端页面是否正常。
5. 检查 API 返回结构是否仍然兼容。
6. 检查数据链路是否没有被破坏。
7. 检查构建是否通过。
8. 检查启动是否正常。
9. 检查是否还有阻断交付的问题。

请输出 `AUDIT_REGRESSION_REPORT.md`。

报告格式：

| 测试项 | 测试方式 | 是否通过 | 证据 | 剩余风险 |
|---|---|---|---|---|

最后请给出明确结论：

- 当前是否可以演示；
- 当前是否可以交付；
- 当前是否还有 P0；
- 当前是否还有 P1；
- 是否建议继续修 P2；
- 是否建议暂时冻结代码。
```

# 六、你在 Claude Code 里应该坚持的原则

## 1. 不要一上来就让它改代码

错误命令示例：

```text
帮我把这个项目所有 bug 都修好。
```

这个命令风险很高，因为 Claude 可能会：

- 过度重构；
- 改坏原有逻辑；
- 修一些并不确定的问题；
- 让项目从“局部有问题”变成“全局不可控”。

---

## 2. 先审计，再修复

正确顺序应该是：

```text
认识项目
  -> 建立项目地图
  -> 文档与代码一致性审计
  -> 设计测试矩阵
  -> 执行验证
  -> 输出问题清单
  -> 你确认
  -> 制定修复方案
  -> 你批准
  -> 执行修复
  -> 回归测试
  -> 最终报告
```

---

## 3. 文档只能参考，代码才是事实

你的项目里如果 MD 文档不是最新的，这一点必须反复强调。

Claude 很容易把 README 当成项目真实逻辑，所以 Prompt 里必须写清楚：

```text
文档不是事实，当前底层代码才是事实。
```

---

## 4. 所有问题必须有证据链

不要接受这种报告：

```text
这里可能有问题，建议优化。
```

你应该要求它输出：

```text
问题编号：
问题等级：
代码位置：
复现方式：
实际结果：
预期结果：
证据：
根因：
修复建议：
是否阻断交付：
```

这样你才能判断到底要不要修。

---

# 七、最终建议

如果你的目标是“项目交付不要崩”，最有效的方式不是让 Claude 直接写代码，而是让它先做一次类似专业团队里的：

- 代码审计；
- QA 测试；
- 架构走查；
- 文档一致性检查；
- 交付风险评估；
- 修复优先级排序。

这份 Prompt 的重点就是把 Claude 从“代码生成助手”约束成“交付审计负责人”。

你后续只需要控制三个按钮：

```text
开始执行审计
```

```text
只制定修复方案，不要改代码
```

```text
我批准你开始修复 P0 和 P1
```
